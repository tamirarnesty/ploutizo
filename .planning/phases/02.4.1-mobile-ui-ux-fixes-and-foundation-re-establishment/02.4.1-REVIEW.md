## Warnings

### WR-01: Stale local state after add/archive mutations in `CategoriesSettings` and `MerchantRulesSettings`

**File:** `apps/web/src/components/settings/CategoriesSettings.tsx:72-86`
**Also affects:** `apps/web/src/components/settings/MerchantRulesSettings.tsx:20-22`

**Issue:** Both components maintain a local copy of server data to support optimistic drag-reorder UX. In `CategoriesSettings`, `localCategories` is seeded once from server data (via `initialized` ref guard) and never updated again. In `MerchantRulesSettings`, `localRules` starts empty and is only populated on first drag. In both cases, once the local state is populated, the `displayX = localX.length > 0 ? localX : serverX` selector permanently prefers the stale local copy over server data.

Consequence: after a user adds a new category/rule or archives an existing one (which invalidates the TanStack Query cache and fetches fresh server data), the new server data is ignored. The newly added item never appears in the list, and archived items continue to show until the page is refreshed.

**Fix for `CategoriesSettings`:** Drop the one-time guard pattern and instead merge server updates into local state reactively, preserving drag order but appending/removing items that changed:

```tsx
// Replace the initialized-ref approach with a sync effect that
// updates localCategories whenever the server set changes:
const prevCategoryIds = useRef<Set<string>>(new Set())

useEffect(() => {
  if (catLoading) return
  const serverIds = new Set(categories.map((c) => c.id))
  const localIds = new Set(localCategories.map((c) => c.id))

  const added = categories.filter((c) => !localIds.has(c.id))
  const removedIds = [...localIds].filter((id) => !serverIds.has(id))

  if (added.length === 0 && removedIds.length === 0) return

  setLocalCategories((prev) => [
    ...prev.filter((c) => !removedIds.includes(c.id)),
    ...added,
  ])
  prevCategoryIds.current = serverIds
}, [categories, catLoading])
```

Apply the same pattern to `MerchantRulesSettings`.

### WR-02: `animate-pulse` in `MerchantRulesSettings` missing `motion-safe:` prefix

**File:** `apps/web/src/components/settings/MerchantRulesSettings.tsx:41`

**Issue:** The skeleton loader uses `animate-pulse` without the `motion-safe:` variant. Per the web-design-guidelines motion rule, any `animate-*` usage must respect `prefers-reduced-motion`. `CategoriesSettings` already applies `motion-safe:animate-pulse` correctly on line 110 â€” `MerchantRulesSettings` is inconsistent.

**Fix:**
```tsx
// Before
<div key={i} className="h-12 animate-pulse rounded bg-muted" />

// After
<div key={i} className="h-12 motion-safe:animate-pulse rounded bg-muted" />
```

### WR-03: `&&` conditional render in `MerchantRulesSettings` violates `rendering-conditional-render` convention

**File:** `apps/web/src/components/settings/MerchantRulesSettings.tsx:74`

**Issue:** The `RuleDialog` is conditionally rendered with `&&`:

```tsx
{dialogRule !== false && (
  <RuleDialog rule={dialogRule} onClose={() => setDialogRule(false)} />
)}
```

Per the project's `rendering-conditional-render` rule (CLAUDE.md / vercel-react-best-practices), ternary operators must be used for conditional rendering when the false branch should render nothing. `&&` can render `0` or `false` to the DOM depending on the left-hand expression type. `CategoriesSettings` already uses ternary correctly at line 278.

**Fix:**
```tsx
{dialogRule !== false ? (
  <RuleDialog rule={dialogRule} onClose={() => setDialogRule(false)} />
) : null}
```

### IN-01: `SidebarTrigger` in `AppSidebar` footer has no `aria-label`

**File:** `apps/web/src/components/AppSidebar.tsx:75`

**Issue:** The footer `<SidebarTrigger />` renders an icon-only button (collapse/expand toggle). Per the accessibility rule in the web-design-guidelines, all icon-only buttons must have an `aria-label`. Without one, screen readers have no way to identify its purpose.

**Fix:** Pass an `aria-label` appropriate to its current collapsed state. If `SidebarTrigger` accepts `aria-label`, pass it directly; otherwise wrap with a tooltip or accessible label:

```tsx
<SidebarTrigger aria-label="Toggle sidebar" />
```

### IN-02: `useRef` initialised with `undefined` when explicit type excludes `undefined`

**File:** `apps/web/src/hooks/useThemeKeyboardShortcut.ts:12`

**Issue:** `useRef<(e: KeyboardEvent) => void>(undefined)` passes `undefined` as the initial value for a ref typed as `(e: KeyboardEvent) => void`. Under React 19 type overloads, the no-argument form `useRef<T | undefined>()` is the idiomatic way to declare an optionally-unset ref. The explicit `undefined` argument can cause TypeScript to infer a wider `T | undefined` union than intended and may produce a type error under stricter future typings.

The runtime behaviour is correct (the optional chain `handlerRef.current?.(e)` guards the `undefined` case).

**Fix:**
```ts
// Before
const handlerRef = useRef<(e: KeyboardEvent) => void>(undefined)

// After â€” no-arg form; T | undefined is inferred correctly
const handlerRef = useRef<(e: KeyboardEvent) => void>()
```
