### D-01: Layout height constraint — fixes TopBar scrolling + document-level horizontal overflow

**File:** `apps/web/src/routes/_layout.tsx`

**Problem:** `SidebarProvider`'s wrapper div has `min-h-svh` hardcoded, allowing it to grow beyond the viewport. TopBar scrolls with page content; wide content causes document-level horizontal scroll.

**Fix:** Wrap `SidebarProvider` in our own container div. We do NOT add layout-override classes directly to `SidebarProvider` — the shadcn component should remain updatable without us fighting its internals.

```tsx
function LayoutShell() {
  return (
    <div className="flex h-dvh flex-col overflow-hidden">
      <SidebarProvider className="flex-col flex-1 min-h-0">
        <TopBar />
        <div className="flex min-h-0 flex-1">
          <AppSidebar />
          <SidebarInset>
            <main className="flex-1 overflow-auto p-6">
              <Outlet />
            </main>
          </SidebarInset>
        </div>
      </SidebarProvider>
    </div>
  )
}
```

- Outer `div.h-dvh.overflow-hidden`: our container — provides the viewport constraint independently of SidebarProvider's internals
- `SidebarProvider` gets only `flex-col flex-1 min-h-0` (standard flex layout; `min-h-0` overrides `min-h-svh` to allow flex shrink)

**Expected result:** TopBar fixed; only `<main>` scrolls. No horizontal document-level scroll.

### D-02: Sidebar collapse button missing — sidebar overflows viewport

**File:** `apps/web/src/components/AppSidebar.tsx`

**Problem:** The sidebar component base class has `fixed inset-y-0 h-svh`. The current `className="top-10"` overrides `top` to `2.5rem`, but `h-svh` remains `100svh`. Result: sidebar runs from `2.5rem` to `2.5rem + 100svh`, extending `2.5rem` beyond the viewport bottom — the footer (and the collapse SidebarTrigger inside it) is clipped below the fold.

**Fix:** Change to `className="top-10 h-[calc(100svh-2.5rem)]"`. `tailwind-merge` (used in packages/ui `cn()`) will override `h-svh` → sidebar now runs from `2.5rem` to the bottom of the viewport exactly. Footer is visible.

**Expected result:** Collapse toggle appears at the bottom of the sidebar on desktop.

### D-07: 'd' key keyboard shortcut to cycle theme

**Files:** New file `apps/web/src/hooks/useThemeKeyboardShortcut.ts` + `apps/web/src/routes/_layout.tsx`

**Behavior:** Pressing `d` (no modifier keys) anywhere in the app cycles the theme (system → light → dark → system). Does NOT fire when focus is inside `INPUT`, `TEXTAREA`, `SELECT`, or a `contenteditable` element.

**Hook implementation:**
```ts
// apps/web/src/hooks/useThemeKeyboardShortcut.ts
import { useEffect, useRef } from "react"
import { useTheme } from "next-themes"

const cycleMap = { system: "light", light: "dark", dark: "system" } as const
type Theme = keyof typeof cycleMap

export function useThemeKeyboardShortcut() {
  const { theme, setTheme } = useTheme()

  // Store handler in ref so the effect registers once but always reads latest theme
  // (advanced-event-handler-refs pattern — avoids re-registering on every theme change)
  const handlerRef = useRef<(e: KeyboardEvent) => void>()
  handlerRef.current = (e: KeyboardEvent) => {
    if (e.key !== "d") return
    if (e.metaKey || e.ctrlKey || e.altKey || e.shiftKey) return
    const target = e.target as HTMLElement
    if (
      target.tagName === "INPUT" ||
      target.tagName === "TEXTAREA" ||
      target.tagName === "SELECT" ||
      target.isContentEditable
    ) return
    setTheme(cycleMap[(theme ?? "system") as Theme])
  }

  useEffect(() => {
    const handler = (e: KeyboardEvent) => handlerRef.current?.(e)
    window.addEventListener("keydown", handler)
    return () => window.removeEventListener("keydown", handler)
  }, []) // empty deps — registers once; latest theme always available via ref
}
```

**Usage:** Call `useThemeKeyboardShortcut()` in `LayoutShell` (inside `_layout.tsx`).

### D-09: Dialog mobile padding — standardize RuleDialog and CategoryDialog width

**Files:** `apps/web/src/components/settings/RuleDialog.tsx`, `apps/web/src/components/settings/CategoryDialog.tsx`

**Problem:** `DialogContent` base class has `max-w-[calc(100%-2rem)] sm:max-w-sm`. When `className="max-w-md"` (RuleDialog) or `className="max-w-sm"` (CategoryDialog) is passed, `tailwind-merge` removes the base `max-w-[calc(100%-2rem)]` since they conflict at the same breakpoint. On mobile (<448px), `max-w-md` (448px) doesn't constrain the dialog → dialog fills the full screen width with no padding.

**Fix:** Use `w-[calc(100%-2rem)] max-w-md` for **both** dialogs. The `w-[calc(100%-2rem)]` is a `width` property (not `max-width`) so it doesn't conflict and ensures 1rem padding on each side on mobile screens. Both dialogs get the same standard width.

- RuleDialog: change `className="max-w-md"` → `className="w-[calc(100%-2rem)] max-w-md"`
- CategoryDialog: change `className="max-w-sm"` → `className="w-[calc(100%-2rem)] max-w-md"`

### D-11: Mobile sidebar auto-close on navigation

**File:** `apps/web/src/components/AppSidebar.tsx`

On mobile the sidebar is a sheet overlay. After tapping a nav item the route changes but the sheet stays open — the user must tap the overlay to close it.

**Fix:** Import `useSidebar` from `@ploutizo/ui/components/sidebar`. Store the sidebar context in a ref so `closeMobile` only registers once and never causes re-renders when `isMobile` changes (`rerender-defer-reads` — `isMobile` is only used inside the callback, not in render). Add `onClick={closeMobile}` to the `render` Link of every nav item (including Settings).

```tsx
const sidebarCtx = useSidebar()
const sidebarRef = useRef(sidebarCtx)
sidebarRef.current = sidebarCtx

const closeMobile = useCallback(() => {
  const { isMobile, setOpenMobile } = sidebarRef.current
  if (isMobile) setOpenMobile(false)
}, [])

// Each nav item:
render={<Link to={to} onClick={closeMobile} />}

// Settings item:
render={<Link to="/settings" onClick={closeMobile} />}
```

`useCallback` and `useRef` are imported from React. `useSidebar` is exported from `@ploutizo/ui/components/sidebar`.
