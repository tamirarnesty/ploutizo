## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Client sends pre-calculated amounts — `assignees: [{memberId, amountCents, percentage}]`. API validates sum of `amountCents` equals transaction `amount`. LRM is the UI's responsibility.
- **D-02:** If `assignees` omitted on create, transaction created with no assignees. No server-side auto-split.
- **D-03:** On PATCH, `assignees` is a full replace-all (delete existing, insert new). Same sum-validation applies.
- **D-04:** Joined response for all GET endpoints. Each transaction includes: `assignees: [{memberId, amountCents, percentage, memberName}]`, `tags: [{id, name, colour}]`, `categoryId + categoryName + categoryIcon`, `accountId + accountName + accountType`, all transaction table columns.
- **D-05:** GET `/api/transactions/:id` returns same shape as the list — no separate detail endpoint.
- **D-06:** Full filtering: `type`, `accountId`, `dateFrom/dateTo`, `categoryId`, `tagIds` (comma-sep or array, OR logic), `assigneeId`.
- **D-07:** Offset pagination: `?page=1&limit=50`. Response envelope: `{data: [...], total, page, limit}`.
- **D-08:** Configurable sort: `?sort=date|amount&order=asc|desc`. Default: `sort=date&order=desc`.
- **D-09:** `badRequest()` helper returns `{error: {code: 'BAD_REQUEST', message: string}}` with 400.
- **D-10:** Per-type field validation via Zod discriminated unions. Missing required fields return 400 with `{error: {code: 'VALIDATION_ERROR', errors: []}}`.
- **D-11:** Split sum mismatch returns 400 with `{error: {code: 'BAD_REQUEST', message: 'Assignee amounts must sum to transaction amount'}}`.

## Summary

Phase 03.2 adds a single Hono route file (`apps/api/src/routes/transactions.ts`) with full CRUD for all six transaction types. The Zod discriminated union validators are already complete in `@ploutizo/validators` — no validator work needed. The schema (`transactions`, `transactionAssignees`, `transactionTags`) is fully migrated from Phase 03.1 and exported from the `@ploutizo/db/schema` barrel.

The primary complexity is the joined GET response: each transaction row must be enriched with category name/icon, account name/type, assignee name from `orgMembers`, and tag name/colour. The recommended approach is a base list query followed by two parallel subqueries (assignees, tags) keyed on transaction IDs — avoiding a cartesian product from multi-level left joins on the list endpoint.

Test infrastructure mirrors every existing route test: Vitest with `@ploutizo/db` and `@hono/clerk-auth` mocked. The transaction mock will be more complex due to `db.transaction()` with multi-table writes plus the Drizzle query builder chain for joined reads.

**Primary recommendation:** Use a base query with `.where` clauses for filtering + pagination, then resolve assignees and tags in two separate `db.select()` calls batched in `Promise.all`. Return the merged result per transaction.

### Pattern 5: Joined GET Response — Recommended Approach [ASSUMED]

The Drizzle left-join approach on a list produces a cartesian product when a transaction has N assignees and M tags: N×M rows per transaction. Recommended approach: base list query + two parallel enrichment queries.

```typescript
// Step 1: base paginated query
const baseRows = await db
  .select({
    // transaction columns
    id: transactions.id,
    orgId: transactions.orgId,
    type: transactions.type,
    amount: transactions.amount,
    date: transactions.date,
    description: transactions.description,
    merchant: transactions.merchant,
    categoryId: transactions.categoryId,
    categoryName: categories.name,
    categoryIcon: categories.icon,
    accountId: transactions.accountId,
    accountName: accounts.name,
    accountType: accounts.type,
    // ... all other transaction columns
    deletedAt: transactions.deletedAt,
    createdAt: transactions.createdAt,
    updatedAt: transactions.updatedAt,
  })
  .from(transactions)
  .leftJoin(categories, eq(transactions.categoryId, categories.id))
  .leftJoin(accounts, eq(transactions.accountId, accounts.id))
  .where(and(
    eq(transactions.orgId, orgId!),
    isNull(transactions.deletedAt),
    // ... dynamic filters
  ))
  .orderBy(/* dynamic sort */)
  .limit(limit)
  .offset((page - 1) * limit)

const txIds = baseRows.map((r) => r.id)

// Step 2: parallel enrichment
const [assigneeRows, tagRows] = await Promise.all([
  txIds.length
    ? db.select({
        transactionId: transactionAssignees.transactionId,
        memberId: transactionAssignees.memberId,
        amountCents: transactionAssignees.amountCents,
        percentage: transactionAssignees.percentage,
        memberName: orgMembers.displayName,
      })
      .from(transactionAssignees)
      .innerJoin(orgMembers, eq(transactionAssignees.memberId, orgMembers.id))
      .where(inArray(transactionAssignees.transactionId, txIds))
    : Promise.resolve([]),

  txIds.length
    ? db.select({
        transactionId: transactionTags.transactionId,
        id: tags.id,
        name: tags.name,
        colour: tags.colour,
      })
      .from(transactionTags)
      .innerJoin(tags, eq(transactionTags.tagId, tags.id))
      .where(inArray(transactionTags.transactionId, txIds))
    : Promise.resolve([]),
])

// Step 3: merge in memory
const assigneeMap = groupBy(assigneeRows, 'transactionId')
const tagMap = groupBy(tagRows, 'transactionId')

const data = baseRows.map((row) => ({
  ...row,
  assignees: assigneeMap[row.id] ?? [],
  tags: tagMap[row.id] ?? [],
}))
```

Note: `inArray` imported from `drizzle-orm`. `groupBy` is a small local helper (no lodash needed).

### Pattern 6: Dynamic Where + Sort [ASSUMED based on Drizzle docs patterns]

```typescript
import { and, asc, desc, eq, gte, inArray, isNull, lte, SQL } from 'drizzle-orm'

// Build conditions array dynamically
const conditions: SQL[] = [
  eq(transactions.orgId, orgId!),
  isNull(transactions.deletedAt),
]
if (type) conditions.push(eq(transactions.type, type))
if (accountId) conditions.push(eq(transactions.accountId, accountId))
if (dateFrom) conditions.push(gte(transactions.date, dateFrom))
if (dateTo) conditions.push(lte(transactions.date, dateTo))
if (categoryId) conditions.push(eq(transactions.categoryId, categoryId))
if (assigneeId) {
  // Subquery: transaction must exist in transactionAssignees for this member
  conditions.push(
    exists(
      db.select({ one: sql`1` })
        .from(transactionAssignees)
        .where(and(
          eq(transactionAssignees.transactionId, transactions.id),
          eq(transactionAssignees.memberId, assigneeId)
        ))
    )
  )
}
if (parsedTagIds.length > 0) {
  conditions.push(
    exists(
      db.select({ one: sql`1` })
        .from(transactionTags)
        .where(and(
          eq(transactionTags.transactionId, transactions.id),
          inArray(transactionTags.tagId, parsedTagIds)
        ))
    )
  )
}

const orderCol = sort === 'amount' ? transactions.amount : transactions.date
const orderFn = order === 'asc' ? asc : desc

query.where(and(...conditions)).orderBy(orderFn(orderCol))
```

### Anti-Patterns to Avoid

- **Multi-level left join on list:** `transactions LEFT JOIN transactionAssignees LEFT JOIN transactionTags` produces N×M rows per transaction when a transaction has multiple assignees and multiple tags. Use the parallel sub-query approach instead.
- **Omitting `isNull(transactions.deletedAt)`:** The partial index `transactions_active_idx` does not enforce query scoping — every active-data query must include this condition explicitly.
- **Omitting `eq(transactions.orgId, orgId!)`:** Tenant isolation is mandatory on every query. The tenantGuard middleware only blocks unauthenticated requests — it does not add automatic WHERE clauses.
- **Using `percentage` field in balance math:** `transactionAssignees.percentage` is `numeric` — Drizzle returns it as a `string`. It is display-only. All math uses `amountCents`.
- **Inline string `percentage` coercion omission:** When inserting assignees, `percentage` must be converted: `a.percentage?.toString()`. The column is `numeric`, not `number`.

## Common Pitfalls

### Pitfall 1: Cartesian Product on Multi-Level Left Join
**What goes wrong:** Joining `transactionAssignees` AND `transactionTags` in the same query produces N×M rows per transaction (one row per assignee×tag combination). A transaction with 2 assignees and 3 tags generates 6 rows, duplicating monetary values.
**Why it happens:** SQL JOINs are set operations — they multiply, not aggregate.
**How to avoid:** Use the parallel sub-query approach (Pattern 5). Fetch base transaction rows, then fetch assignees and tags separately with `inArray(transactionId, txIds)`, merge in JavaScript.

### Pitfall 2: Forgetting deletedAt Filter
**What goes wrong:** Soft-deleted transactions appear in list and single-get results.
**Why it happens:** The partial index `transactions_active_idx` exists for performance — it does NOT automatically exclude deleted rows from queries.
**How to avoid:** Every query must include `.where(... isNull(transactions.deletedAt) ...)`.

### Pitfall 3: type Immutability on PATCH
**What goes wrong:** Allowing `type` to be patched breaks the discriminated union invariant and can leave type-specific columns in inconsistent state (e.g., a former `transfer` now typed `expense` but still has `toAccountId` set).
**How to avoid:** `updateTransactionSchema` already omits `type`. Document in code that type is immutable post-creation.

### Pitfall 4: percentage Column as String
**What goes wrong:** Attempting arithmetic on `transactionAssignees.percentage` (e.g., summing percentages to validate they equal 100) will fail at runtime — Drizzle returns `numeric` columns as JavaScript `string`.
**How to avoid:** Never use `percentage` in math. It is display-only. Validate splits exclusively using `amountCents`.

### Pitfall 5: tagIds Query Parameter Parsing
**What goes wrong:** `?tagIds[]=x&tagIds[]=y` vs `?tagIds=x,y` — Hono's `c.req.query('tagIds')` only returns a single string value. Array params require `c.req.queries('tagIds[]')` for the bracket syntax.
**How to avoid:** Support both: try `c.req.queries('tagIds[]')` first; fall back to splitting `c.req.query('tagIds')` on commas if non-empty. Filter out empty strings.

### Pitfall 6: inArray with Empty Array
**What goes wrong:** `inArray(col, [])` generates invalid SQL (`col IN ()`) and throws a Drizzle error.
**How to avoid:** Guard every `inArray` call: `txIds.length ? db.select()...where(inArray(..., txIds)) : Promise.resolve([])`.

### Pitfall 7: PATCH 404 Scope
**What goes wrong:** A PATCH that returns 0 rows could be due to wrong orgId (cross-tenant attempt) or genuinely missing ID. Both should return 404 — never expose which.
**How to avoid:** `.where(and(eq(transactions.id, id), eq(transactions.orgId, orgId!), isNull(transactions.deletedAt)))` — if 0 rows returned, always 404.

## Open Questions

1. **Single `helpers.ts` vs inline `badRequest`**
   - What we know: `badRequest()` is needed by transactions route and potentially future routes.
   - Recommendation: Create `apps/api/src/lib/helpers.ts` with `badRequest` exported — consistent with how existing routes share no utility functions (they are all simple enough). If future routes need it, they import from there.

2. **tagIds query param bracket vs comma syntax**
   - What we know: Hono handles both via `.query()` and `.queries()`. D-06 says "comma-separated UUIDs or repeated `?tagIds[]=x&tagIds[]=y`".
   - Recommendation: Support both syntaxes in the route — try `c.req.queries('tagIds[]')` first, fall back to comma-split of `c.req.query('tagIds')`.
