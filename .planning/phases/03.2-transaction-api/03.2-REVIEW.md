**Reviewed:** 2026-04-10

## Summary

Transaction API layer (routes, services, queries) plus supporting helpers and tests. The layered architecture is clean and the tenant-scoping pattern is consistent. Four logic issues found: NaN propagation in pagination params, incomplete split-sum validation on PATCH, a potentially bypassable `refundOf` ownership check, and three `as unknown as DrizzleTransaction` casts that suppress a real type mismatch. No security vulnerabilities or data-loss risks.

## Warnings

### WR-01: NaN propagates through pagination clamp for non-numeric query params

**File:** `apps/api/src/routes/transactions.ts:45-46`
**Issue:** `Number("abc")` returns `NaN`. `Math.max(1, NaN)` and `Math.min(200, NaN)` both return `NaN`, which then flows into `buildListQuery` as the `limit` and `offset` values. Drizzle will pass `NaN` to the SQL driver, resulting in a malformed query or driver error rather than a clean 400.
**Fix:**
```typescript
const rawPage = Number(c.req.query('page'))
const rawLimit = Number(c.req.query('limit'))
const page = Number.isFinite(rawPage) && rawPage >= 1 ? Math.floor(rawPage) : 1
const limit = Number.isFinite(rawLimit) ? Math.min(200, Math.max(1, Math.floor(rawLimit))) : 50
```

### WR-02: PATCH split-sum validation skipped when assignees are patched without amount

**File:** `apps/api/src/services/transactions.ts:107`
**Issue:** The guard `data.amount !== undefined` means that a PATCH that replaces `assignees` but omits `amount` skips sum validation entirely. The new assignees are written against a stored amount they may not match. This silently corrupts the split.
**Fix:** Fetch the existing transaction's amount when `data.amount` is absent and assignees are being replaced, then validate against the stored value:
```typescript
if (data.assignees && data.assignees.length > 0) {
  const amountToCheck = data.amount ?? (await fetchTransactionById(id, orgId))?.amount
  if (amountToCheck !== undefined) {
    const splitError = validateSplitSum(amountToCheck, data.assignees)
    if (splitError) throw new Error(splitError)
  }
}
```

### WR-03: `refundOf` ownership check only fires for `type === 'refund'`

**File:** `apps/api/src/routes/transactions.ts:32`
**Issue:** If the Zod `createTransactionSchema` permits `refundOf` on non-`'refund'` types (e.g., it's an optional field on the base shape), a caller can supply `refundOf` pointing to another org's transaction on an `expense` type and bypass the ownership check. The check is gated on `data.type === 'refund'` rather than on the presence of `data.refundOf`.
**Fix:** Gate on field presence, not type:
```typescript
if (data.refundOf) {
  const owned = await checkRefundOfOwnership(data.refundOf, orgId!)
  if (!owned) return badRequest(c, 'refundOf transaction not found in this org')
}
```
If the schema already enforces that `refundOf` can only be set when `type === 'refund'`, this is a non-issue — but the defence-in-depth check at the route level should still reflect that intent.

### WR-04: `tx as unknown as DrizzleTransaction` cast hides a type mismatch

**File:** `apps/api/src/services/transactions.ts:117, 127, 131`
**Issue:** Three double-cast sites suppress TypeScript's type checker rather than resolving the actual incompatibility between Drizzle's inferred transaction type and the exported `DrizzleTransaction` alias. If Drizzle's internal type changes across versions, this will break silently at runtime.
**Fix:** Align the `DrizzleTransaction` type definition with what `db.transaction()` actually infers. The cleanest approach is to derive the type from Drizzle's own inference rather than constructing it manually:
```typescript
// queries/transactions.ts
import type { Parameters } from 'drizzle-orm/neon-serverless'
// Use typeof db.transaction's callback parameter instead of manual construction
export type DrizzleTransaction = Parameters<Parameters<typeof db.transaction>[0]>[0]
```
This eliminates the cast entirely.

### IN-01: `buildListQuery` is declared `async` but contains no `await`

**File:** `apps/api/src/lib/queries/transactions.ts:113`
**Issue:** The function signature is `async function buildListQuery(...)` but the body returns a Drizzle query builder synchronously — there is no `await`. The `async` wrapper adds an unnecessary promise layer and is misleading. Callers `await` it correctly, so there is no runtime bug, but it signals intent that isn't there.
**Fix:** Remove `async`:
```typescript
export function buildListQuery(params: ListQueryParams) {
  // ...
  return db.select(TX_COLUMNS)...
}
```

### IN-02: Test mock for `db.transaction` does not simulate rollback on error

**File:** `apps/api/src/__tests__/transactions.test.ts:71-94`
**Issue:** The `db.transaction` mock simply invokes the callback and returns its result. There is no test covering the scenario where the callback throws — i.e., no test verifies that a mid-transaction failure (e.g., assignee insert fails) results in a 500 rather than a partial 201 with a broken state. This is a test coverage gap, not a runtime bug.
**Fix:** Add a test case that mocks `createTransaction` to throw and asserts the route returns 500 (or whatever the global error handler returns), to confirm the transaction boundary is exercised end-to-end.
