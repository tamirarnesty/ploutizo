# Phase 03.2: Transaction API - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

All six transaction types (expense, refund, income, transfer, settlement, contribution) can be created, read, updated, and soft-deleted via the Hono API. Field validation, split math validation, filtering, pagination, and sorting are all part of this phase. No UI work.

</domain>

<decisions>
## Implementation Decisions

### Split Input Contract
- **D-01:** Client sends pre-calculated amounts — `assignees: [{memberId, amountCents, percentage}]` in the create payload. API validates that sum of `amountCents` equals transaction `amount`. The Largest Remainder Method is the UI's (03.4) responsibility before submit.
- **D-02:** If `assignees` is omitted on create, the transaction is created with no assignees (unassigned). No server-side auto-split from memberIds.
- **D-03:** On PATCH, if `assignees` is provided it is a full replace-all (delete existing rows, insert new rows). Same sum-validation applies.

### GET Response Shape
- **D-04:** Joined response for all GET endpoints (list and single). Each transaction includes:
  - `assignees: [{memberId, amountCents, percentage, memberName}]` — memberName sourced from `orgMembers`
  - `tags: [{id, name, colour}]`
  - `categoryId` + `categoryName` + `categoryIcon`
  - `accountId` + `accountName` + `accountType`
  - All transaction table columns
- **D-05:** GET `/api/transactions/:id` returns the same shape as the list — no separate richer detail endpoint. 03.4 edit form uses list-shape data.

### Filter + Pagination
- **D-06:** Full filtering implemented in 03.2. Query params supported:
  - `type` — transaction type enum value
  - `accountId` — UUID
  - `dateFrom` / `dateTo` — ISO date strings (YYYY-MM-DD)
  - `categoryId` — UUID
  - `tagIds` — comma-separated UUIDs or repeated `?tagIds[]=x&tagIds[]=y`; OR logic (EXISTS in transaction_tags for any provided tag)
  - `assigneeId` — UUID (filters by member in transaction_assignees)
- **D-07:** Offset-based pagination: `?page=1&limit=50`. Response envelope: `{data: [...], total, page, limit}`.
- **D-08:** Configurable sort: `?sort=date|amount&order=asc|desc`. Default: `sort=date&order=desc`. Allows 03.3 DataGrid columns to be sortable server-side.

### Validation + Error Handling
- **D-09:** `badRequest()` helper added in this phase — reusable across all routes. Returns `{error: {code: 'BAD_REQUEST', message: string}}` with 400.
- **D-10:** Per-type field validation via Zod discriminated unions (already in `@ploutizo/validators`). Missing required fields (expense without categoryId, income without incomeType, transfer without toAccountId) return 400 with `{error: {code: 'VALIDATION_ERROR', errors: []}}`.
- **D-11:** Split sum mismatch (sum of assignee `amountCents` ≠ transaction `amount`) returns 400 with `{error: {code: 'BAD_REQUEST', message: 'Assignee amounts must sum to transaction amount'}}`.

### Business Logic
- **D-12:** Transfer transactions excluded from budget/expense/income calculations — enforced in Phase 4+ queries; no special handling at write time in this phase.
- **D-13:** Refund `refundOf` FK is optional. Unlinked refunds are valid. No validation that the linked transaction exists in the same org (that FK check happens at DB level via `onDelete: set null`). Wait — actually the API should validate orgId ownership of the refund_of transaction if provided.
- **D-14:** All SUM aggregates cast to bigint: `sql\`SUM(${transactions.amount})::bigint\`` — enforced wherever aggregate queries appear.
- **D-15:** All active-data queries filter `deletedAt IS NULL` via the partial index. Soft delete sets `deletedAt = now()`.

### Claude's Discretion
- Route file location: `apps/api/src/routes/transactions.ts`
- Exact Drizzle join query structure (left joins vs. subqueries for assignees/tags)
- Whether to use a single aggregate query or parallel queries for the joined response
- Tag filter implementation detail (EXISTS subquery vs. JOIN)
- How `refundOf` org-ownership is validated (select before insert, or rely on FK)

</decisions>

## Existing Code Insights

### Reusable Assets
- `createTransactionSchema` / `updateTransactionSchema` in `@ploutizo/validators` — already complete; do not rewrite
- `transactionTypeEnum`, `incomeTypeEnum`, `investmentTypeEnum` in `packages/db/src/schema/enums.ts` — use directly in Drizzle queries
- `getAuth(c)` from `@hono/clerk-auth` — standard orgId extraction pattern already in all routes
- `db` from `@ploutizo/db` — Neon WebSocket Pool + Drizzle client

### Established Patterns
- Error shape: `{error: {code: 'VALIDATION_ERROR', errors: result.error.issues}}` for Zod failures (all existing routes)
- Soft-delete pattern: partial index `transactions_active_idx` on `deleted_at IS NULL` — all queries must `.where(isNull(transactions.deletedAt))`
- Tenant isolation: every query includes `.where(eq(table.orgId, orgId!))` — never omit
- DB transactions for multi-table writes: `db.transaction(async (tx) => {...})` (accounts.ts pattern)

### Integration Points
- `apps/api/src/index.ts` — new `transactionsRouter` registered at `/api/transactions`
- `packages/db/src/schema/index.ts` — `transactionAssignees` and `transactionTags` must be exported if not already
- `tenantGuard()` middleware already covers `/api/*` — no additional auth needed in the route handler beyond `orgId` extraction

</code_context>
