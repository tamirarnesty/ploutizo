# Phase 03.4.4: App shell resilience — 404 page & error boundary - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Add `notFoundComponent` and `errorComponent` to the TanStack Router root route (`__root.tsx`). Users never see a blank screen on unknown routes or unhandled render errors. Both components render outside the `_layout` shell — no sidebar, standalone full-page layout, same level as Onboarding.

</domain>

<decisions>
## Implementation Decisions

### 404 Page — Visual Design
- **D-01:** Terminal/dot-art aesthetic. Status bar across the top (left: `● STATUS — 404`, right: current path rendered as `ploutizo/~/missing`-style). Centerpiece: "404" rendered in braille/dot characters as a large art block. Label below art: `ERROR · 0x404` in spaced monospace. Heading + subtext below that. Action buttons at bottom.
- **D-02:** Art subject: `"404"` in Unicode dot/braille style characters. This is the primary visual element.
- **D-03:** Theme: follows the app theme (light/dark/system) — dot art and status bar use `text-muted-foreground` / `bg-background` tokens. No forced dark mode.
- **D-04:** Single action button: "Go to Dashboard" → `/dashboard`.

### Error Page — Visual Design
- **D-05:** Same terminal/dot-art aesthetic as the 404 page for visual consistency. Status bar top (left: `● STATUS — ERROR`). Art subject and exact copy: Claude's Discretion (e.g., "500" or "ERR" in dot art).
- **D-06:** Theme: follows app theme (same as 404).

### Error Page — Recovery Actions
- **D-07:** Two actions: "Try again" (calls `reset()` from TanStack Router's `ErrorComponentProps`) + "Go to Dashboard" (link to `/dashboard`). No hard reload button.

### Dev Mode Error Detail
- **D-08:** In `import.meta.env.DEV` only: render `error.message` as a monospace line below the action buttons, plus a "Show details" toggle that expands `error.stack` in a scrollable `<pre>` block. Collapsed by default so it doesn't overwhelm the page. Prod shows nothing beyond the user-facing message.

### Implementation Notes
- `notFoundComponent` and `errorComponent` are wired on `createRootRoute({})` in `apps/web/src/routes/__root.tsx`.
- Both components are standalone files imported into `__root.tsx` — not inlined.
- Use `Text` component from `@ploutizo/ui/components/text` for heading/subtext. Use shadcn `Button` for action buttons.
- Dot/braille art rendered as a `<pre>` block with `font-family: monospace`. Use `text-muted-foreground` at reduced opacity (e.g. `opacity-40`) to give the subtle dot effect matching the reference.
- Status bar and `ERROR · 0x404` label use spaced uppercase monospace text (`tracking-widest font-mono text-xs text-muted-foreground`).

### Claude's Discretion
- Exact dot/braille character art for "404" and the error page symbol (craft to look good at a reasonable font size)
- Exact heading and subtext copy for each page
- Whether to include a muted trace-style footer line at the bottom — use judgment based on how it looks (no real trace ID exists yet; see Deferred Ideas)
- Exact spacing and sizing of the art block relative to the page height

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Text` component (`@ploutizo/ui/components/text`): use `as="h1" variant="h1"` for the heading, `variant="body-sm"` + `className="text-muted-foreground"` for subtext.
- shadcn `Button` (`@ploutizo/ui/components/button`): use `variant="outline"` for secondary actions, default for primary.
- `createRootRoute` already exists in `__root.tsx` — add `notFoundComponent` and `errorComponent` keys to the existing options object.

### Established Patterns
- Standalone full-page centering: `<div className="flex min-h-screen items-center justify-center bg-background">` — matches Onboarding.
- Monospace/terminal text: `font-mono text-xs tracking-widest text-muted-foreground` — consistent with terminal aesthetic in shadcn components.
- Theme tokens: always use `bg-background`, `text-foreground`, `text-muted-foreground` — never hardcode colors.

### Integration Points
- `apps/web/src/routes/__root.tsx`: add two new keys to `createRootRoute({})` options.
- TanStack Router `ErrorComponentProps` type (from `@tanstack/react-router`): provides `error: Error` and `reset: () => void`.
- `notFoundComponent` receives no special props — it's a plain React component.

</code_context>

<specifics>
## Specific Ideas

- Reference design: terminal/hacker 404 page with braille/dot-art "404" centerpiece, `ERROR · 0x404` label, status bar at top, action buttons at bottom. The dot art uses Unicode braille or dot characters in a `<pre>` block at reduced opacity.
- "This page wandered off." style copy — friendly but technical-flavored, matching the aesthetic.
- Status bar right side can show a mocked path: `ploutizo/~/missing` (static string, not the actual URL, for cleaner aesthetics — or use actual `window.location.pathname` if it looks good).

</specifics>

<deferred>
## Deferred Ideas

### Request ID / Trace ID infrastructure
Add a request ID concept for FE/BE correlation and future logging relationships. Would require:
- Hono middleware that generates a UUID per request and attaches it as a response header (e.g. `X-Request-Id`)
- `apiFetch` in `apps/web/src/lib/queryClient.ts` reads and surfaces the ID on error
- Error boundary and/or dev tools can display the ID, enabling correlation between client errors and API logs

This is a cross-cutting infrastructure concern — not scoped to the error page UI. Should be its own phase, ideally before a logging/observability phase.

</deferred>
