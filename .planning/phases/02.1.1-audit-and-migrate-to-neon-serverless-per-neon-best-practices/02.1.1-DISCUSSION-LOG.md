## Background / Context Request

The user asked for context before selecting gray areas: why the current implementation was
chosen, what its pros and cons are, and why we would or would not want transactions.

**Summary of explanation provided:**

- Current `postgres.js` was chosen because: "persistent Node.js server → postgres.js is
  correct; neon-serverless WebSocket driver is unnecessary" (REQUIREMENTS.md).
- The original decision conflated `neon-http` (HTTP-only, no transactions) with
  `@neondatabase/serverless` in WebSocket mode (full transaction support).
- Three drivers exist: `postgres.js` (TCP pool), `@neondatabase/serverless` HTTP mode (no
  transactions), `@neondatabase/serverless` WebSocket mode (full transactions + Neon's
  recommended approach for all environments).
- `postgres.js` holds TCP connections that prevent Neon autosuspend; the WebSocket driver
  reconnects cleanly when Neon's compute wakes/suspends.
- Transactions are required for: webhook seed atomicity (current), import batch (future),
  settlement recording (future), CSV deduplication (future).
