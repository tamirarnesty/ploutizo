<decisions>
## Implementation Decisions

### Driver

- **D-01:** Use `@neondatabase/serverless` in **WebSocket mode** with `drizzle-orm/neon-serverless`.
  This is Neon's officially recommended approach for all environments, including persistent
  Node.js servers.
- **D-02:** `postgres.js` is removed entirely. The `postgres` package is removed from
  `packages/db/package.json`. `@neondatabase/serverless` replaces it.
- **D-03:** Override of prior decision in STATE.md: "Use `postgres.js` direct Neon connection
  (not `neon-http`)" — that decision conflated neon-http (HTTP-only, no transactions) with
  the WebSocket mode of `@neondatabase/serverless` (full transaction support). WebSocket mode
  is the correct Neon-recommended driver and supersedes that earlier decision.

### Why WebSocket mode, not HTTP mode

- **D-04:** HTTP mode (`neon()` fetch transport) has no transaction support — same reason
  `neon-http` was rejected originally. WebSocket mode (`Pool` from `@neondatabase/serverless`)
  has full transaction support and is suitable for the persistent Railway/Hono server.
- **D-05:** The WebSocket driver allows Neon's compute to autosuspend (postgres.js held open
  TCP connections that prevented scale-to-zero). WebSocket reconnects cleanly when Neon wakes.

### neonConfig setup

- **D-06:** Set `neonConfig.webSocketConstructor = globalThis.WebSocket` in `client.ts`
  before constructing the Pool. Node 22 (this project's runtime) has native WebSocket
  globally — no `ws` package needed.

### Connection lifecycle

- **D-07:** Keep the module-scoped singleton pattern. The Pool is created once at module
  load, same as the current `postgres.js` client. No per-request connections.

### Route audit scope

- **D-08:** Audit all existing route files for postgres.js-specific patterns:
  - Raw SQL tagged template literals (`sql`...`` from postgres.js` — not the Drizzle `sql`)
  - postgres.js `BEGIN`/`COMMIT`/`ROLLBACK` direct calls
  - Any `client.end()` or `client.unsafe()` calls
  Drizzle ORM query methods (`.select()`, `.insert()`, `.update()`, `.delete()`) are
  driver-agnostic and require no changes. Only driver-specific patterns need updating.

### Test strategy

- **D-09:** Mirror the current mock approach in `client.test.ts`. Replace:
  - `vi.mock('postgres', ...)` → `vi.mock('@neondatabase/serverless', ...)`
  - `vi.mock('drizzle-orm/postgres-js', ...)` → `vi.mock('drizzle-orm/neon-serverless', ...)`
  Same unit test pattern, same coverage intent. The test verifies the client initializes
  and that the Pool is constructed with `DATABASE_URL`.

### Claude's Discretion

- Exact Pool constructor options (connection limit, idle timeout) — start with Neon defaults
- Whether to use `Pool` or `neon` export from `@neondatabase/serverless` (Pool is correct
  for Drizzle's `drizzle-orm/neon-serverless` adapter in WebSocket mode)
- Mock implementation shape for the new vi.mock calls

</decisions>

## Existing Code Insights

### Current implementation (to replace)
- `packages/db/src/client.ts` — Module-scoped `postgres()` call with `max: 10`, wrapped by
  `drizzle({ client, schema })`. Replace with `new Pool({ connectionString: DATABASE_URL })`
  from `@neondatabase/serverless` and `drizzle()` from `drizzle-orm/neon-serverless`.
- `drizzle.config.ts` — Uses `url: DATABASE_URL` directly (no runtime driver import). No
  changes needed — Drizzle Kit is independent of the runtime client.

### Established patterns
- `packages/db` is consumed only by `apps/api` — it is already Node.js-specific; adding
  `neonConfig.webSocketConstructor = globalThis.WebSocket` in client.ts is appropriate.
- All route files import `db` from `@ploutizo/db` — no direct `postgres.js` imports in routes.
- Drizzle ORM query methods are driver-agnostic; routes are unlikely to have postgres.js-
  specific patterns, but the audit will confirm.

### Integration points
- `apps/api/src/routes/*.ts` — All import `db` from `@ploutizo/db`. After client.ts swap,
  these continue to work without changes (Drizzle ORM API is identical across drivers).
