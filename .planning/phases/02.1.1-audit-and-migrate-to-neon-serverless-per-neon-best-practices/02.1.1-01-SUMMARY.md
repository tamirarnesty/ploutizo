## Decisions Made
- **WebSocket Pool over TCP:** Chosen explicitly for scale-to-zero even though Neon docs recommend TCP for persistent servers (D-05 reasoning in CONTEXT.md)
- **No ws package:** Node 22 has `globalThis.WebSocket` natively — `ws` package not needed
- **connectionString only:** No `max: 10` or other postgres.js pool options — neon serverless Pool manages its own connection per request

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed Pool constructor mock — arrow function not constructable**
- **Found during:** Task 2 GREEN (first test run after client.ts rewrite)
- **Issue:** Plan spec used `vi.fn(() => ({}))` (arrow function) as Pool mock factory. Arrow functions cannot be used as constructors — `new Pool(...)` throws `TypeError: () => ({}) is not a constructor`
- **Fix:** Changed to `vi.fn(function(this) { return this })` — regular function is constructable
- **Files modified:** packages/db/src/__tests__/client.test.ts
- **Verification:** `pnpm --filter @ploutizo/db test` exits 0, all 6 tests pass
- **Committed in:** a948fef (Task 2 GREEN commit)
