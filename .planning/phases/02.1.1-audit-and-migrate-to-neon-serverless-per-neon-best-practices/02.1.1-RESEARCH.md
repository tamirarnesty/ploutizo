### Locked Decisions

- **D-01:** Use `@neondatabase/serverless` in **WebSocket mode** with `drizzle-orm/neon-serverless`. Neon's officially recommended approach for all environments, including persistent Node.js servers.
- **D-02:** `postgres.js` is removed entirely. The `postgres` package is removed from `packages/db/package.json`. `@neondatabase/serverless` replaces it.

This phase is a **surgical driver swap** inside `packages/db`. The only source file that changes functionally is `packages/db/src/client.ts`. The dependency change is `postgres` → `@neondatabase/serverless` in `packages/db/package.json`. The test file `client.test.ts` needs its mocks updated to reflect the new imports. No schema changes, no API changes, no route logic changes.

The route audit (D-08) is expected to find **no postgres.js-specific patterns** — all six route files confirmed to use Drizzle ORM query methods exclusively (`.select()`, `.insert()`, `.update()`, `.delete()`, `.transaction()`). None import from `postgres` directly. The audit is a verification step, not remediation work.

One important note on ecosystem context: Neon's own connection-methods guide and Drizzle's official docs both recommend TCP (`postgres.js` or `node-postgres`) for long-running Node.js servers. The WebSocket mode is primarily designed for edge/serverless runtimes without TCP. The user has reviewed this tradeoff and explicitly chosen WebSocket mode for the scale-to-zero benefit (D-05). The planner should document this rationale in code comments.

**Primary recommendation:** Swap `packages/db/src/client.ts` to use `Pool` from `@neondatabase/serverless` with `neonConfig.webSocketConstructor = globalThis.WebSocket`, update `drizzle()` call to import from `drizzle-orm/neon-serverless`, update package.json deps, update test mocks. Confirm route files are clean via grep audit.

### Pattern 1: WebSocket Pool Singleton (Client Module)

**What:** Module-scoped Pool instantiation with `neonConfig` set before Pool construction.

**When to use:** Long-running Node.js server with singleton DB client.

**Example:**
```typescript
// Source: Neon official skill docs + drizzle-orm/neon-serverless docs
import { Pool, neonConfig } from '@neondatabase/serverless'
import { drizzle } from 'drizzle-orm/neon-serverless'
import * as schema from './schema/index.js'

// Node 22 has native WebSocket — no 'ws' package needed
neonConfig.webSocketConstructor = globalThis.WebSocket

const pool = new Pool({ connectionString: process.env.DATABASE_URL! })

export const db = drizzle({ client: pool, schema })
```

**Note on `drizzle()` call signature:** Drizzle docs show both `drizzle(pool)` and `drizzle({ client: pool })`. Use the object form (`{ client: pool, schema }`) to match the existing pattern in `client.ts` and enable schema-aware queries.

### Pattern 2: Route Transaction Usage (No Change Required)

**What:** All existing Drizzle `db.transaction(async (tx) => { ... })` calls in routes work identically across drivers — `drizzle-orm/neon-serverless` implements the same transaction API as `drizzle-orm/postgres-js`.

**Confirmed in:** `accounts.ts` (2 transaction blocks), `categories.ts` (1), `merchant-rules.ts` (1). No changes needed.

### Pattern 3: Test Mock Structure

**What:** Mirror the existing vi.mock pattern — mock the driver and ORM adapter, assert Pool is constructed with DATABASE_URL.

**Example (updated from current `client.test.ts`):**
```typescript
// Source: mirrors current client.test.ts structure (D-09)
vi.mock('@neondatabase/serverless', () => ({
  Pool: vi.fn(() => ({})),
  neonConfig: {},
}))
vi.mock('drizzle-orm/neon-serverless', () => ({
  drizzle: vi.fn(() => ({ _: 'drizzle-instance' })),
}))

// Assertion:
const { Pool } = (await import('@neondatabase/serverless')) as unknown as { Pool: ReturnType<typeof vi.fn> }
await import('../client.js')
expect(Pool).toHaveBeenCalledWith({ connectionString: 'postgresql://test:test@localhost/test' })
```

**Key difference from current test:** Current test asserts `postgres(url, { max: 10 })`. New test asserts `new Pool({ connectionString: url })`. Remove the `max: 10` assertion — Neon defaults will be used (D-Discretion).

### Anti-Patterns to Avoid

- **Setting `neonConfig.webSocketConstructor = ws` and installing `ws`:** Not needed on Node 22. Adds unnecessary dependency.
- **HTTP mode (`neon()` instead of `Pool`):** No transaction support. Rejected by D-04.
- **Per-request Pool instantiation:** Neon's own serverless guidance says "create, use, and close within same invocation" — but this project uses a persistent server with a singleton. The singleton pattern (D-07) is correct here.
- **Changing `drizzle.config.ts`:** Drizzle Kit uses the `url` field directly — it does NOT use the runtime `Pool` or `client.ts`. Zero changes needed there.

## Common Pitfalls

### Pitfall 1: `neonConfig` must be set before `new Pool()`

**What goes wrong:** Pool is constructed before `neonConfig.webSocketConstructor` is set — Pool uses HTTP mode or throws because no WebSocket constructor is registered.

**Why it happens:** Module-level code runs top-to-bottom. If `new Pool(...)` appears before `neonConfig.webSocketConstructor = ...`, the Pool is initialized without WebSocket support.

**How to avoid:** Always set `neonConfig.webSocketConstructor` on the line immediately before `new Pool(...)`.

**Warning signs:** Runtime error "WebSocket is not defined" or silent fallback to HTTP mode (transactions fail with "no transaction support").

### Pitfall 2: vi.mock for `@neondatabase/serverless` must mock `neonConfig` as a mutable object

**What goes wrong:** Mock returns `neonConfig: undefined` or `neonConfig: {}` frozen — `client.ts` tries to set `neonConfig.webSocketConstructor` and throws.

**Why it happens:** The default Vitest auto-mock doesn't preserve the mutability of exported objects.

**How to avoid:** Mock factory must return `neonConfig: {}` as a plain writable object.

**Warning signs:** Test throws `TypeError: Cannot set property 'webSocketConstructor' of undefined`.

### Pitfall 3: Pool constructor argument shape differs from postgres.js

**What goes wrong:** Passing `max: 10` or other postgres.js options to `new Pool(...)`.

**Why it happens:** Muscle memory from the old `postgres(url, { max: 10 })` call.

**How to avoid:** `new Pool({ connectionString: process.env.DATABASE_URL! })` — only `connectionString` needed for basic setup. Neon defaults handle pool sizing.

**Warning signs:** TypeScript error — `max` is not in `PoolConfig` type from `@neondatabase/serverless`.

### Pitfall 4: Forgetting to update `drizzle-orm` import path (not just the driver)

**What goes wrong:** `client.ts` imports `drizzle` from `drizzle-orm/postgres-js` but constructs a `Pool` — Drizzle's postgres-js adapter doesn't understand a `Pool` instance.

**Why it happens:** Partial edit — changed the driver import but not the Drizzle adapter import.

**How to avoid:** Both imports must change together: `postgres` → `@neondatabase/serverless` AND `drizzle-orm/postgres-js` → `drizzle-orm/neon-serverless`.

**Warning signs:** TypeScript error on the `drizzle({ client: pool })` call — type mismatch between `Pool` and expected `postgres.Sql`.

### Pitfall 5: `seeds.test.ts` doesn't need changes — confirm before touching

**What goes wrong:** Unnecessary edits to `seeds.test.ts` that mock `client.ts` directly (not the driver packages) — those mocks are already driver-agnostic.

**Why it happens:** Over-broad refactor scope.

**How to avoid:** `seeds.test.ts` mocks `'../client.js'` directly — it doesn't know or care about the underlying driver. It is unaffected by this change. Do not modify it.

## Open Questions

1. **Neon Pool default connection limits**
   - What we know: `new Pool({ connectionString })` uses Neon defaults; exact defaults not documented in skill refs
   - What's unclear: Whether Neon's free/launch tier connection limits require a smaller pool cap than postgres.js `max: 10`
   - Recommendation: Start with defaults. If connection exhaustion occurs on Railway deploy, revisit with explicit `max` option. REQUIREMENTS.md already flags "Neon connection limits — verify current plan tier limits before launching."

2. **`drizzle()` call signature variant**
   - What we know: Both `drizzle(pool)` and `drizzle({ client: pool, schema })` are valid per official docs
   - What's unclear: Whether the `drizzle(pool)` shorthand vs `drizzle({ client: pool, schema })` has schema-inference implications
   - Recommendation: Use `drizzle({ client: pool, schema })` — matches existing pattern, ensures schema-typed queries work.
