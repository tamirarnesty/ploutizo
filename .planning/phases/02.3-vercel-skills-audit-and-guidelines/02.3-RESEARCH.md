### Pattern 1: TanStack Query as SWR Equivalent

**What:** TanStack Query already satisfies the `client-swr-dedup` rule intent — queries with the same key are deduplicated across component instances.

**When to use:** All data fetching hooks already follow this pattern. No SWR migration needed.

**SPA-specific `staleTime` configuration:**

```typescript
// apps/web/src/lib/queryClient.ts — add staleTime default
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 30_000, // 30s stale-while-revalidate
    },
  },
})
```

**Confidence:** HIGH — verified against TanStack Query docs behavior.

### Pattern 2: Parallel Fetches via useQueries (waterfall fix)

**What:** `AccountForm` loads `useGetAccountMembers` first, then mounts `AccountFormInner` which loads `useGetOrgMembers`. This is a waterfall — two sequential renders.

**Fix:** Use `useQueries` or fire both hooks at the top level without early-return gating.

```typescript
// Source: TanStack Query useQueries pattern
import { useQueries } from "@tanstack/react-query"

const results = useQueries({
  queries: [
    accountMembersQueryOptions(accountId),
    orgMembersQueryOptions(),
  ],
})
const [accountMembersResult, orgMembersResult] = results
```

**Alternative fix:** Remove the `isLoading` early-return guard from `AccountForm` and render `AccountFormInner` immediately; let TanStack Query handle the pending state inside the inner component with skeleton UI.

### Pattern 3: Suspense Boundaries for Route-Level Loading

**What:** TanStack Router supports `pendingComponent` on routes and React Suspense wrappers. Currently no Suspense boundaries exist — loading state is handled inline with `isLoading` flags.

**SPA context:** The `async-suspense-boundaries` rule as written targets RSC async components. For a SPA with TanStack Query, the equivalent pattern is `useSuspenseQuery` + `<Suspense fallback={<Skeleton />}>` at the route component level.

**Current state:** All components use `useQuery` (not `useSuspenseQuery`) and handle loading state manually. This is an audit decision point: either migrate to `useSuspenseQuery` + Suspense, or accept `useQuery` + manual skeleton as compliant for a SPA.

**Recommendation:** For the current app size, `useQuery` + skeleton is acceptable. Document this SPA adaptation in the pruned SKILL.md rather than force-migrating to `useSuspenseQuery`.

### Pattern 4: Barrel Import Policy for data-access

**What:** `apps/web/src/lib/data-access/*/index.ts` files are internal barrels that re-export hooks. Components import from these barrels (`import { useGetAccounts } from "@/lib/data-access/accounts"`).

**Assessment:** These are internal module boundaries, not external library barrels. The `bundle-barrel-imports` rule targets external library barrels (lucide-react, @mui/material) that prevent tree-shaking. Internal barrels within the same bundle are tree-shaken by Vite and do not have the import cost penalty described in the rule.

**Action:** The data-access barrels are acceptable. No change required. Document this SPA/Vite-specific exception in the pruned SKILL.md.

### Pattern 5: React 19 — No `forwardRef`

**What:** Project is on React 19.2.4. The `react19-no-forwardref` rule applies. Refs can be passed as regular props.

**Current state:** `packages/ui` has shadcn components. Shadcn generates `forwardRef` wrappers. These need to be checked — if any custom components in `apps/web` use `forwardRef`, they should be migrated. Shadcn components from the library are third-party and less urgent.

### Anti-Patterns to Avoid

- **`setState` during render:** `CategoriesSettings.tsx` lines 55-57 calls `setLocalCategories(categories)` inside the render body (not in an effect) as a synchronous state initialization. This triggers an extra render cycle. Fix: use `useState(categories)` lazy initializer or `useEffect` for sync.
- **Inline component definitions:** `__root.tsx` defines `TokenInitializer` and `RootDocument` as `const` functions inside the module scope (not inside another component's render) — these are module-level const declarations, NOT inline components. They are acceptable. Confirmed clean.
- **`&&` conditionals with numeric left side:** Multiple files use `{count && <X />}` patterns. These are safe if the condition is always boolean (TypeScript prevents `number && JSX` when types are correct). The `rendering-conditional-render` rule specifically warns about `count && ...` where `count` is a number. Check `field.state.meta.errors.length > 0 &&` patterns — `length > 0` produces a boolean, so these are safe.

## Common Pitfalls

### Pitfall 1: Misapplying RSC Patterns to TanStack Start SPA

**What goes wrong:** Agent reads `async-suspense-boundaries` and tries to make components `async` and `await` data directly — this is an RSC pattern that doesn't work in a client SPA.

**Why it happens:** The rule file uses Next.js RSC `async function Page()` examples.

**How to avoid:** The pruned SKILL.md (D-01) must include a SPA adaptation note: "For SPAs, use `useSuspenseQuery` + `<Suspense>` instead of `async/await` in components."

**Warning signs:** Agent writes `async function Dashboard() { const data = await fetch(...) }` — this would work in Next.js RSC but causes a runtime error in a client component.

### Pitfall 2: Over-Aggressively Removing the `bundle-barrel-imports` Rule

**What goes wrong:** Agent removes all barrel imports including internal `data-access/*/index.ts` barrels, forcing imports like `import { useGetAccounts } from "@/lib/data-access/accounts/useGetAccounts"`.

**Why it happens:** Rule says "import directly from source files." Internal barrels are not the problem — external library barrels are.

**How to avoid:** The rule fix applies to external library imports (lucide-react, if Vite's `optimizeDeps` isn't sufficient). Internal app barrels bundled by Vite are safe. Document this distinction explicitly in the pruned rule.

### Pitfall 3: `"use client"` Directive Removal Side Effects

**What goes wrong:** Agent removes `"use client"` from `packages/ui` components. These directives are harmless in Vite SPA context (they're no-ops) but removing them could cause issues if the UI package is ever used in an RSC-capable environment.

**Why it happens:** Directives look like dead code in a SPA context.

**How to avoid:** Leave `"use client"` directives in `packages/ui` unchanged. Only audit `apps/web` for RSC-specific patterns.

### Pitfall 4: `CategoriesSettings` setState-During-Render Fix Introducing Infinite Loop

**What goes wrong:** Naive fix for the `setLocalCategories` during render (lines 55-57) — moving it to a `useEffect` with `categories` as dependency triggers on every refetch/mutation response.

**Why it happens:** The local state is used for optimistic reordering. The original code "initializes once" but using `useEffect` would re-sync on every server invalidation, wiping the optimistic order.

**How to avoid:** Use `useState(() => initialValue)` lazy initializer pattern, or restructure to derive from server state via `useMemo` with a separate drag-pending flag.

### Pitfall 5: web-design-guidelines Fetch Failing Silently

**What goes wrong:** The `web-design-guidelines` SKILL.md instructs the agent to fetch from `https://raw.githubusercontent.com/vercel-labs/web-interface-guidelines/main/command.md` before each review. If the fetch fails (network unavailable in sandbox), the audit runs against no rules.

**Why it happens:** Network access may be restricted during planning/execution.

**How to avoid:** Verify the URL is reachable before audit runs. The full guideline content was retrieved during research (URL confirmed working as of 2026-04-03). Consider caching the fetched guidelines locally for the duration of the phase.

### Parallel TanStack Query Fetches (fixes AccountForm waterfall)

```typescript
// Remove the AccountForm shell + AccountFormInner split.
// Fire both queries from one component.
import { useQuery } from "@tanstack/react-query"

export const AccountForm = ({ account, onClose }: AccountFormProps) => {
  // Both queries fire simultaneously — no sequential waterfall
  const { data: existingMembers = [], isLoading: membersLoading } =
    useGetAccountMembers(account?.id ?? null)
  const { data: members = [], isLoading: orgLoading } = useGetOrgMembers()

  if (membersLoading || orgLoading) {
    return <div className="flex flex-1 items-center justify-center"><Spinner /></div>
  }

  return <AccountFormInner account={account} existingMembers={existingMembers} orgMembers={members} onClose={onClose} />
}
```

### useState Lazy Initializer (fixes CategoriesSettings double render)

```typescript
// Instead of: setLocalCategories(categories) during render
// Use: derive state once with a ref-based initialization
const [localCategories, setLocalCategories] = useState<Array<Category>>([])
const initializedRef = useRef(false)

// On first successful data load, initialize local state once
useEffect(() => {
  if (!initializedRef.current && !catLoading && categories.length > 0) {
    initializedRef.current = true
    setLocalCategories(categories)
  }
}, [catLoading, categories])
```

## Open Questions

1. **`async-suspense-boundaries` SPA equivalent**
   - What we know: The rule targets async RSC components. TanStack Query provides `useSuspenseQuery` as the SPA equivalent.
   - What's unclear: Whether migrating from `useQuery` to `useSuspenseQuery` provides meaningful UX improvement for this app's query sizes and network conditions.
   - Recommendation: Keep `useQuery` + skeleton approach as default; document `useSuspenseQuery` as an optional upgrade in the pruned SKILL.md. Do not force-migrate all queries in this phase.

2. **`bundle-barrel-imports` for lucide-react with Vite**
   - What we know: Vite's `optimizeDeps` pre-bundles lucide-react for fast dev HMR. The barrel cost described in the rule (200-800ms cold start) is most acute for SSR/cold-start environments like Next.js serverless functions.
   - What's unclear: Whether Vite production builds tree-shake lucide-react named imports correctly.
   - Recommendation: Add `lucide-react` to `vite.config.ts` `optimizeDeps.include` as a low-risk improvement. Do not rewrite all icon imports to subpaths (TypeScript type issues with subpath imports for some lucide icons).

3. **forwardRef audit scope**
   - What we know: React 19 deprecates `forwardRef`. The `react19-no-forwardref` rule applies.
   - What's unclear: Whether shadcn-generated components in `packages/ui` should be migrated.
   - Recommendation: Migrate only custom components written for this project in `apps/web`. Leave shadcn-generated components in `packages/ui` as-is (third-party generated code, migration risk outweighs benefit at this phase).
