## Summary

This review covers the full file set for Phase 02.3. The following files are clean and require no changes: `AppSidebar.tsx`, `AccountSheet.tsx`, `Accounts.tsx`, `AccountsTable.tsx`, `Dashboard.tsx`, `Onboarding.tsx`, `CategoryDialog.tsx`, `HouseholdSettings.tsx`, `MerchantRuleRow.tsx`, `RuleDialog.tsx`, `queryClient.ts` (one info item noted), `__root.tsx`, `_layout.accounts.tsx`, `_layout.dashboard.tsx`, `_layout.tsx`, `index.tsx`, `onboarding.tsx`.

Issues cluster in three areas:

1. **`rendering-conditional-render` violations** — `&&` is used throughout in place of ternary operators for conditional JSX rendering. Thirteen instances across seven files. CLAUDE.md mandates ternaries when the false branch should render nothing.

2. **`animate-pulse` without `motion-safe:` guard** — Three skeleton loader elements across two files use bare `animate-pulse`. CLAUDE.md requires all `animate-*` classes to respect `prefers-reduced-motion`.

3. **Waterfall: `isLoading` early return gates sibling query hooks** — `AccountForm.tsx` returns a spinner while `useGetAccountMembers` loads, preventing the `useGetOrgMembers` query in `AccountFormInner` from firing in parallel. This is a `async-*` waterfall pattern violation.

Additional: a dead exported component (`LayoutShell.tsx`), missing `autoComplete` attributes across all text inputs, and a missing runtime guard on `VITE_API_URL`.

No critical security vulnerabilities were found.

## Warnings

### WR-01: `isLoading` Early Return Creates Waterfall — `AccountForm.tsx`

**File:** `apps/web/src/components/accounts/AccountForm.tsx:62-73`
**Issue:** `AccountForm` returns a spinner while `useGetAccountMembers` is loading. This prevents `AccountFormInner` from mounting, which means the `useGetOrgMembers` query (line 78 of `AccountFormInner`) does not fire until after the members query resolves. This is the exact waterfall pattern `async-*` rules prohibit: an `isLoading` early return that gates other query hooks from mounting. Both queries are independent and should fire in parallel.
**Fix:** Hoist both queries into `AccountForm` and pass the data down, removing the early return:
```tsx
export const AccountForm = ({ account, onClose }: AccountFormProps) => {
  const { data: existingMembers = [], isLoading: membersLoading } =
    useGetAccountMembers(account?.id ?? null)
  const { data: members = [], isLoading: membersOrgLoading } = useGetOrgMembers()

  if (membersLoading || membersOrgLoading) {
    return (
      <div className="flex flex-1 items-center justify-center">
        <Spinner />
      </div>
    )
  }

  return (
    <AccountFormInner
      account={account}
      existingMembers={existingMembers}
      members={members}
      onClose={onClose}
    />
  )
}
```

### IN-02: Missing `autoComplete` Attributes on Text Inputs

**File:** `apps/web/src/components/accounts/AccountForm.tsx:151`, `:198`, `:217`; `apps/web/src/components/settings/CategoryForm.tsx:74`; `apps/web/src/components/settings/RuleForm.tsx:138`, `:164`; `apps/web/src/components/settings/HouseholdSettingsForm.tsx:49`; `apps/web/src/components/categories/LucideIconPicker.tsx:162`
**Issue:** CLAUDE.md `web-design-guidelines` require `autoComplete` on all text inputs — using `autoComplete="off"` for custom/non-browser-fillable fields. None of the eight `<Input>` elements across these files carry an `autoComplete` attribute. Without it, browsers may attempt autofill with incorrect values (e.g. filling "Last 4 digits" with a credit card number, or "Pattern" with an address).
**Fix:** Add `autoComplete="off"` to each custom field input:
```tsx
// AccountForm.tsx — account name (line 151)
<Input id="account-name" autoComplete="off" ... />

// AccountForm.tsx — institution (line 198)
<Input id="account-institution" autoComplete="organization" ... />

// AccountForm.tsx — last four (line 217)
<Input id="account-last-four" autoComplete="off" ... />

// CategoryForm.tsx — category name (line 74)
<Input id="category-name" autoComplete="off" autoFocus ... />

// RuleForm.tsx — pattern (line 138)
<Input autoComplete="off" ... />

// RuleForm.tsx — renameTo (line 164)
<Input autoComplete="off" ... />

// HouseholdSettingsForm.tsx — threshold (line 49)
<Input id="settlement-threshold" type="number" autoComplete="off" ... />

// LucideIconPicker.tsx — search (line 162)
<Input autoFocus autoComplete="off" ... />
```

### IN-03: `VITE_API_URL` Cast as `string` Without Runtime Guard

**File:** `apps/web/src/lib/queryClient.ts:4`
**Issue:** `import.meta.env.VITE_API_URL as string` silently coerces `undefined` to the string `"undefined"` if the env var is missing. All API calls would then target `"undefined/path"` and fail with opaque network errors rather than a clear startup failure.
**Fix:**
```ts
const API_BASE_URL = import.meta.env.VITE_API_URL
if (!API_BASE_URL) {
  throw new Error("VITE_API_URL is not set. Add it to apps/web/.env.")
}
```

### IN-04: `lucide-react` Not in `optimizeDeps.include`

**File:** `apps/web/src/components/categories/LucideIconPicker.tsx` (import source)
**Issue:** `LucideIconPicker.tsx` statically imports ~55 named icons from `lucide-react`. While tree-shakeable in production builds, Vite's dev-mode pre-bundler may not pre-optimize this large module, causing slow cold starts in development. CLAUDE.md `bundle-*` guidance recommends adding heavy external libraries to `optimizeDeps.include` in `vite.config.ts`.
**Fix:**
```ts
// apps/web/vite.config.ts
export default defineConfig({
  optimizeDeps: {
    include: ["lucide-react"],
  },
})
```
