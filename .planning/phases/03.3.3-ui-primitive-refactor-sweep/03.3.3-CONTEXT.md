# Phase 03.3.3: UI Primitive Refactor Sweep - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace all raw HTML primitives in `apps/web` custom components with their shadcn or `<Text>` equivalents. Scope excludes: elements inside shadcn base component children (those stay as-is for upgrade safety), and elements inside `packages/ui/src/components/reui/`. Includes a small packages/ui addition: `error` variant on `<Text>` and a `SwatchButton` helper for ColourPicker.

</domain>

<decisions>
## Implementation Decisions

### Scope boundary
- **D-01:** Migration applies only to custom components in `apps/web/src/` and custom components in `packages/ui`. Raw HTML inside shadcn primitive children (e.g., `<span>` inside `SidebarMenuButton`) stays untouched — shadcn upgrade compatibility.
- **D-02:** `packages/ui/src/components/reui/` is entirely out of scope. No changes there.

### Heading elements (<h1>–<h6>)
- **D-03:** All raw heading elements in custom components → `<Text as="hN" variant="hN">`. Where the existing className uses `font-heading text-xl font-semibold` (= the `h3` variant), use `<Text as="h2" variant="h3">` to preserve the semantic element while matching the visual variant.
- **D-04:** `<h1>Index Route</h1>` in `apps/web/src/routes/index.tsx` is dev scaffolding — skip (no user-visible text to style).
- **D-05:** `<h1 className="text-xl font-semibold">Settings</h1>` in `apps/web/src/routes/_layout.settings/route.tsx` → `<Text as="h1" variant="h3">` (text-xl font-semibold matches h3 variant).

### Paragraph elements (<p>)
- **D-06:** Muted description paragraphs (`text-sm text-muted-foreground`) → `<Text variant="body-sm" className="text-muted-foreground">`.
- **D-07:** Error message paragraphs (`text-xs text-destructive`) → `<Text variant="error">`. Requires adding the `error` variant to `packages/ui/src/components/text.tsx` first.
- **D-08:** `error` variant definition: `"text-xs text-destructive leading-normal"` — mirrors `caption` but with `text-destructive` instead of `text-muted-foreground`.
- **D-09:** Other `<p>` patterns with mixed classes (e.g., `truncate font-mono text-sm`) → `<Text>` with appropriate variant + className overrides.

### Span elements (<span>)
- **D-10:** Text-content spans (spans with direct text content + typography classes: font-*, text-*) → `<Text as="span" variant="...">` with className for any one-off overrides.
- **D-11:** Structural spans (flex containers, layout helpers, min-w-0 wrappers with no direct text) → audit semantic correctness: if used as a block/flex container, convert to `<div>`. If inline structural (e.g., wrapping inline elements without block behavior), keep as `<span>`.
- **D-12:** The `<span />` empty spacer in AccountForm → keep or convert to `<div />` based on layout role.
- **D-13:** Spans inside shadcn component children (e.g., `<span>{label}</span>` inside `SidebarMenuButton`) → leave as-is per D-01.
- **D-14:** `AppLogo`: `<span className="text-lg font-medium text-foreground">Ploutizo</span>` → `<Text as="span" variant="body" className="text-lg font-medium text-foreground">`. (body variant = text-base; override with text-lg.)

### Button elements (<button>)
- **D-15:** ColourPicker's color swatch `<button>` is NOT converted to shadcn `<Button>`. It uses `role="radio"` and custom solid-color styling incompatible with Button base styles.
- **D-16:** Extract swatch button logic into a `SwatchButton` primitive co-located with ColourPicker (or in `packages/ui/src/components/` if reuse is anticipated). `SwatchButton` wraps a raw `<button>` with the swatch-specific styles, type="button", role="radio", aria-checked, aria-label — all preserved.

### packages/ui changes
- **D-17:** Add `error` variant to `TextVariant` union and `variantClasses` map in `packages/ui/src/components/text.tsx`.
- **D-18:** `SwatchButton` — small scoped component, not exported from the UI package's main surface unless ColourPicker is also moved there. Decision on placement left to planner.

### Claude's Discretion
- Exact Tailwind classes for any one-off Text overrides (variant + className combinations)
- Whether SwatchButton is in packages/ui or co-located in apps/web/src/components/categories/
- Exact div/span conversion decisions for ambiguous structural wrappers (MemberRow flex spans, etc.)
- Order of migration: can go file-by-file or element-type by element-type

</decisions>

## Existing Code Insights

### Reusable Assets
- `<Text>` from `@ploutizo/ui/components/text` — available after Phase 03.3.2; accepts `as`, `variant`, `className`
- `TextVariant` / `TextAs` types exported from the same file
- `cn()` from `@ploutizo/ui/lib/utils` — for className merges at call sites

### Established Patterns
- Heading migration pattern: `<h2 className="font-heading text-xl font-semibold">X</h2>` → `<Text as="h2" variant="h3">X</Text>` (h3 variant = font-heading text-xl font-semibold leading-snug)
- Error text pattern (post-migration): `<Text variant="error">` — requires D-17 first
- `font-heading text-xl font-semibold` appears across 5+ files as the consistent page/section heading style — all map to `variant="h3"`

### Integration Points
- `packages/ui/src/components/text.tsx` must be updated (add `error` variant) before any `variant="error"` usage in apps/web
- `SwatchButton` must exist before migrating `ColourPicker`
- No route changes, no API changes, no query changes — pure component refactor

</code_context>

<specifics>
## Specific Ideas

- `font-heading text-xl font-semibold` is the most common heading pattern in the codebase — it maps to `variant="h3"`. Many of these elements use `as="h1"` or `as="h2"` for semantics.
- Structural `<span>` used as flex containers (MemberRow has `<span className="flex min-w-0 items-center gap-2">`) should be audited: block-level flex containers should be `<div>`, not `<span>`.
- AppLogo: since it renders inline brand text, `<Text as="span">` is correct; use className to override the base variant size (body = text-base, needs text-lg).
- SwatchButton: preserve all existing accessibility attributes (`role="radio"`, `aria-checked`, `aria-label`).

</specifics>
