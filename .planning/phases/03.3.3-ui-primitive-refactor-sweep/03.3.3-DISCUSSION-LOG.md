## Error text `<p>` elements

| Option | Description | Selected |
|--------|-------------|----------|
| Leave as raw `<p>` | Error text is framework-rendered; not worth migrating | |
| Migrate with caption + class override | `<Text variant="caption" className="text-destructive">` | |
| Add `error` variant + migrate | New variant `text-xs text-destructive leading-normal` on Text component | ✓ |

**User's choice:** Migrate, and consider if an error variant/pattern is valid and useful → decided to add `error` variant to Text in packages/ui, then migrate all error `<p>` to `<Text variant="error">`.

**Notes:** User confirmed the `error` variant concept: `text-xs text-destructive leading-normal`.

## Span migration criteria

| Option | Description | Selected |
|--------|-------------|----------|
| Text-content only | Spans with direct text + typography classes → Text; structural stays | ✓ (with addition) |
| All non-layout spans | Any span that isn't purely a flex/grid container | |
| Conservative: named text only | Only spans displaying identifiable user-facing names/labels | |

**User's choice:** Text-content spans → `<Text as="span">`. Structural spans → audit whether they should be `<div>` instead of `<span>` (block-level flex containers should be divs).

**Notes:** Broader than the "leave structural alone" option — structural spans that are semantically wrong (flex containers) should be converted to `<div>`.

## ColourPicker `<button>` exception

| Option | Description | Selected |
|--------|-------------|----------|
| Leave as raw `<button>` | Intentional exception, already accessible | |
| Convert with className overrides | `<Button size='icon-xs' variant='ghost' ...>` with overrides | |
| Extract to SwatchButton helper | Custom primitive wrapping `<button>` with swatch styles | ✓ |

**User's choice:** Create a `SwatchButton` helper (either in packages/ui or co-located with ColourPicker) — wraps raw `<button>` with swatch-specific styles. Investigated shadcn registry; found `icon-xs` = size-6 (exact match), but Button base styles conflict (border, hover:bg-muted, rounded-[radius-md] vs rounded).

**Notes:** shadcn CLI was used to check button docs; `icon-xs` size noted for future reference. Base Button incompatible with color swatch role=radio use case.

## Sidebar spans

| Option | Description | Selected |
|--------|-------------|----------|
| Leave as `<span>` | SidebarMenuButton expects `<span>` children per shadcn convention | ✓ |
| Migrate to `<Text as='span'>` | Consistent rule, but may conflict with shadcn sidebar styling | |

**User's choice:** Leave all elements inside shadcn primitives as-is. The migration rule: only custom components in `apps/web` and `packages/ui`. Shadcn base components must remain unmodified for future upgrade compatibility. This generalizes to ALL shadcn primitive children — not just sidebar.

**Notes:** This decision established the scope boundary: migration applies only to custom component code, never to shadcn component internals.
