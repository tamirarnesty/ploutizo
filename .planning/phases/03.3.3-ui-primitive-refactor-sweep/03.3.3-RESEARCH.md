## Summary

This phase is a mechanical refactor sweep with zero logic changes. Every file is already read. All migration decisions are locked in CONTEXT.md. The research job is to produce a precise per-file element inventory so the planner can write deterministic tasks without re-reading source.

The single new capability needed in `packages/ui` is the `error` variant on `<Text>`. The current `text.tsx` has `TextVariant` as a union type and `variantClasses` as a `Record<TextVariant, string>` — adding `error` requires one line in each. This must happen before any `variant="error"` usage in `apps/web`.

`SwatchButton` is a 10-line wrapper around a raw `<button>` — no library lookup needed. It preserves the existing swatch button's accessibility attributes and custom solid-color styling.

**Primary recommendation:** Sequence tasks as (1) add `error` variant to `text.tsx`, (2) create `SwatchButton`, (3) migrate file-by-file in dependency order — packages/ui changes first, then apps/web files from simplest to most complex.

  name: string
  checked: boolean
  onClick: () => void

## Common Pitfalls

### Pitfall 1: Adding `error` variant AFTER using it
**What goes wrong:** TypeScript error `"error" is not assignable to type TextVariant` in all migrated files. Build fails.
**How to avoid:** Wave 0 task must be: add `error` variant to `text.tsx`, then migrate error `<p>` elements.

### Pitfall 2: Replacing structural spans with `<Text as="span">`
**What goes wrong:** `<Text>` applies `text-base leading-relaxed` (body default) to a flex container — adds unwanted typography styles to layout elements.
**How to avoid:** Use the structural span → `<div>` conversion path (D-11). Only use `<Text as="span">` on spans with direct text content.

### Pitfall 3: Migrating spans inside shadcn/FieldLabel children
**What goes wrong:** `<Text>` body styles override the parent label's typography context (e.g., `leading-relaxed` inside `leading-none` label).
**How to avoid:** D-13 exemption — spans inside `<FieldLabel>` / shadcn primitive children stay as raw `<span>`.

### Pitfall 4: `<Text>` default element is `<p>`
**What goes wrong:** Writing `<Text variant="caption">` without `as` renders a `<p>` block element where an inline `<span>` is expected (e.g., inside a flex row that expects inline content).
**How to avoid:** Always specify `as="span"` for inline text in flex/grid layouts. Only omit `as` for standalone paragraph-level text.

### Pitfall 5: h3 variant adds `font-heading`
**What goes wrong:** Some heading targets have `text-xl font-semibold` without `font-heading`. Migrating to `variant="h3"` will add `font-heading` (Noto Serif Variable) — visual change beyond "mechanical refactor".
**How to avoid:** This is intentional per D-03 and D-05. The Settings `<h1>` at route.tsx line 40 (`text-xl font-semibold`, no `font-heading`) is explicitly mapped to `variant="h3"` which will add the heading font. This is the desired design system outcome.

### Pitfall 6: Dynamic className in TransactionColumns amount cell
**What goes wrong:** The amount cell uses a dynamic className string `${amountClass}` combined with static classes. Migrating to `<Text as="span">` must preserve the dynamic join.
**How to avoid:** Use `className={cn("block text-right text-sm font-medium", amountClass)}` — the body-sm variant provides `text-sm leading-relaxed`; override `leading-relaxed` with no-op or add `leading-none` if needed. Alternatively use `className={`block text-right font-medium ${amountClass}`}` and rely on variant for `text-sm`. Test visually.
