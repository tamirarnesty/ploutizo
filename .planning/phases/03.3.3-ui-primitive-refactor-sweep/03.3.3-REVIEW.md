## Summary

This sweep introduces the `Text` component in `packages/ui` and migrates all raw typographic HTML elements across `apps/web` to use it. The implementation is clean and the migration is thorough. One critical bug exists in `TransactionColumns.tsx` (barrel import of the entire lucide-react namespace). Four warnings cover logic gaps or missing safety nets. Four info items flag minor quality issues.

## Critical Issues

### CR-01: Barrel import of entire lucide-react namespace at module level

**File:** `apps/web/src/components/transactions/TransactionColumns.tsx:1`
**Issue:** `import * as LucideIcons from 'lucide-react'` pulls every icon in the library into the bundle — hundreds of components — even though only one icon name is ever resolved at runtime via `DynamicLucideIcon`. Lucide-react is not reliably tree-shaken from a namespace import; Vite/Rollup cannot statically analyse `(LucideIcons as Record<string, unknown>)[name]` and will include the full namespace. This directly violates `bundle-barrel-imports` and will meaningfully inflate the initial JS bundle.

**Fix:**
`DynamicLucideIcon` is only used to render `categoryIcon` on transaction rows. The icon name comes from the same `ICON_MAP` already defined in `LucideIconPicker.tsx`. Share/re-export that map and use it here instead of a namespace import:

```tsx
// In LucideIconPicker.tsx — add export
export { ICON_MAP }

// In TransactionColumns.tsx — replace the namespace import
import { ICON_MAP } from '@/components/categories/LucideIconPicker'

export const DynamicLucideIcon = ({ name, size = 16 }: { name: string | null; size?: number }) => {
  if (!name) return <Tag size={size} />
  const Icon = ICON_MAP[name]
  return Icon ? <Icon size={size} aria-hidden="true" /> : <Tag size={size} />
}
```

This eliminates the namespace import entirely. The `Tag` and `MoreHorizontal` named imports on line 2 are fine.

## Warnings

### WR-01: `MerchantRulesSettings` local order never resets when server data refreshes

**File:** `apps/web/src/components/settings/MerchantRulesSettings.tsx:22-24`
**Issue:** `localRules` is initialised as `[]` and only updated via `handleReorder`. After a successful `reorderRules.mutate()` the TanStack Query cache is invalidated and `rules` (from `useGetMerchantRules`) refreshes, but `localRules` retains the in-memory order indefinitely — even if the server returns a different order (e.g. another session reordered, or the mutation failed silently). `displayRules` will always prefer `localRules` once it is non-empty, so stale ordering is permanently locked in for the lifetime of the component.

`CategoriesSettings.tsx` has the same pattern but gates initialisation behind `initialized.current` (lines 74–80), which at least ensures the initial server data is applied once. `MerchantRulesSettings` has no such guard and never seeds from server data at all — `localRules` starts empty and only grows from user drags, meaning if the user never drags, `displayRules` falls through to the live `rules` array correctly, but after any drag the server array is permanently shadowed.

**Fix:** Mirror the `initialized.current` guard from `CategoriesSettings`, or seed `localRules` via lazy `useState` init so the server snapshot is captured once and then maintained locally:

```tsx
const [localRules, setLocalRules] = useState<MerchantRule[]>([])
const initialized = useRef(false)
useEffect(() => {
  if (!initialized.current && !isLoading && rules.length > 0) {
    initialized.current = true
    setLocalRules(rules)
  }
}, [rules, isLoading])
```

### WR-02: `&&` conditional rendering with non-boolean left-hand side

**File:** `apps/web/src/components/settings/MerchantRuleRow.tsx:53`
**Issue:** `{rule.renameTo && <Text ...>}` — `rule.renameTo` is a `string | null | undefined`. If it is ever an empty string `""`, the expression short-circuits to `""` which React renders as nothing (fine), but more importantly if it is `0` or any other falsy non-boolean value the render silently drops. The project guideline (`rendering-conditional-render`) requires ternary for conditionals where the false branch should render nothing, to avoid accidental rendering of falsy primitives.

**Fix:**
```tsx
{rule.renameTo ? (
  <Text variant="caption">
    → {rule.renameTo}
  </Text>
) : null}
```

### WR-03: `MemberRow` `AlertDialogContent` missing `AlertDialogHeader` wrapper

**File:** `apps/web/src/components/settings/MemberRow.tsx:45-62`
**Issue:** The `AlertDialogContent` in `MemberRow` renders `AlertDialogTitle` and `AlertDialogDescription` without wrapping them in `AlertDialogHeader`. Compare to every other alert dialog in the codebase (`MerchantRuleRow.tsx`, `CategoriesSettings.tsx`, `AccountForm.tsx`) which all use the `AlertDialogHeader` wrapper. Without it, the shadcn layout styles (flex-col spacing between title and description) are not applied, potentially causing inconsistent visual rendering.

**Fix:**
```tsx
<AlertDialogContent className="w-[calc(100%-2rem)] max-w-md">
  <AlertDialogHeader>
    <AlertDialogTitle>
      Remove {member.firstName ?? member.displayName}?
    </AlertDialogTitle>
    <AlertDialogDescription>
      Remove {member.displayName} from this household? They will lose access immediately.
    </AlertDialogDescription>
  </AlertDialogHeader>
  <AlertDialogFooter>
    ...
  </AlertDialogFooter>
</AlertDialogContent>
```

### WR-04: Form-level submit error uses `Text variant="body-sm"` instead of `variant="error"` in `AccountForm`

**File:** `apps/web/src/components/accounts/AccountForm.tsx:409`
**Issue:** The form-level mutation error is rendered as:
```tsx
<Text variant="body-sm" className="text-destructive">
  {String(submitError)}
</Text>
```
All other forms in this codebase (`CategoryForm.tsx:121`, `HouseholdSettingsForm.tsx:78`, `RuleForm.tsx:218`) consistently use `<Text variant="error">` for the form-level submit error. Using `variant="body-sm" className="text-destructive"` applies `text-sm leading-relaxed` instead of the `error` variant's `text-xs text-destructive leading-normal`, producing visually inconsistent error presentation.

**Fix:**
```tsx
<Text variant="error">{String(submitError)}</Text>
```

### IN-02: `getInitials` in `TransactionColumns` produces at most 2 characters but name implies initials

**File:** `apps/web/src/components/transactions/TransactionColumns.tsx:31`
**Issue:** `name.trim().slice(0, 2).toUpperCase()` produces the first two characters of the name string, not actual initials (e.g. "John Smith" → "JO" not "JS"). This is a naming/logic mismatch — the function is named `getInitials` but doesn't extract initials.
**Fix:** Either rename to `getAbbreviation`, or implement proper initials extraction:
```ts
const getInitials = (name: string | null): string => {
  if (!name) return '?'
  const parts = name.trim().split(/\s+/)
  return parts.length >= 2
    ? (parts[0][0] + parts[parts.length - 1][0]).toUpperCase()
    : name.slice(0, 2).toUpperCase()
}
```

### IN-04: `AppLogo` comment explains the override but variant selection is unintuitive

**File:** `apps/web/src/components/AppLogo.tsx:4`
**Issue:** `variant="body"` is used and then overridden with `className="text-lg"`. The `body` variant already applies `text-base`; overriding to `text-lg` works but `variant="body"` becomes misleading. Consider using `as="span"` with no variant and a full explicit className, or adding a `brand` variant to `Text` if this pattern recurs.
