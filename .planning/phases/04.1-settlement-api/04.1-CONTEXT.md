# Phase 4.1: Settlement API - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a read endpoint that computes per-account, per-member outstanding balances at query time and returns them as settlement card data. Add a simplified write endpoint to record a settlement payment. Add a schema migration to support statement due dates on accounts. No UI work in this phase.

**Scope anchor:** The settlement concept is **member-owes-card**, not member-owes-member. Each household member has an outstanding balance on each card based on their expense splits (personal + their share of shared transactions) minus settlements already recorded. The person-to-person "who owes who" framing is NOT the primary model — the card balance is. Any cross-member summary the UI needs is derived client-side.

**Every transaction is either personal (1 assignee, full amount) or shared (multiple assignees, split amounts summing to full transaction amount).** The balance formula is identical for both cases: each member owes their `amount_cents` splits on the account.

**`eachPersonPaysOwn` flag removed in this phase.** The flag was designed for person-to-person settlement (to prevent "Emily owes Tamir for Amex"). In the card-centric model it's redundant — all accounts work the same way. Remove from schema and accounts API.

**Out of scope:** Settlement reminder notifications (Phase 7.1), settlement history list UI (Phase 4.2), the "Credit Card Balances" account card table UI (Phase 4.2), budget integration.

</domain>

<decisions>
## Implementation Decisions

### Endpoint shape

- **D-01:** New route file `apps/api/src/routes/settlements.ts`. Mounted at `/api/settlements`. Two endpoints: `GET /api/settlements` (balance read) and `POST /api/settlements` (record payment).
- **D-02:** `GET /api/settlements` response shape:
  ```ts
  {
    accounts: Array<{
      account: { id, name, type, institution, lastFour, statementDueDay },
      totalBalanceCents: number,   // sum of all member balanceCents
      members: Array<{ member: { id, name, avatarUrl }, balanceCents: number }>,
      dueDate: string | null,      // next statement due date computed from statementDueDay
      status: 'due_soon' | 'on_track' | null   // null when statementDueDay is null
    }>
  }
  ```
  No `pairBalances`, no `netLines`, no `threshold` field. Cross-member summary is derived client-side.
- **D-03:** `POST /api/settlements` simplified payload: `{ payerMemberId, accountId, amountCents, date }`. Server auto-fills `type: 'settlement'`, generates description, creates `transaction_assignees` row for `payerMemberId`. Delegates to the existing transaction create service internally (or wraps it).
- **D-04:** Response includes `name + avatarUrl` per member in each balance row — avoids a second `/members` fetch in the UI.

### Balance formula

- **D-05:** Outstanding balance = `SUM(transaction_assignees.amount_cents)` for expense + refund transactions on that account assigned to that member, **minus** `SUM(settlement transaction amount)` where `accountId` = that account and the settlement is assigned to that member.
  - Only `expense` and `refund` transaction types create a balance obligation.
  - `settlement` transactions (via `accountId`) reduce the balance.
  - `income`, `transfer`, `contribution` types are excluded from balance math.
  - **Refunds reduce balance** (same direction as a settlement — refund is money coming back).
  - All `SUM` aggregates must cast to `bigint` per established convention.
- **D-06:** Settlement transaction account field mapping: `accountId` = the expense account being settled (the card whose balance decreases). `counterpartAccountId` = where the payment came from (bank account / cash / e-transfer). Balance query JOINs settlement transactions on `accountId`.
- **D-07:** `eachPersonPaysOwn` flag is **removed** in this phase — schema migration drops the column, accounts API validators and responses updated to remove the field. All accounts participate in settlement calculations equally.
- **D-08:** Accounts where ALL member balances = 0 are omitted from the `accounts` array. Only accounts with at least one non-zero member balance are returned.
- **D-09:** API returns all non-archived account types (not just credit cards). The UI applies a credit card type filter for the "Credit Card Balances" view. The API is account-type-agnostic.
- **D-10:** Overpayment is allowed (no server-side cap on `amountCents` vs. outstanding balance). A member who overpays gets a negative balance — displayed as a credit in the UI (Phase 4.2).
- **D-11:** For settlement pairs direction (when the UI needs it): use account `account_members` to identify the account owner. v1 assumes 1 primary owner per account. Multi-owner non-`eachPersonPaysOwn` accounts are not a v1 concern.

### Statement due date (schema migration)

- **D-12:** Add `statementDueDay` integer column (1–31, nullable) to the `accounts` table. No existing accounts need a default — nullable is correct (not all account types have a statement due date).
- **D-13:** `dueDate` in the response is computed at query time: given `statementDueDay`, compute the next upcoming date relative to today. If `statementDueDay` is null, return `null` for `dueDate` and `null` for `status`.
- **D-14:** `status` values: `'due_soon'` = due within 7 days; `'on_track'` = more than 7 days away; `null` = no due date set.

### Indexes (migration)

- **D-15:** Add index `(memberId)` on `transaction_assignees`. Speeds up "all expense splits for member X" without requiring denormalization.
- **D-16:** Add index `(memberId)` on `account_members`. Speeds up "all accounts owned by member X" for settlement direction lookup.
- **D-17:** **Do NOT** denormalize `accountId` onto `transaction_assignees`. Rely on the JOIN with the existing `transactions_org_account_idx` on `(orgId, accountId)`. Acceptable for v1 scale.

### Validation and errors

- **D-18:** `POST /api/settlements` DomainError cases (follow existing `DomainError(status, message)` pattern):
  - `accountId` not found in org → 404 `"Account not found"`
  - `accountId.archivedAt` is set → 400 `"Cannot settle an archived account"`
  - `payerMemberId` not in org → 404 `"Member not found in this household"`
  - `amountCents ≤ 0` → handled by Zod validator (positive integer)
- **D-19:** `tenantGuard` (existing middleware) handles org-scoping for both endpoints — no additional org ownership checks needed.

### Frontend cache invalidation

- **D-20:** After `POST /api/settlements` succeeds, invalidate `['settlements']` query key only. The transactions list (`['transactions']`) is **not** invalidated on settlement — the settlement card view is the primary surface for this data and has its own cache. (Note: if the transaction list ever renders settlements, add `['transactions']` invalidation then.)

### Claude's Discretion

- Exact query implementation (single SQL query vs. multiple round trips). Recommendation: single query with GROUP BY joining `transactions → transaction_assignees → account_members` to get all balances in one pass.
- Whether `POST /api/settlements` creates the transaction record directly or calls the existing `createTransaction` service. Recommendation: call the existing service to reuse validation and audit trail.
- Semver range for any new dependencies (none expected — this phase uses only existing packages).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `apps/api/src/services/transactions.ts` `createTransaction` — call this from the POST /api/settlements handler to create the underlying settlement transaction record. Handles `transaction_assignees` insert automatically.
- `apps/api/src/lib/validator.ts` `appValidator` — Hono validator wrapper; use for the POST body Zod schema.
- `apps/api/src/lib/errors.ts` `DomainError` — existing error class for 4xx domain errors.
- `packages/db/src/schema/auth.ts` `orgMembers` — has `settlementThreshold` and display fields (name, avatar) for member info in response.

### Established Patterns
- Route structure: `new Hono<AppEnv>()` → handlers → export default. Mount in `apps/api/src/index.ts`.
- All handlers: `const orgId = c.get('orgId')` first — set by `tenantGuard` middleware.
- Query layer: DB queries in `apps/api/src/lib/queries/` (or `src/services/`), not inline in route handlers.
- Drizzle `SUM` aggregation: always `sql\`SUM(${field})::bigint\`` — never raw numeric.
- Soft-delete filter: all active-data queries use `isNull(transactions.deletedAt)` — partial index `transactions_active_idx` covers this.

### Integration Points
- New route mounted in `apps/api/src/index.ts` — follow the existing `app.route('/api/settlements', settlementsRouter)` pattern.
- `accounts` migration: add `statementDueDay` column via Drizzle migration in `packages/db/src/migrations/`.
- `transaction_assignees` and `account_members` index migrations: add alongside `statementDueDay` in a single migration file.

</code_context>

- The mockup showed "Amex Gold is shared — each person pays their own ($1,300 TA · $800 EM). <deferred>
## Deferred Ideas

- **Person-to-person summary panel** ("Emily owes Tamir $360") — discussed and removed from this phase. If this view is needed, it can be derived client-side from the account card data. Could be added as an optional `pairBalances` + `netLines` field in a future iteration without a breaking change.
- **Settlement threshold in GET /api/settlements response** — discussed and dropped. Threshold is available via `GET /api/households/settings` if the UI needs it.
- **`['transactions']` cache invalidation on settlement** — deferred. Only invalidate `['settlements']`. If the transaction list ever renders settlements inline, add `['transactions']` invalidation at that point.
- **Multi-owner card settlement direction** — deferred to v2. v1 assumes 1 primary owner per account for settlement direction. Multi-owner accounts are not expected in practice in v1.

</deferred>
