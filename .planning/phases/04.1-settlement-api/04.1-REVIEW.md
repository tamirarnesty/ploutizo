# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-08T00:00:00Z
**Depth:** standard
**Files Reviewed:** 19
**Status:** issues_found

## Summary

Phase 04.1 adds the GET and POST `/api/settlements` endpoints plus the `statementDueDay` column migration. The core logic — balance aggregation query, `computeNextDueDate` pure function, `createSettlement` service — is well-structured and the test coverage is thorough. Two critical defects were found: the `statementDueDay` column is unwritable through any API surface (the entire dueDate/status feature silently returns `null` in production), and an uncaught `TypeError` in the CORS origin handler crashes on requests without an `Origin` header.

---

## Critical Issues

### CR-01: `statementDueDay` has no write path — dueDate/status will always be null in production

**File:** `packages/types/src/index.ts:18-29`, `packages/validators/src/accounts.ts:13-24`, `apps/web/src/components/accounts/AccountForm.tsx:118-135`

**Issue:** The migration adds `statement_due_day` to the `accounts` table and `fetchSettlementBalances` reads it, but no code writes it. All three layers that would need to carry the field are missing it:

- `Account` interface (`packages/types/src/index.ts`) — no `statementDueDay` field
- `createAccountSchema` / `updateAccountSchema` (`packages/validators/src/accounts.ts`) — no `statementDueDay` field, so the API rejects it as an unknown key
- `AccountForm.tsx` — no UI field for `statementDueDay`
- `useCreateAccount.ts` / `useUpdateAccount.ts` body types — no `statementDueDay` field

Since the column is always `NULL`, `computeNextDueDate` always receives `null` and returns `{ dueDate: null, status: null }`. The whole dueDate/status feature is dead in production.

**Fix:** Wire the field through all layers:

```ts
// packages/types/src/index.ts — add to Account interface
statementDueDay: number | null

// packages/validators/src/accounts.ts — add to createAccountSchema and updateAccountSchema
statementDueDay: z.number().int().min(1).max(31).nullable().optional()

// apps/web/src/lib/data-access/accounts/useCreateAccount.ts
// apps/web/src/lib/data-access/accounts/useUpdateAccount.ts — add to body type
statementDueDay?: number | null

// AccountForm.tsx — add a numeric input (1-31) for credit_card accounts
// and include statementDueDay in the payload passed to mutate()
```

### CR-02: CORS `origin` callback crashes with `TypeError` on requests without an `Origin` header

**File:** `apps/api/src/index.ts:27-31`

**Issue:** The Hono `cors()` `origin` callback receives `undefined` when a request has no `Origin` header (same-origin, curl, server-to-server health checks, Railway health probes). The comparison `origin === 'https://ploutizo.app'` evaluates to `false`, and then `origin.endsWith('.ploutizo.app')` throws `TypeError: Cannot read properties of undefined (reading 'endsWith')`. This crashes the request before the route handler fires — including `GET /health`.

Confirmed with Node.js:
```
> undefined.endsWith('.ploutizo.app')
TypeError: Cannot read properties of undefined (reading 'endsWith')
```

**Fix:**

```ts
origin: (origin) => {
  if (!origin) return 'https://ploutizo.app'   // no Origin header — default
  return origin === 'https://ploutizo.app' ||
    origin.endsWith('.ploutizo.app') ||
    origin === 'http://localhost:3000'
      ? origin
      : 'https://ploutizo.app'
},
```

Note: this bug pre-dates this phase but `apps/api/src/index.ts` is modified in 04.1 (adding `settlementsRouter`), so it is in scope.

## Warnings

### WR-01: `createSettlement` bypasses `validateSplitSum` — split invariant relies on caller discipline

**File:** `apps/api/src/services/settlements.ts:136-149`

**Issue:** `createTransaction` does not call `validateSplitSum` internally — it is only called at the route layer in `POST /api/transactions`. `createSettlement` calls `createTransaction` directly, bypassing the route. The invariant is currently safe because `createSettlement` constructs `amount: data.amountCents` and `assignees: [{ amountCents: data.amountCents }]` (sum always equals amount), but it relies entirely on the caller constructing consistent values. Any future change to the `createSettlement` logic that miscalculates `assignees` will silently store inconsistent data with no server-side guard.

**Fix:** Move the split-sum check inside `createTransaction` so every call path is guarded:

```ts
// services/transactions.ts — inside createTransaction, before db.transaction()
if (assignees && assignees.length > 0) {
  const splitError = validateSplitSum(data.amount, assignees)
  if (splitError) throw new DomainError(400, splitError)
}
```

Then remove the redundant check from the route handler.

### WR-02: TOCTOU — account archived between validation check and transaction write

**File:** `apps/api/src/services/settlements.ts:124-149`

**Issue:** `fetchAccountForSettlement` checks `archivedAt` at line 124-128, then `memberBelongsToOrg` at line 131-132, then `createTransaction` (which opens a DB transaction) at line 136. There is no lock on the account row between the check and the write. A concurrent `PATCH /api/accounts/:id` that archives the account between lines 128 and 136 will create a settlement on an archived account, violating the invariant enforced at line 127.

**Fix:** Either perform the `archivedAt` check inside the `db.transaction()` in `createTransaction` using `SELECT ... FOR UPDATE`, or re-fetch the account inside a transaction. The simplest approach is to delegate to a new lower-level function that does the insert inside a transaction with a re-check:

```ts
// Option: pass the validation into createTransaction as a pre-write check,
// or run fetchAccountForSettlement inside the transaction block in createSettlement.
```

### WR-03: `lastFour` is not validated as digits at the API layer

**File:** `packages/validators/src/accounts.ts:17`

**Issue:** `createAccountSchema` validates `lastFour` only with `.max(4)` — it accepts non-digit characters like `"abcd"`. The web form strips non-digits at the UI layer (`e.target.value.replace(/\D/g, '')`), but a direct API call can store arbitrary text in this column.

**Fix:**

```ts
lastFour: z.string().regex(/^\d{4}$/, 'Last four must be exactly 4 digits').optional()
```

(Adjust to `max(4)` if partial entry should be allowed: `z.string().regex(/^\d{1,4}$/).optional()`)

### WR-04: `isAllowedParty` is exported but not wired — subdomain auth bypass in production

**File:** `apps/api/src/index.ts:40-50`

**Issue:** `isAllowedParty` (lines 40-43) implements regex-based subdomain validation, but `clerkMiddleware` receives only the static `authorizedParties` array (lines 47-50). Subdomain tokens (e.g. `https://preview.ploutizo.app`) are validated by `isAllowedParty` but that function is never called by the middleware. The comment says "Phase 2+" but `isAllowedParty` is exported — other code could import it expecting it to be enforced. The actual `authorizedParties` list contains only two entries; subdomain requests from deploy previews will either be rejected by Clerk (if Clerk enforces `authorizedParties` strictly) or accepted without the subdomain regex check.

**Fix:** Either remove `isAllowedParty` and the export until Phase 2, or replace the static array with a dynamic check using a Clerk-compatible pattern. Add a comment clarifying the current state to avoid confusion.

### IN-03: `DELETE /api/accounts/:id/archive` test asserts 200 OR 404 — not a meaningful assertion

**File:** `apps/api/src/__tests__/accounts.test.ts:241-244`

**Issue:** The test accepts any status in `[200, 404]`. This is not a useful assertion — it would pass even if the route returned 500 (it doesn't, but a regression could change the behavior without failing this test). The test description also says "returns 200 or 404" which is not a testable contract.

**Fix:** Either mock the DB to return a known result (row found → 200, row not found → 404) and assert the specific expected status per case, or replace with two separate tests that set up the DB mock explicitly.
