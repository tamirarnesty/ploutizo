# Phase 03.3.2: Implement Text Typography Component - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Build `<Text as variant className>` as the single typography primitive in `packages/ui`. `as` controls the DOM element, `variant` controls the visual preset. No new features, no app-level changes. Phase 03.3.3 depends on this component existing before sweeping `apps/web`.

</domain>

<decisions>
## Implementation Decisions

### Component API
- **D-01:** `as` prop controls DOM element (`h1`–`h6`, `p`, `span`, `div`, `label`). Defaults to `p`.
- **D-02:** `variant` prop controls visual preset (`h1`, `h2`, `h3`, `body`, `body-sm`, `caption`, `label`). Defaults to `body`.
- **D-03:** `className` merged last via `cn()` — used for color, weight, italic, and one-off overrides at the call site.
- **D-04:** `as` and `variant` are fully decoupled. Any combination is valid (e.g., `<Text as="span" variant="h2">` renders a `<span>` with h2 visual styles).

### Variant Style Map
- **D-05:** Heading variants use `font-heading` (Noto Serif Variable) — consistent with the existing codebase pattern (`font-heading text-xl font-semibold` already used for section headings):
  - `h1`: `font-heading text-3xl font-semibold tracking-tight leading-tight`
  - `h2`: `font-heading text-2xl font-semibold tracking-tight leading-tight`
  - `h3`: `font-heading text-xl font-semibold leading-snug`
- **D-06:** Body/utility variants use the default font (Geist Variable via `--font-sans`):
  - `body`: `text-base leading-relaxed`
  - `body-sm`: `text-sm leading-relaxed`
  - `caption`: `text-xs text-muted-foreground leading-normal`
  - `label`: `text-sm font-medium leading-none`

### Label Variant Semantics
- **D-07:** `<Text variant="label">` is a visual typography preset only — no peer-disabled pseudo-classes, no form semantics. Use it for "label for something that isn't explicitly an input" (section labels, description list terms, column headers, etc.). Shadcn `<Label>` remains the correct choice for actual `<label>` elements paired with form inputs.

### Export
- **D-08:** File lives at `packages/ui/src/components/text.tsx`. Consumed via the existing wildcard path export `"./components/*"` as `@ploutizo/ui/components/text` — no changes to `package.json` exports required.
- **D-09:** Export both the `Text` component and the `TextVariant` / `TextAs` TypeScript types from the file. Phase 03.3.3 will need to reference variants by type.

### Claude's Discretion
- Implementation of the polymorphic `as` prop (union type + `React.ElementType` or explicit union — whichever is cleaner with React 19 and no `forwardRef`)
- Whether to use `cva` / `class-variance-authority` for the variant map or a plain `Record<variant, string>` lookup
- `ref` forwarding approach (React 19 — pass `ref` as a regular prop, no `forwardRef`)

</decisions>
