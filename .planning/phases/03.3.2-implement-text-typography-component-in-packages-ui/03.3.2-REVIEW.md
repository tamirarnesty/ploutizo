# Phase 03.3.2: Code Review Report

**Reviewed:** 2026-04-17
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Single new polymorphic `Text` component. React 19 ref-as-prop pattern is correctly implemented. Two type-safety issues exist in the generic prop type: the `ref` type is too broad, and the `ComponentPropsWithoutRef<T>` spread over a union type collapses to an intersection, losing element-specific props for callers. No security issues. No runtime bugs in the happy path.

## Warnings

### WR-01: `ref` type is too broad — breaks strict-mode callers

**File:** `packages/ui/src/components/text.tsx:22`
**Issue:** `ref?: React.Ref<HTMLElement>` types the ref as the base `HTMLElement` rather than the specific element type selected by `as`. A caller passing `as="p"` with a `RefObject<HTMLParagraphElement>` will get a TypeScript error in strict mode because `HTMLParagraphElement` is narrower than `HTMLElement`. The component renders correctly at runtime, but the type contract is wrong.
**Fix:**
```tsx
// Replace the static React.Ref<HTMLElement> with a generic that resolves
// to the correct element type based on `as`:
type TextProps<T extends TextAs = "p"> = {
  as?: T
  variant?: TextVariant
  className?: string
  ref?: React.Ref<React.ComponentRef<T>>  // narrows to the correct element type
  children?: React.ReactNode
} & Omit<React.ComponentPropsWithoutRef<T>, "as" | "variant" | "className" | "ref" | "children">
```

### WR-02: `ComponentPropsWithoutRef<T>` over a union collapses to an intersection — loses element-specific props

**File:** `packages/ui/src/components/text.tsx:24`
**Issue:** When `T` is an unresolved generic constrained to a union (`TextAs`), TypeScript resolves `React.ComponentPropsWithoutRef<T>` as the intersection of all member element prop types, not as a conditional type. This means element-specific props (e.g. `htmlFor` for `as="label"`, `id` is preserved but narrower props are not) are not surfaced in autocomplete or enforced by the type checker. In practice `<Text as="label">` will silently lose `htmlFor` type support.
**Fix:** Use a distributive conditional type to correctly map `T` to its element props:
```tsx
type ElementPropsFor<T extends TextAs> = T extends TextAs
  ? React.ComponentPropsWithoutRef<T>
  : never

type TextProps<T extends TextAs = "p"> = {
  as?: T
  variant?: TextVariant
  className?: string
  ref?: React.Ref<React.ComponentRef<T>>
  children?: React.ReactNode
} & Omit<ElementPropsFor<T>, "as" | "variant" | "className" | "ref" | "children">
```
This preserves the discriminated union so `as="label"` correctly surfaces `htmlFor`.

## Info

### IN-01: `"label"` name collision between `TextAs` and `TextVariant`

**File:** `packages/ui/src/components/text.tsx:5-6`
**Issue:** Both `TextAs` and `TextVariant` include `"label"`, but they refer to unrelated concepts — an HTML element tag vs. a typography style. Consumers reading the API may confuse the label element (`as="label"`) with the label variant (`variant="label"`). The collision is not a bug but is a readability and discoverability risk in a shared library.
**Fix:** Consider renaming the `TextVariant` `"label"` to `"form-label"` or `"input-label"` to disambiguate. Low priority; only worth changing before the component is widely adopted.

### IN-02: No `displayName` set on the generic function component

**File:** `packages/ui/src/components/text.tsx:26`
**Issue:** Generic function components assigned without a name expression sometimes appear as anonymous or as the variable name in React DevTools. Setting a display name makes debugging easier in a shared UI package.
**Fix:**
```tsx
Text.displayName = "Text"
```
Add after the function declaration, before the export.
