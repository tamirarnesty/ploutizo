<objective>
Fix TransactionTagPicker so existing tags appear and inline create works.

Purpose: Two root causes:
1. `Combobox` (= `ComboboxPrimitive.Root` from Base UI) does NOT accept `items` or `filteredItems` props — they are silently ignored, so no items appear.
2. `ComboboxList` is a plain `&lt;ul&gt;` wrapper — it does NOT accept a render function as children. The `{(item: string) => &lt;ComboboxItem&gt;}` pattern silently renders nothing.

Fix: Render items as static JSX children inside ComboboxList. Use Base UI's built-in filtering (Combobox.Root with `filterFn` or default filtering on `value` prop of each item).

Output: TransactionTagPicker fully rewritten to use static JSX item rendering.
</objective>

<!-- Props to REMOVE from Combobox root (not valid Base UI Combobox.Root props) -->
items={items}          // REMOVE
filteredItems={...}    // REMOVE

<!-- Base UI filtering behavior -->
When `multiple` is true and items are rendered as static JSX ComboboxItems,
Base UI filters them automatically by matching the user's input against each
item's `value` prop (case-insensitive substring match by default).
The createOption item uses a `__create__` prefix value — it will match the input
because the input text IS the suffix after __create__. However, to ensure the
create option always appears when there is input that doesn't match, render it
OUTSIDE the main filter by using a separate section or rely on the fact that
Base UI filters on `value` prop: since `createOption = '__create__' + tagInputValue`,
and the user typed `tagInputValue`, the Base UI filter WILL match it (tagInputValue
is a substring of the value). This is the desired behavior.

<!-- ComboboxValue render prop (chips display) — this IS a render prop, and it works -->
<ComboboxValue>
  {(selectedValue: string[] | null) => (
    <>
      {(selectedValue ?? []).map((name) => (
        <ComboboxChip key={name}>{name}</ComboboxChip>
      ))}
      <ComboboxChipsInput placeholder={...} autoComplete="off" />
    </>
  )}
</ComboboxValue>
```
This pattern is correct and should be preserved.
