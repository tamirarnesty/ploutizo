## Summary

Phase 03.4 wires up a `TransactionSheet` + `TransactionForm` pair following the established AccountSheet/AccountForm pattern exactly. The primary technical complexity is the discriminated union form — TanStack Form's `useAppForm` must manage a flat superset of all type-specific fields (some undefined per branch), while `form.Subscribe` on the `type` field drives conditional rendering. Field clearing on type change uses per-field `listeners.onChange` callbacks.

The split section is a standalone component tree (`SplitSection` → `AssigneeRow`) that owns the % / $ toggle mode and runs LRM arithmetic in a pure client-side function. The refund linker uses the existing `Combobox` component (base-ui-backed, NOT Popover+Command — see Standard Stack). The API needs two targeted patches: swap `updateTransactionSchema` → `createTransactionSchema` on PATCH, and add a `?description=` query param to GET.

**Primary recommendation:** Clone the AccountForm outer/inner loading-gate pattern. Keep form values as a flat superset interface (all optional type-specific fields), run `toApiPayload()` in the submit handler to convert dollars→cents and strip undefined type-specific fields, then validate against `createTransactionSchema` before sending to the API.

### Pattern 1: Loading Gate (AccountForm reference)

**What:** Fire all queries at top level simultaneously; render spinner until all resolve; mount form only when data is complete. No `useEffect`/`form.reset`.

**When to use:** Any form that requires async data (org members, existing transaction, categories, accounts).

```tsx
// Source: apps/web/src/components/accounts/AccountForm.tsx (VERIFIED)
export const TransactionForm = ({ transaction, onClose }: TransactionFormProps) => {
  // All queries fire simultaneously — no waterfall
  const { data: tx, isLoading: txLoading } = useGetTransaction(transaction?.id ?? null)
  const { data: orgMembers = [], isLoading: orgLoading } = useGetOrgMembers()
  const { data: categories = [], isLoading: catLoading } = useGetCategories()
  const { data: accounts = [], isLoading: accLoading } = useGetAccounts()

  if (txLoading || orgLoading || catLoading || accLoading) {
    return (
      <div className="flex flex-1 items-center justify-center">
        <Spinner />
      </div>
    )
  }

  return (
    <TransactionFormInner
      key={transaction?.id ?? "new"}
      transaction={tx ?? null}
      orgMembers={orgMembers}
      categories={categories}
      accounts={accounts}
      onClose={onClose}
    />
  )
}
```

**Edit-mode defaultValues note:** `defaultValues.amount` must be `transaction.amount / 100` (DB stores integer cents; form displays dollars). The inner component receives `tx` which has `amount` in cents — convert before passing to `defaultValues`.

### Pattern 3: Discriminated Union Form — Flat Superset with listeners

**What:** TanStack Form works best with a single concrete type for `defaultValues`. For a discriminated union with 6 branches, use a flat superset type with all optional type-specific fields. On type change, clear the previous type's fields with `form.setFieldValue` in a `listeners.onChange` on the type field.

**TanStack Form canonical pattern** for clearing dependent fields on change:

```tsx
// Source: TanStack Form docs - listeners guide (CITED: tanstack.com/form/docs/framework/react/guides/listeners)
<form.AppField
  name="type"
  listeners={{
    onChange: ({ fieldApi }) => {
      // onChange fires after the field value is already updated — clear unconditionally.
      // No condition needed: if the type didn't change, these fields were already undefined.
      fieldApi.form.setFieldValue('categoryId', undefined)
      fieldApi.form.setFieldValue('refundOf', undefined)
      fieldApi.form.setFieldValue('incomeType', undefined)
      fieldApi.form.setFieldValue('incomeSource', undefined)
      fieldApi.form.setFieldValue('toAccountId', undefined)
      fieldApi.form.setFieldValue('settledAccountId', undefined)
      fieldApi.form.setFieldValue('investmentType', undefined)
    },
  }}
>
```

**Why no `if (newType !== getFieldValue('type'))` guard:** By the time `onChange` fires, `fieldApi.form.getFieldValue('type')` already returns the new value. The guard would always be false, making the clear a no-op. Clear unconditionally — resetting `undefined` to `undefined` is safe.

**Form values type** — flat superset, all type-specific fields optional:

```ts
// This is the recommended approach for TanStack Form with discriminated union schemas
// [ASSUMED] — TanStack Form does not prescribe a specific approach for discriminated unions.
// Recommendation based on: (1) TanStack Form types defaultValues as TFormData (single concrete type),
// (2) AccountForm precedent (no union in that form), (3) need to access fields like name="categoryId"
// without TypeScript rejecting them as absent from the union branch.
interface TransactionFormValues {
  // Shared base
  accountId: string
  amount: number        // USER INPUT: dollars (e.g. 12.99); convert to cents in toApiPayload()
  date: string          // ISO YYYY-MM-DD
  description: string | undefined
  merchant: string | undefined
  assignees: AssigneeFormRow[]
  tagIds: string[]
  // Type discriminant
  type: 'expense' | 'income' | 'transfer' | 'settlement' | 'refund' | 'contribution'
  // Type-specific (all optional — only relevant branch populated)
  categoryId: string | undefined      // expense (required), refund (optional)
  refundOf: string | undefined        // refund (optional)
  incomeType: string | undefined      // income (required)
  incomeSource: string | undefined    // income (optional)
  toAccountId: string | undefined     // transfer (required)
  settledAccountId: string | undefined // settlement (optional)
  investmentType: string | undefined  // contribution (optional)
}
```

**Submit transform and validation:** Never validate raw form values against `createTransactionSchema` — the schema requires `amount` as integer cents, but the form stores dollars. Always transform first:

```ts
// In useTransactionForm.ts submit handler — transform then validate
function toApiPayload(value: TransactionFormValues) {
  return {
    ...value,
    amount: Math.round(value.amount * 100),  // dollars → cents
    // type-specific fields already cleared on type change via listeners
  }
}

// In onSubmit:
const payload = toApiPayload(value)
const result = createTransactionSchema.safeParse(payload)
if (!result.success) {
  form.setErrorMap({ onSubmit: "Validation error." })
  return
}
await createTransaction.mutateAsync(payload)
```

**Note:** `createTransactionSchema` accepts the full discriminated union. Undefined optional type-specific fields are stripped by Zod — no need to omit them manually.

### Pattern 5: LRM Algorithm (pure function)

**What:** Largest Remainder Method distributes integer cents across N assignees so they sum exactly to `total`. Returns `{ amountCents, percentage }[]`.

```ts
// [ASSUMED] — standard LRM algorithm, not unique to this project
export function lrmSplit(
  totalCents: number,
  memberIds: string[]
): { memberId: string; amountCents: number; percentage: number }[] {
  if (memberIds.length === 0) return []
  const base = Math.floor(totalCents / memberIds.length)
  const remainder = totalCents - base * memberIds.length
  const exactPct = 100 / memberIds.length
  return memberIds.map((memberId, i) => ({
    memberId,
    amountCents: i < remainder ? base + 1 : base,
    percentage: parseFloat(exactPct.toFixed(3)),
  }))
}
```

**Note:** `percentage` stored in DB is a DISPLAY CACHE ONLY (numeric column, Drizzle returns as string). Never use for balance math. LRM `amountCents` is authoritative.

### Anti-Patterns to Avoid

- **form.reset() in useEffect**: Do not call `form.reset()` after async data loads. Use the loading gate (render spinner, then mount form with full `defaultValues`).
- **Sequentially gating queries**: Do not `const { data: tx } = useGetTransaction(id)` then `const { data: members } = useGetOrgMembers({ enabled: !!tx })`. Fire all at top level, block on combined loading.
- **Reading `percentage` for arithmetic**: `transactionAssignees.percentage` is a Drizzle `numeric` column — returns as `string`. Never add or compare it numerically.
- **Nesting form.Subscribe unnecessarily**: One top-level Subscribe for `type` is sufficient; TypeFields receives `type` as a plain prop.
- **Forgetting the `amount` dollars→cents conversion**: API and DB store unsigned integer cents. Input is dollars. Convert in `toApiPayload()` — never in defaultValues or field display.
- **Validating raw form values against createTransactionSchema**: Schema requires integer cents; form stores dollars. Always run `toApiPayload()` before calling `safeParse`.
- **Sending non-discriminated PATCH payload**: `updateTransactionSchema` on the current PATCH route is a flat partial with no `type` field. D-08 requires changing to `createTransactionSchema`. Failing to do this means type changes cannot be enforced server-side.
- **if-guarding listeners.onChange**: The guard `if (newType !== getFieldValue('type'))` is always false — the field value is already updated when onChange fires. Clear stale fields unconditionally.

## Common Pitfalls

### Pitfall 1: Discriminated Union TypeScript with form.AppField
**What goes wrong:** `form.AppField name="categoryId"` errors at compile time when `TransactionFormValues` is typed as the discriminated union (because `categoryId` doesn't exist on the `transfer` branch).
**Why it happens:** TanStack Form's `AppField name` is constrained to keys of `TFormData`. Discriminated union has different keys per branch.
**How to avoid:** Use a flat superset interface as `TFormData` (all optional type-specific fields). Run `createTransactionSchema.safeParse` only on submit after `toApiPayload()` transform, not as per-field validators.
**Warning signs:** TypeScript error on `name="categoryId"` saying field doesn't exist in form values.

### Pitfall 2: Dollars vs. Cents — Form, LRM, and Submit Must All Agree
**What goes wrong:** User enters 12.99; form stores 12.99 (dollars); `toApiPayload()` is skipped or applied inconsistently; API receives 12.99 or LRM receives 12 (truncated float cents); split sum validation fails.
**Why it happens:** Three different layers touch `amount`: form display (dollars), LRM (cents), API (cents). If any layer skips conversion, the values diverge.
**How to avoid:** (1) `defaultValues.amount = transaction.amount / 100` (DB cents → form dollars for edit mode); (2) `toApiPayload()` converts back: `Math.round(value.amount * 100)`; (3) pass converted `amountCents` to LRM; (4) API receives consistent cents.
**Warning signs:** `"Assignee amounts must sum to transaction amount"` API error on valid-looking forms; LRM totals row showing wrong amounts.

### Pitfall 3: Stale Type-Specific Fields on Type Change
**What goes wrong:** User selects "expense" (categoryId set), changes to "transfer" (toAccountId set), submits. `createTransactionSchema` for `transfer` may reject extra fields or fail required field checks.
**Why it happens:** Flat superset form values retain previous type's fields unless explicitly cleared.
**How to avoid:** Use `listeners.onChange` on the `type` field to call `form.setFieldValue` on each type-specific field unconditionally. Do NOT add an `if (newType !== getFieldValue('type'))` guard — it's always false at onChange time.
**Warning signs:** 400 validation errors on type switch; server complaining about unexpected fields.

### Pitfall 4: `percentage` from Drizzle is a string
**What goes wrong:** `assignee.percentage + 0` returns `"0.500"` — string concatenation.
**Why it happens:** Drizzle returns `numeric` columns as JS strings for precision safety.
**How to avoid:** Always `parseFloat(assignee.percentage ?? "0")` for display. Never do arithmetic on it. LRM produces its own fresh percentage values.
**Warning signs:** `"50.000"` displayed as-is without formatting; `NaN` in totals row arithmetic.

### Pitfall 5: Combobox onInputValueChange vs. controlled value
**What goes wrong:** Combobox `value` (selected item) and `onInputValueChange` (search term) are separate concerns. Clearing the field may not fire `onValueChange` if the user just clears the input.
**Why it happens:** base-ui Combobox distinguishes between the display input text and the committed value.
**How to avoid:** Watch `onValueChange` for `undefined`/`null` to detect clear. D-17 (clearing refundOf does NOT reset split) is handled here — only update split on a positive selection.
**Warning signs:** Split reset unexpectedly on clear; stale `refundOf` value after user deletes the text.

### Pitfall 6: `buildColumns` signature change for onEdit callback
**What goes wrong:** `TransactionColumns.tsx` `buildColumns(setDeleteId)` currently takes only one arg. Adding `onEdit` callback requires updating every callsite.
**Why it happens:** `buildColumns` passes callbacks to column cell renderers that fire actions up to the parent.
**How to avoid:** Add `onEdit` as a second parameter: `buildColumns(setDeleteId, onEdit)`. Update `TransactionsTable.tsx` to accept and pass `onEdit`. Update `Transactions.tsx` to pass the `setState` handler.
**Warning signs:** TypeScript error in TransactionsTable where buildColumns is called.
