# Phase 03.4: Transaction Forms UI - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can create and edit any of the six transaction types through a single Sheet-based form. The form conditionally renders type-specific fields, a split/assignee section, and a refund linker. The "Add transaction" button and the "Edit" action in the transactions table ⋮ menu (both disabled in 03.3) are wired in this phase. API patches within scope: update PATCH to accept full discriminated union, add description filter for refund combobox.

</domain>

<decisions>
## Implementation Decisions

### Form Container
- **D-01:** 560px side Sheet (right) for both create and edit. Wider than AccountSheet (440px) to accommodate type-specific field groups + split section. Sheet stays open while the transactions list is visible behind it.
- **D-02:** Same `TransactionSheet` component used for create (no `transaction` prop / null) and edit (transaction row data as prop). Initialized with existing field values as `defaultValues` for edit mode. `key` prop at call site forces remount on transaction switch — same pattern as AccountSheet.
- **D-03:** "Add transaction" button (currently disabled in Transactions.tsx) wired to open TransactionSheet with no transaction prop.
- **D-04:** "Edit" action in the TransactionsTable ⋮ dropdown (currently disabled with tooltip "Available in next update") wired to open TransactionSheet with the row's transaction data.
- **D-05:** "Delete" action available in the edit form footer (in addition to the table ⋮ menu). Uses the existing `useDeleteTransaction` hook. Confirm dialog before delete. On success, close Sheet + invalidate `["transactions"]` query.

### Transaction Type
- **D-06:** Type is **fully mutable** in both create and edit. Shown as a standard `Select` in all modes — no read-only badge, no disabled state.
- **D-07:** When type changes (create or edit), type-specific fields from the previous type are cleared. Shared base fields (accountId, amount, date, description, merchant, assignees, tags) are preserved.
- **D-08:** The PATCH payload for edit will use the discriminated union schema (`createTransactionSchema`) rather than the current flat `updateTransactionSchema`. API patch required: update `PATCH /api/transactions/:id` to validate with `createTransactionSchema` (or a new `updateTransactionSchema` that includes the type discriminant). This ensures type-specific required fields are enforced on type change (e.g. changing to `income` requires `incomeType`).

### Type-Specific Fields (conditional rendering by type)
- **expense:** categoryId (Select — required), no extra fields
- **refund:** categoryId (Select — optional), refundOf (combobox — optional; see Refund Linker)
- **income:** incomeType (Select — required: direct_deposit, e_transfer, cash, cheque, other), incomeSource (Input — optional)
- **transfer:** toAccountId (Select — required; distinct from accountId/from-account)
- **settlement:** settledAccountId (Select — optional)
- **contribution:** investmentType (Select — optional: tfsa, rrsp, fhsa, resp, non_registered, other)

### Split Section
- **D-09:** Split section is **always visible** below the type-specific fields. No collapsing.
- **D-10:** Starts **empty** in create mode — no assignees pre-populated. User adds assignees one by one via an "Add assignee" member picker.
- **D-11:** When an assignee is added or removed, split resets to even distribution across the new count (Largest Remainder Method — LRM runs client-side in SplitSection). This matches the PROJECT.md requirement: "Adding/removing an assignee resets the split to even across the new count."
- **D-12:** **Global % | $ toggle** for the entire split section. All rows show both % and $ — input mode (editable) vs. computed (display-only). Toggling between modes recalculates all rows via LRM.
- **D-13:** In edit mode, split section is initialized from the existing transaction's `assignees` array from the GET response (D-04 in 03.2 CONTEXT — joined response includes `assignees: [{memberId, amountCents, percentage}]`).

### Refund Linker
- **D-14:** For `type = refund`, a searchable combobox (`refundOf` field) appears in the type-specific fields section. Labeled "Refund of" with a "(optional)" hint.
- **D-15:** Combobox queries `GET /api/transactions` with a new `description` search filter param (API patch within this phase). Results show: description/merchant • date • formatted amount. Debounced on input.
- **D-16:** Selecting an original transaction pre-fills the split section with the original transaction's assignees (amounts and percentages). User can override.
- **D-17:** Clearing the `refundOf` field does NOT reset the split — the pre-filled split stays unless the user explicitly modifies it.
- **D-18:** The description search filter in the API (`?description=...`) performs a case-insensitive substring match on `transactions.description || transactions.merchant`. Only returns non-deleted transactions for the current org.

### Edit Mode — Import Value Display
- **D-19:** For transactions with an `importBatchId` (sourced from CSV import), show the original imported description/merchant below the current field value as a muted caption: `└ Original: GROCERY STORE #0023`. Displayed only when the field has an `originalDescription` or `originalMerchant` value AND it differs from the current value.
- **D-20:** The GET response already includes all transaction columns — no extra API call needed. If `originalDescription` / `originalMerchant` columns don't exist in the schema yet, they should be added in this phase (or deferred note: may require schema patch).

### Component Decomposition
- **D-21:** Shell/Form split per 02.1 D-10:
  - `TransactionSheet.tsx` — Sheet chrome (560px, right side), open/close state, Sheet header with title ("Add transaction" | "Edit transaction")
  - `TransactionForm.tsx` — all form fields, mutations, `useAppForm` call; receives `transaction: Transaction | null` and `onClose`
- **D-22:** Type-specific sections extracted as separate components:
  - `TransactionTypeFields.tsx` — conditional render based on `type` field value; renders the correct type-specific fields via `form.Subscribe`
  - Each section stays simple enough to inline; extract to named components (`ExpenseFields`, `RefundFields`, etc.) if any section exceeds ~15 lines
- **D-23:** Split section extracted as its own component tree:
  - `SplitSection.tsx` — owns % vs $ mode toggle, LRM logic; receives `amount`, `value` (assignees array), `onChange`, `orgMembers`
  - `AssigneeRow.tsx` — single assignee row with member name, % input, $ display (or reverse), remove button
  - `AddAssigneeButton.tsx` or inline button — member picker (combobox or select) that adds a new assignee
- **D-24:** Separate hook files for custom hooks in `apps/web/src/components/transactions/hooks/`:
  - `useTransactionForm.ts` — wraps `useAppForm` call with schema, defaultValues factory, and submit handler logic
  - Any other form-level derived state that would clutter TransactionForm.tsx

### Data-Access Hooks (new, in `apps/web/src/lib/data-access/transactions/`)
- **D-25:** `useCreateTransaction` — POST /api/transactions
- **D-26:** `useUpdateTransaction(id)` — PATCH /api/transactions/:id (payload: discriminated union)
- **D-27:** `useGetTransaction(id)` — GET /api/transactions/:id; used to load transaction data into edit form. Returns same joined shape as list (D-05 in 03.2 CONTEXT).
- **D-28:** `useSearchTransactions(description)` — GET /api/transactions?description=...&limit=20; used by refund combobox. Debounced; only fires when description.length >= 2.
- `useDeleteTransaction` and `useRestoreTransaction` already exist — no changes needed for list. Edit form delete uses the same `useDeleteTransaction` hook.

### Vercel Skills Enforcement (per CLAUDE.md)
- **D-29:** `vercel-react-best-practices` applies to all components in this phase. Specifically: no waterfall fetches (all queries fire at top level in TransactionForm/TransactionSheet), no setState-during-render, `useCallback`/`useMemo` for stable split section callbacks.
- **D-30:** `vercel-composition-patterns` applies. No boolean props for type-specific behavior — use `form.Subscribe` to derive type and conditionally render. No `renderX` prop patterns. `ref` passed as regular prop (React 19).
- **D-31:** `web-design-guidelines` applies. All inputs have `autocomplete` attributes, all icon-only buttons have `aria-label`, truncation uses `min-w-0`.

### Claude's Discretion
- LRM algorithm implementation detail (how % remainder is distributed across assignees)
- Exact Tailwind classes for Split section layout (% input width, $ display alignment)
- Whether `AddAssigneeButton` is a `<Select>` or a `<Combobox>` (whichever is cleaner given existing member list size)
- Error message copy for form validation
- Whether `originalDescription`/`originalMerchant` columns are present in the schema — if missing, defer D-19 display to a follow-up phase and note in plan

</decisions>

- `TransactionFormSchema` (alias for `createTransactionSchema`) already exported from `@ploutizo/validators`

- `Avatar` / `AvatarGroup` from `@ploutizo/ui/components/avatar` — for member picker in SplitSection
- `useGetOrgMembers` from `@/lib/data-access/org` — member list for assignee picker
- `useDeleteTransaction`, `useGetTransactions`, `useRestoreTransaction` — already exist in data-access/transactions

### Established Patterns
- Shell/Form split: `AccountSheet.tsx` → `AccountForm.tsx` (exact pattern to replicate)
- Loading gate: render spinner while async data loads, then mount form with complete `defaultValues` (no `useEffect`/`form.reset`)
- `form.Subscribe` for conditional rendering based on field values (used in AccountForm for ownership-dependent memberIds)
- `key` prop at call site forces remount on entity switch: `key={transaction?.id ?? "new"}`
- `onSettled` query invalidation: invalidate `["transactions"]` on create, update, and delete success
- Existing `formatCurrency(cents)` in `apps/web/src/lib/formatCurrency.ts` — use for amount display in split rows and refund combobox results

### Integration Points

- `apps/api/src/routes/transactions.ts` — Two patches: (1) PATCH accepts discriminated union, (2) GET accepts `description` query param

## Specific Ideas

- Component decomposition model explicitly confirmed: TransactionSheet → TransactionForm → SplitSection/AssigneeRow + TypeSpecificFields + separate `hooks/` files for custom hooks (useTransactionForm, etc.)
- "Add assignee" button in SplitSection: member picker (select or combobox from org members)
- Refund combobox shows: description/merchant • date • formatted amount in each option
- Delete in edit form footer mirrors AccountSheet Archive action pattern — AlertDialog confirm, then mutate, then onClose
