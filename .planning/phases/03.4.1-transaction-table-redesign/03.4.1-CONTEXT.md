# Phase 03.4.1: Transaction v2 — Schema Migration, Form & Table Redesign - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Three interlinked deliverables: (1) DB migration — drop 5 columns, add 3; (2) Form redesign — locked auto-filled descriptions, shadcn Toggle assignee section, Settlement "Paid from" update, notes field, contribution account picker showing member name; (3) Table redesign — signed/color-coded amounts, single `A → B` account column, refund sub-line, Internal filter shortcut.

Date grouping in the table is explicitly **deferred** — keep table structure as-is.

</domain>

<decisions>
## Implementation Decisions

### Schema Migration
- **D-01:** Drop columns: `merchant`, `income_source`, `to_account_id`, `settled_account_id`, `investment_type`
- **D-02:** Add columns: `counterpart_account_id` (UUID FK nullable → accounts), `raw_description` (text nullable), `notes` (text nullable)
- **D-03:** `counterpart_account_id` is the single replacement for both `to_account_id` (transfers) and `settled_account_id` (settlements). Data migration: copy existing `to_account_id` values into `counterpart_account_id` for transfer rows; copy `settled_account_id` values for settlement rows.
- **D-04:** `description` is now required (NOT NULL) on all 6 types. Existing nullable `description` rows must be backfilled before adding the NOT NULL constraint — use `merchant` value as fallback, then a default string if both are null.
- **D-05:** `raw_description` stores the original bank/import memo (currently in `merchant` for imported rows). Data migration: copy `merchant` into `raw_description` for rows with an `importBatchId`.

### Validators
- **D-06:** `baseTransactionSchema` — remove `merchant` field; add `notes: z.string().optional()`; `description` is already `.min(1)` required (no change needed).
- **D-07:** Per-type schema changes:
  - `incomeTransactionSchema` — remove `incomeSource` field entirely
  - `transferTransactionSchema` — rename `toAccountId` → `counterpartAccountId`
  - `settlementTransactionSchema` — rename `settledAccountId` → `counterpartAccountId` (optional)
  - `contributionTransactionSchema` — remove `investmentType` field entirely
- **D-08:** `updateTransactionSchema` updated to match; `counterpartAccountId` replaces both FK fields.

### Form — Removed Fields
- **D-09:** `income_source` text input removed from `IncomeFields.tsx` entirely (field gone from schema).
- **D-10:** `investmentType` Select removed from `ContributionFields.tsx` entirely (field gone from schema).

### Form — Locked Descriptions
- **D-11:** Auto-filled locked descriptions are generated client-side from selected account names, in real time as the user picks accounts:
  - Transfer: `Transfer from [From Account] to [To Account]`
  - Settlement: `Settlement — paid from [Account]` (or `Settlement` if no counterpart set)
  - Contribution: `[Account Name] contribution`
  - Linked refund: `Refund of [original transaction description]`
- **D-12:** Lock is UI-only — no schema flag. Locked state = description field rendered as read-only with a lock icon or muted style + an "Edit" / unlock affordance inline. Unlocking makes the field a normal text input; override persists after save.
- **D-13:** Non-auto-filled types (expense, income, unlinked refund) show a normal required text input for description.

### Form — Settlement "Paid from"
- **D-14:** Settlement "Paid from" field maps to `counterpartAccountId`. Shows tracked accounts only (tracked = all household accounts). No untracked options (Cash/E-Transfer/Cheque/Other) in this phase — deferred for later.
- **D-15:** `SettlementFields.tsx` replaces `settledAccountId` field with `counterpartAccountId` field using the same account list Select.

### Form — Assignee Toggle Section
- **D-16:** Replaces the current `SplitSection` component (or wraps it). New design:
  - Top area: one shadcn `Toggle` (pressed/unpressed) per org member. Member name + avatar on the toggle button.
  - When a toggle is pressed: member is added to the split; split recalculates to even distribution (LRM) across all pressed members.
  - When a toggle is un-pressed: member is removed; remaining members' splits recalculate even (LRM).
  - Below toggles: "Customize split" `Collapsible` (shadcn) — collapsed by default.
  - When expanded: shows the same % | $ toggle rows as the current `SplitSection` (editable per-member %, $ display or vice versa; LRM on manual edit).
- **D-17:** Create mode default:
  - 2+ org members → all toggles un-pressed (empty start, same as today).
  - Single org member → that member's toggle auto-pressed (100% split).
- **D-18:** Edit mode: toggles initialized from the existing transaction's `assignees` array — pressed for members in the array, un-pressed for absent members. Customize split expander initialized with their amounts/percentages.
- **D-19:** Component naming: `AssigneeSection.tsx` for the new top-level component; the existing `SplitSection.tsx` and `AssigneeRow.tsx` can be reused inside the Customize split expander.

### Form — Contribution Account Picker
- **D-20:** `ContributionFields.tsx` — account picker (counterpartAccountId or a separate account field?) shows member name alongside account type. Example label: `TFSA — Emily`. Achieved by joining accounts with their owner orgMember name in the picker options.
- **Note:** Contribution uses `accountId` (primary account for the contribution). No `counterpartAccountId` needed for contributions — the account already identifies the investment account.

### Form — Notes Field
- **D-21:** `notes` (new text nullable) field added to the base form. Always rendered at the bottom of the form, side by side with the tags field (`TransactionTagPicker`). Optional on all types.

### Table — Amount Column
- **D-22:** Amount column is signed + color-coded per type:
  - Expense: red, negative (`-$X.XX`)
  - Refund: green, positive (`+$X.XX`)
  - Income: green, positive (`+$X.XX`)
  - Transfer / Settlement / Contribution: neutral (no sign, no color)

### Table — Account Column
- **D-23:** Single Account column renders:
  - For types with a counterpart (`counterpartAccountId` set): `[Primary Account] → [Counterpart Account]`
  - For all other types: just the primary account name
  - Requires API to join and return `counterpartAccount` (name) alongside `account` in the transaction row response.

### Table — Refund Sub-line
- **D-24:** Refund rows get a sub-line rendered below the main row:
  - Format: `↩ [date] · [formatted amount]` — shows original transaction's date and amount
  - Clicking the sub-line opens the original transaction in `TransactionSheet` edit mode
  - Requires API to join and return `refundOfTransaction: { id, date, amountCents }` in refund rows.
  - Implementation: custom cell rendering in `TransactionColumns.tsx` for the description or a dedicated sub-row mechanism. **Do not use DataGrid row grouping or hack the DataGrid — render sub-line inside the description cell or via a stacked cell renderer.**

### Table — Internal Filter Shortcut
- **D-25:** Filter bar gets an "Internal" shortcut that selects `type IN (transfer, settlement, contribution)` together. In `transactionSearch.ts` / `TransactionFilterFields.tsx`, this is a composite type option. API already supports `type` filter; may need to support multiple type values or a `typeGroup=internal` param.

### Table — Date Grouping
- **D-26:** Date grouping is **deferred**. Table rows are not grouped by date in this phase. No DataGrid modifications or row sentinel hacks. Current date column stays as-is.

### Claude's Discretion
- Exact lock/unlock affordance styling (lock icon, muted border, edit button position)
- Whether the sub-line in the refund cell is inside the description `<td>` or a separate rendered element
- Tailwind classes for the Toggle button sizing and spacing
- Whether `AssigneeSection.tsx` completely replaces `SplitSection.tsx` or wraps it
- Whether "Internal" filter uses a new `typeGroup` API param or comma-separated `type` values

</decisions>

<specifics>
## Specific Ideas

- Contribution account picker shows member name alongside account type (e.g., "TFSA — Emily") — per REQUIREMENTS §4.1
- Locked description unlock affordance: UI-only lock with inline edit affordance (not a separate modal)
- Refund sub-line: `↩ date · amount` format; clicking opens original transaction in `TransactionSheet` edit mode
- Settlement "Paid from" = tracked accounts only; untracked options (Cash/E-Transfer/etc.) explicitly deferred
- Single org member → assignee auto-pressed on create; multi-member → empty start

</specifics>

## Deferred Ideas

- **Date grouping in table** — rows grouped by date with section headers; deferred to avoid DataGrid hacks
- **Untracked "Paid from" options** — Cash, E-Transfer, Cheque, Other for Settlement; would need a `counterpart_method` column; explicitly deferred; add to backlog if needed
- **investmentType field** — removed from form in this phase; if needed in future, add as a separate column or notes convention
- **income_source text field** — removed from form; free-text income context can go in notes if needed

</deferred>
