# Phase 03.4.1 Plan 08: Filter Operator URL Persistence and API Enforcement Summary

Filter operators now round-trip through the URL as `{field}_op` params and are enforced in `buildConditions` with full Drizzle operator branching for all 6 filter fields.

## What Was Built

**Task 1 тАФ URL persistence (web):**
- `transactionSearchSchema` gains 6 `_op` fields (`type_op`, `accountId_op`, `categoryId_op`, `assigneeId_op`, `tagIds_op`, `dateRange_op`)
- `buildCleanSearch` strips default-value operators (`is`, `is_any_of`, `between`) so only non-default values appear in URLs
- `filtersToSearch` encodes operator into URL params; `empty`/`not_empty` operators persist without a value param
- `searchToFilters` restores operator from URL params on page load тАФ operators survive refresh
- URL-sync effect simplified: operators are now authoritative in URL, no local preservation needed
- `TransactionQueryParams` extended with operator fields; `fetchTransactions` forwards all 6 to API
- `hasActiveFilters` extended to detect `empty`/`not_empty` operator-only filter states

**Task 2 тАФ API enforcement:**
- `ListQueryParams` gains all 6 `_op` fields
- `ne`, `not`, `notInArray` imported from drizzle-orm
- `buildConditions` operator branching:
  - `type`: `ne`/`notInArray` for `is_not`; `inArray`/`eq` for `is` (default)
  - `accountId`: `ne` for `is_not`
  - `categoryId`: `isNull`/`isNotNull` for `empty`/`not_empty`; `ne` for `is_not`
  - `assigneeId`: `NOT EXISTS` for `empty`/`is_not`; `EXISTS` for `not_empty`/`is`
  - `tagIds`: per-tag `EXISTS` for `includes_all`; `NOT EXISTS+inArray` for `is_not_any_of`/`excludes_all`; `NOT EXISTS` no filter for `empty`; `EXISTS` no filter for `not_empty`
  - `dateRange`: `after` uses only `dateFrom` (gte); `before` uses only `dateTo` (lte); `between` unchanged
- `routes/transactions.ts` parses all 6 operator query params via `c.req.query()`

## Deviations from Plan
