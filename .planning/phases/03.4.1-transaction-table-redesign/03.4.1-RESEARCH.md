## Summary

Phase 03.4.1 is a three-layer synchronized change: DB schema migration, backend validator/query update, and frontend form+table redesign. The key constraint is that all three layers must move in lockstep — the DB migration generates new TypeScript types via Drizzle inference, which cascade to validators, which cascade to the API service layer, which cascade to `TransactionRow`, which cascade to every consumer.

The migration is non-trivial because it involves both DDL and DML: new columns must be added nullable first, backfill SQL must run, then constraints can be tightened and old columns dropped. Drizzle Kit generates only DDL — the backfill SQL must be manually inserted into the generated migration file before applying.

On the frontend, the `TransactionFormValues` flat superset pattern (required by TanStack Form's static `keyof` typing) means field renames must be reflected in exactly three places atomically: `TransactionFormValues` in types.ts, `buildDefaultValues` in useTransactionForm.ts, and `toApiPayload` in useTransactionForm.ts. The new `AssigneeSection` component wraps existing `SplitSection` (do not delete it) and adds shadcn Toggle buttons above. Both `Toggle` and `Collapsible` are already installed in `packages/ui`.

**Primary recommendation:** Execute in strict dependency order — migration first, validators second, API third, `TransactionRow` type fourth, form types fifth, then form UI components, then table columns last.

### Pattern 1: Drizzle Migration with Manual DML Backfill

Drizzle Kit generates DDL-only migration files. For this phase's data backfills, the executor must:
1. Run `pnpm --filter @ploutizo/db migrate:generate` to produce the DDL skeleton
2. Manually insert `UPDATE` SQL statements before the DROP statements
3. Run `pnpm --filter @ploutizo/db migrate:apply`

**Ordering within the generated migration file (must be this exact order):**

```sql
-- Step 1: Add new nullable columns (safe — no constraint yet)
ALTER TABLE transactions ADD COLUMN counterpart_account_id UUID REFERENCES accounts(id);
ALTER TABLE transactions ADD COLUMN raw_description TEXT;
ALTER TABLE transactions ADD COLUMN notes TEXT;

-- Step 2: Backfill counterpart_account_id (MANUALLY INSERTED)
UPDATE transactions SET counterpart_account_id = to_account_id
  WHERE type = 'transfer' AND to_account_id IS NOT NULL;
UPDATE transactions SET counterpart_account_id = settled_account_id
  WHERE type = 'settlement' AND settled_account_id IS NOT NULL;

-- Step 3: Backfill raw_description (MANUALLY INSERTED)
UPDATE transactions SET raw_description = merchant
  WHERE import_batch_id IS NOT NULL AND merchant IS NOT NULL;

-- Step 4: Backfill description NOT NULL (MANUALLY INSERTED)
UPDATE transactions SET description = COALESCE(merchant, 'Imported transaction')
  WHERE description IS NULL;

-- Step 5: Add NOT NULL constraint on description (now safe)
ALTER TABLE transactions ALTER COLUMN description SET NOT NULL;

-- Step 6: Drop old columns (Drizzle generates these — move after backfills)
ALTER TABLE transactions DROP COLUMN merchant;
ALTER TABLE transactions DROP COLUMN income_source;
ALTER TABLE transactions DROP COLUMN to_account_id;
ALTER TABLE transactions DROP COLUMN settled_account_id;
ALTER TABLE transactions DROP COLUMN investment_type;
```

### Pattern 2: TX_COLUMNS Update

`TX_COLUMNS` in `apps/api/src/lib/queries/transactions.ts` is the single source of truth for the API response projection. Remove 5 fields, add 3 direct fields, add 2 join-derived fields. The list query also needs a second accounts table alias for the counterpart join and a self-join for `refundOfTransaction`.

```typescript
// Source: apps/api/src/lib/queries/transactions.ts (verified)
// Remove from TX_COLUMNS:
//   merchant, incomeSource, incomeSource, toAccountId, settledAccountId, investmentType

// Add to TX_COLUMNS:
counterpartAccountId: transactions.counterpartAccountId,
rawDescription: transactions.rawDescription,
notes: transactions.notes,
counterpartAccountName: counterpartAccounts.name,   // from LEFT JOIN alias
refundOfTransaction: {                               // from self-join or subquery
  id: refundSource.id,
  date: refundSource.date,
  amountCents: refundSource.amount,
},
```

The counterpart join requires a Drizzle alias:
```typescript
import { alias } from 'drizzle-orm'
const counterpartAccounts = alias(accounts, 'counterpart_accounts')
// LEFT JOIN counterpartAccounts ON transactions.counterpartAccountId = counterpartAccounts.id
```

The refundOfTransaction join requires a self-alias:
```typescript
const refundSource = alias(transactions, 'refund_source')
// LEFT JOIN refundSource ON transactions.refundOf = refundSource.id
```

### Pattern 3: Internal Filter — inArray

Current `buildConditions` type filter uses `eq()`. For comma-separated type values:

```typescript
// Source: apps/api/src/lib/queries/transactions.ts (verified — inArray already imported at line 18)
if (params.type) {
  const types = params.type.split(',').map(t => t.trim())
  if (types.length === 1) {
    conditions.push(eq(transactions.type, types[0] as typeof transactions.type._.data))
  } else {
    conditions.push(inArray(transactions.type, types as typeof transactions.type._.data[]))
  }
}
```

The route handler must split `type` query param on `,` before passing to `buildConditions`.

### Anti-Patterns to Avoid

- **Running migrate:apply before inserting backfill SQL:** The generated file will drop columns before data is copied; data is permanently lost.
- **Dropping `investmentTypeEnum` in the same migration as the column drop:** The enum is also used in `packages/db/src/schema/enums.ts`. Removing the PG enum type requires verifying no other table references it. Safe default: remove the enum from the Drizzle schema definition (which stops generating new references) but leave the PG type in the DB (inert after column drop). [VERIFIED: investmentTypeEnum only used in transactions.ts within packages/db/src — safe to remove from schema file once column drops; PG DROP TYPE is optional.]
- **Updating `TransactionRow` before fixing `TX_COLUMNS`:** TypeScript will compile but runtime shape mismatch causes undefined fields in the UI.
- **Using `&&` for conditional JSX in locked description:** UI-SPEC uses a lock toggle — use ternary operators per `web-design-guidelines` skill rule.
- **Inline component definitions in `buildColumns`:** Cell renderers must be defined outside the `buildColumns` call or as stable references to satisfy `rerender-*` rules.

## Common Pitfalls

### Pitfall 1: TransactionRow Cascade — Updating One Layer Without Others

**What goes wrong:** `TX_COLUMNS` is updated in the API, but `TransactionRow` in `useGetTransactions.ts` still has old field names → components compile (TypeScript infers the old shape) but runtime fields are undefined.

**Why it happens:** `TransactionRow` is a manually-maintained interface, not auto-generated from Drizzle. It doesn't fail to compile when `TX_COLUMNS` changes — it silently diverges.

**How to avoid:** Update `TransactionRow`, `TransactionFormValues` (types.ts), `buildDefaultValues`, and `toApiPayload` in the same commit as the API `TX_COLUMNS` change. Run `pnpm typecheck` after each layer.

**Warning signs:** Amount column shows `NaN`, account column shows blank, refund sub-line never renders.

### Pitfall 2: Migration Column Drop Before Backfill

**What goes wrong:** Drizzle Kit generates a migration that drops columns first (it doesn't know about the data dependency). If applied as-is, `to_account_id` data is gone before it's copied to `counterpart_account_id`.

**Why it happens:** Drizzle Kit compares schema snapshots — it sees "column removed" and generates `DROP COLUMN`. It has no awareness of data migration needs.

**How to avoid:** After generating the migration, open the `.sql` file and move all `DROP COLUMN` statements to the bottom. Insert all `UPDATE` backfill statements above the drops. Always inspect generated migrations before applying.

**Warning signs:** All transfer/settlement rows have `counterpart_account_id = NULL` after migration.

### Pitfall 3: `description` NOT NULL Constraint Without Backfill

**What goes wrong:** Adding `NOT NULL` to an existing column with existing NULL rows causes the migration to fail with a Postgres constraint violation error.

**Why it happens:** Postgres enforces NOT NULL at the point of `ALTER COLUMN ... SET NOT NULL`. Existing NULLs are not tolerated.

**How to avoid:** Run `UPDATE transactions SET description = COALESCE(merchant, 'Imported transaction') WHERE description IS NULL` before the ALTER statement.

**Warning signs:** Migration fails with: `ERROR: column "description" of relation "transactions" contains null values`.

### Pitfall 4: Toggle Pressed State Override

**What goes wrong:** The `Toggle` component from packages/ui has a default pressed style of `data-[state=on]:bg-muted`. The UI-SPEC requires `bg-primary/10 text-primary border-primary/30`. If only these classes are added to `className`, Tailwind specificity may not override the base style.

**Why it happens:** Both classes target `data-[state=on]`. Tailwind doesn't have natural specificity — last-in-cascade wins, which is unpredictable with component libraries.

**How to avoid:** Use `cn()` to compose the base Toggle classes and append the override. If the override doesn't take effect, use Tailwind's `!` important modifier or restructure the conditional className. Do NOT modify `packages/ui/src/components/toggle.tsx` — override at the usage site via className (CLAUDE.md constraint).

**Warning signs:** Toggle pressed state shows gray (`bg-muted`) instead of primary tint.

### Pitfall 5: `toAccountId` Required in transferTransactionSchema

**What goes wrong:** Current `transferTransactionSchema` has `toAccountId: z.string().uuid()` as required (not optional). Renaming to `counterpartAccountId` means validation fails if `toAccountId` is still passed from the form.

**Why it happens:** Both the form payload (`toApiPayload`) and the schema must be updated together. If only the schema is updated, the form sends the old field name and validation rejects it.

**How to avoid:** Update `transferTransactionSchema` and `toApiPayload` in the same PR wave. Test with a transfer creation immediately after.

### Pitfall 6: `filtersToSearch` Ignores Multi-value Type

**What goes wrong:** The "Internal" filter sets `type = "transfer,settlement,contribution"` as a multi-value string. The existing `filtersToSearch` function reads `f.values[0]` for the type field — so it would correctly capture the comma-joined string IF the filter chip stores the value as a single string. But if it stores as separate values in the array, only `values[0]` ("transfer") would be sent.

**How to avoid:** The Internal shortcut should be a single chip with a single value string `"transfer,settlement,contribution"` stored as `values[0]`. This is consistent with how the existing type filter works (single value). The `filtersToSearch` mapping needs no changes. The API route splits on `,` to get the array.
