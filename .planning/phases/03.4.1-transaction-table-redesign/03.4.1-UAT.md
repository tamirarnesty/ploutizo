## Tests

### 1. Amount column — sign and color coding
expected: Expense rows show "−$X.XX" in red; income/refund rows show "+$X.XX" in green; transfer/settlement/contribution rows show "$X.XX" in muted gray with no sign prefix.
result: pass

### 2. Account column — A→B for transfers
expected: For transfer or settlement rows that have a counterpart account, the account column shows "{From Account} → {To Account}" format. For regular expense/income rows, only the primary account name shows.
result: pass

### 3. Refund sub-line
expected: For a transaction with type "refund" that is linked to an original, a second line appears below the description in the table: "↩ {original date} · {original amount}". Clicking that sub-line opens the original transaction in the sheet for editing.
result: pass

### 4. "Internal" type filter shortcut
expected: In the type filter dropdown, an "Internal" option exists. Selecting it filters the list to show only transfer, settlement, and contribution rows (all three at once).
result: pass

### 5. Type badge opacity for internal types
expected: The type badge (e.g. "Transfer", "Settlement", "Contribution") appears at reduced opacity (visually dimmer/muted) compared to expense and income badges which appear at full opacity.
result: pass

### 6. Transfer form — destination account picker
expected: When creating or editing a transfer, the type-specific field shows a "To Account" (or equivalent) picker for the destination account (counterpart). Selecting an account updates the locked description to "Transfer from {source} to {destination}".
result: pass

### 7. Settlement form — "Paid from" label
expected: When the transaction type is "Settlement", the type-specific field shows the label "Paid from" with an "(optional)" hint. Selecting an account updates the locked description.
result: pass

### 8. Contribution form — investment account picker
expected: When the transaction type is "Contribution", the type-specific field shows a picker for investment accounts only (accounts of type INVESTMENT). The options are formatted as "INVESTMENT — {Account Name}".
result: issue
reported: "The 'INVESTMENT — ' prefix shows in the selected/trigger state, causing long account names to overflow and get truncated. Picker works but selected value is hard to read."
severity: minor

### 9. Income form — no income source field
expected: When the transaction type is "Income", there is NO income source field or picker. The form shows only the base fields (account, amount, date, description) plus type-specific controls, with no trace of a removed incomeSource input.
result: pass

### 10. Notes field
expected: Every transaction form (create and edit, all types) shows a "Notes" text area below description. Typing a note and saving persists it. Opening the same transaction in edit mode shows the saved note.
result: issue
reported: "Notes persist on create. On edit: saving updated note, then reopening edit sheet shows old note. After page refresh the updated value is shown — stale cache not invalidated after PATCH."
severity: major

### 11. Assignee toggles — single-member auto-press on create
expected: Open a new transaction sheet (create mode) in a single-member org. The assignee toggle for the only member appears already pressed (highlighted/active) without any user interaction. Submitting without touching the assignee section should succeed with the single member assigned.
result: pass

### 12. Assignee toggles — press/unpress recalculates split
expected: In create mode with 2+ org members, all toggles start un-pressed. Pressing member A assigns 100% to them. Pressing member B splits evenly (50%/50%). Un-pressing member A gives 100% back to B. The "Customize split" collapsible appears when at least one member is pressed.
result: skipped
reason: No 2+ member org available — will re-test after adding a second member

### 13. Assignee edit mode — pre-populated from existing transaction
expected: Opening an existing transaction with assignees in edit mode shows the correct members pre-pressed (highlighted). Members not in the split appear un-pressed. If the split is non-even, the "Customize split" collapsible opens automatically.
result: skipped
reason: Partially verified for single-member only — will re-test for 2+ member org alongside test 12

### 14. Locked description — transfer template
expected: When type is "Transfer" and both source and destination accounts are selected, the description field is read-only and shows "Transfer from {source account name} to {destination account name}". A lock icon appears on the left. Updating either account updates the description in real time.
result: issue
reported: "Locked description and real-time update work correctly. However, the same account can be selected in both 'Account' (from) and 'To account' (to) fields — resulting in 'Transfer from TD Visa to TD Visa'. Same-account transfers should not be allowed."
severity: major

### 15. Locked description — unlock button works
expected: Clicking the edit/pencil icon on a locked description field unlocks it, allowing free-text editing. After unlocking, the lock icon disappears and the field becomes a normal editable input. The custom value persists through submission.
result: pass

### 16. Description lock resets on type change
expected: Unlock the description on a transfer transaction, type custom text, then change the type to "Expense". The description field becomes a regular editable input (no lock). Change type back to "Transfer" — the lock reappears and description resets to the auto-generated template (the custom text from before is gone).
result: pass

### 17. Transfer/settlement create — no description validation error
expected: Create a new transfer transaction. Select source and destination accounts. The description field auto-fills with "Transfer from X to Y" (locked). Submit without touching the description field — the form submits successfully with no "Description is required" error.
result: pass

### 18. Type change clears counterpart account from table
expected: Find or create a transfer transaction that shows "Account A → Account B" in the table. Edit it and change the type to "Expense". Save. In the table, the account column for that row now shows only "Account A" with no "→ Account B" arrow. The counterpart is fully cleared.
result: pass

### 19. Category "is not" filter includes uncategorized rows
expected: In the filter bar, set the category filter to "is not [pick any category]". The results include transactions that have NO category (uncategorized/null rows) in addition to transactions with a different category. Uncategorized transactions are not silently excluded.
result: pass

### 20. Filter operators persist across page refresh
expected: Set any non-default filter operator (e.g. type "is not", category "is empty", date "before"). Refresh the page. The filter operator is still active — the results and the filter UI reflect the operator that was set before refresh.
result: issue
reported: "Multiple date filter issues: (1) Wrong operator set — shows duplicate 'is', missing 'before'/'after'/'is not'/'not between'. Correct set: single-date ops [is, is not, before, after] + range ops [between, not between]. (2) Calendar always shows dual-month range picker regardless of operator — single-date ops should show single-date picker. (3) Display label shows 'From Apr 20, 2026' for single-date ops — should show just 'Apr 20, 2026'. (4) Switching from a single-date op (e.g. 'after') to 'between' throws an error and fully clears the filter. (5) Calendar value button should update in real time as dates are selected (before Apply). (6) Duplicate React key warning in FilterOperatorDropdown for the two 'is' entries."
severity: major
