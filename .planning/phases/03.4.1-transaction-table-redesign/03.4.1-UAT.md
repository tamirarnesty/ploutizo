### 1. Amount column — sign and color coding

result: pass

### 2. Account column — A→B for transfers
expected: For transfer or settlement rows that have a counterpart account, the account column shows "{From Account} → {To Account}" format. For regular expense/income rows, only the primary account name shows.
result: pass

### 3. Refund sub-line
expected: For a transaction with type "refund" that is linked to an original, a second line appears below the description in the table: "↩ {original date} · {original amount}". Clicking that sub-line opens the original transaction in the sheet for editing.
result: pass

### 4. "Internal" type filter shortcut
expected: In the type filter dropdown, an "Internal" option exists. Selecting it filters the list to show only transfer, settlement, and contribution rows (all three at once).
result: pass

### 5. Type badge opacity for internal types

result: pass

### 6. Transfer form — destination account picker
expected: When creating or editing a transfer, the type-specific field shows a "To Account" (or equivalent) picker for the destination account (counterpart). Selecting an account updates the locked description to "Transfer from {source} to {destination}".
result: pass

### 7. Settlement form — "Paid from" label
expected: When the transaction type is "Settlement", the type-specific field shows the label "Paid from" with an "(optional)" hint. Selecting an account updates the locked description.
result: pass

### 8. Contribution form — investment account picker trigger

result: pass

### 9. Income form — no income source field
expected: When the transaction type is "Income", there is NO income source field or picker. The form shows only the base fields (account, amount, date, description) plus type-specific controls, with no trace of a removed incomeSource input.
result: pass

### 10. Notes field — cache invalidation after edit
expected: Every transaction form (create and edit, all types) shows a "Notes" text area below description. Typing a note and saving persists it. Opening the same transaction in edit mode immediately shows the saved note — no page refresh required.
result: pass

### 11. Assignee toggles — single-member auto-press on create
expected: Open a new transaction sheet (create mode) in a single-member org. The assignee toggle for the only member appears already pressed (highlighted/active) without any user interaction. Submitting without touching the assignee section should succeed with the single member assigned.
result: pass

### 12. Assignee toggles — press/unpress recalculates split
expected: In create mode with 2+ org members, all toggles start un-pressed. Pressing member A assigns 100% to them. Pressing member B splits evenly (50%/50%). Un-pressing member A gives 100% back to B. The "Customize split" collapsible appears when at least 2 members are pressed.
result: pass

### 13. Assignee edit mode — pre-populated from existing transaction
expected: Opening an existing transaction with assignees in edit mode shows the correct members pre-pressed (highlighted). Members not in the split appear un-pressed. If the split is non-even, the "Customize split" collapsible opens automatically.
result: pass

### 14. Transfer form — same-account guard
expected: When creating or editing a transfer, the "To Account" destination picker only shows accounts other than the currently selected source account. Selecting the same account for both source and destination is not possible. If the source changes to match the destination mid-session, the destination clears automatically.
result: pass

### 15. Locked description — unlock button works
expected: Clicking the edit/pencil icon on a locked description field unlocks it, allowing free-text editing. After unlocking, the lock icon disappears and the field becomes a normal editable input. The custom value persists through submission.

result: pass
