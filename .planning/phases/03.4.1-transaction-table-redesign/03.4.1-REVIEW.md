# Phase 03.4.1: Code Review Report

**Reviewed:** 2026-04-20
**Depth:** standard
**Files Reviewed:** 22
**Status:** issues_found

## Summary

This phase implements filter operator URL persistence, API-side operator enforcement, and a transaction table redesign. The API layer (`buildConditions`, route guards) and the URL sync logic (`filtersToSearch`/`searchToFilters`) are well-structured. One critical bug exists in the form: the auto-generated description for locked types (transfer/settlement/contribution/refund) is never written back to form field state, causing schema validation to fail on create for those types. Four warnings cover a stale-column PATCH edge case, over-broad error catching, a disconnect between the auto-press UI state and form state, and a `ne()` NULL-exclusion trap. Three info items cover dead code.

## Critical Issues

### CR-01: Locked description never persists to form state — schema validation fails on create

**File:** `apps/web/src/components/transactions/TransactionForm.tsx:330-342`

**Issue:** When the description is locked (type is transfer/settlement/contribution, or refund with a linked original), `lockedValue` is rendered as `<InputGroupInput readOnly value={lockedValue}>` but `field.handleChange(lockedValue)` is never called. Form field state (`value.description`) remains `''` from `buildDefaultValues`. When the form submits, `toApiPayload` reads `value.description.trim()` → `''`, and `baseTransactionSchema.description` is `z.string().min(1)`. The `createTransactionSchema.safeParse` call in the `onSubmit` validator rejects it with "Description is required.", silently blocking submission with no visible error.

This only affects **create mode** (field state starts as `''`). In edit mode the stored description from `buildDefaultValues` is still in state, so it submits correctly.

**Fix:** Sync `lockedValue` into field state when it changes. Replace the `<InputGroupInput>` block's render with an effect-driven sync OR add a `useEffect` alongside the field:

```tsx
// Inside the form.AppField name="description" render function,
// after the isLocked/lockedValue computation:
useEffect(() => {
  if (isLocked) {
    field.handleChange(lockedValue)
  }
}, [isLocked, lockedValue])
```

Alternatively, in `toApiPayload` fall back to computing the template description from form values when `description` is empty and `type` is locked — but the effect approach is cleaner and keeps field state as the source of truth.

## Warnings

### WR-01: Stale type-specific columns after type change on PATCH

**File:** `apps/api/src/services/transactions.ts:139-163`

**Issue:** `updateTransactionScalarsQuery` only sets the columns present in `updateData`. The discriminated union schema for the new type does not include the old type's fields, so switching types (e.g., `transfer → expense`) sends no `counterpartAccountId: null` — the old FK value stays in the DB. Consequence: the table's Account column continues to render "A → B" for what is now an expense. Same issue for `refundOf` (refund → expense) and `incomeType` (income → expense).

**Fix:** In `updateTransaction`, detect when `type` is changing and explicitly null the fields that don't apply to the new type:

```ts
const typeSpecificNulls: Record<string, unknown> = {}
if (data.type) {
  if (!['transfer', 'settlement'].includes(data.type)) typeSpecificNulls.counterpartAccountId = null
  if (data.type !== 'refund') typeSpecificNulls.refundOf = null
  if (data.type !== 'income') typeSpecificNulls.incomeType = null
  if (!['expense', 'refund'].includes(data.type)) typeSpecificNulls.categoryId = null
}
// merge into updateData before passing to updateTransactionScalarsQuery
```

### WR-02: Over-broad catch returns 400 for non-validation errors on PATCH

**File:** `apps/api/src/routes/transactions.ts:155-162`

**Issue:** The try/catch around `updateTransaction` returns `400 BAD_REQUEST` for **any** thrown error — DB connection failures, transaction deadlocks, and internal assertion errors all surface as "bad request" to the client. Only the `validateSplitSum` throw is intentional.

**Fix:** Narrow the catch to only handle the known validation throw, e.g. by using a custom error class:

```ts
// In services/transactions.ts — throw a typed error
class ValidationError extends Error { readonly isValidation = true }
// ...
if (splitError) throw new ValidationError(splitError)

// In routes/transactions.ts
} catch (err) {
  if (err instanceof ValidationError) {
    return c.json({ error: { code: 'BAD_REQUEST', message: err.message } }, 400)
  }
  throw err  // re-throw unexpected errors for the global error handler
}
```

### WR-03: Auto-press UI state is disconnected from form state (single-member org)

**File:** `apps/web/src/components/transactions/AssigneeSection.tsx:49-55`

**Issue:** When `orgMembers.length === 1` in create mode, `initialPressedIds` is set to `[orgMembers[0].id]` so the Toggle renders visually pressed. However `buildDefaultValues` sets `assignees: []` and `onChange` is never called during initialization — form state has zero assignees while the UI shows one member as selected. The user sees a green pressed toggle but no split is actually stored until they click something.

**Fix:** Call `onChange` once on mount via `useEffect` to align form state with the auto-pressed UI state:

```ts
const initialAssignees = useMemo(
  () => (!transaction && orgMembers.length === 1)
    ? lrmSplit(amountCents, [orgMembers[0].id])
    : [],
  // Run once on mount; amountCents intentionally excluded from deps
  // eslint-disable-next-line react-hooks/exhaustive-deps
  []
)

useEffect(() => {
  if (initialAssignees.length > 0) onChange(initialAssignees)
}, []) // one-time init
```

### WR-04: `ne()` on nullable `categoryId` excludes NULL rows from "is_not" results

**File:** `apps/api/src/lib/queries/transactions.ts:151-156`

**Issue:** When `categoryId_op === 'is_not'`, the condition `ne(transactions.categoryId, params.categoryId)` is used. In SQL, `NULL != X` evaluates to `UNKNOWN` (not TRUE), so uncategorized transactions are silently excluded from the result set. Users who select "Category is not Food" would expect uncategorized transactions to appear.

**Fix:** Wrap the `ne` condition with an OR for NULL:

```ts
conditions.push(
  sql`(${transactions.categoryId} != ${params.categoryId} OR ${transactions.categoryId} IS NULL)`
)
```

Or use Drizzle's `or(ne(...), isNull(...))`.

## Info

### IN-01: Dead `onTypeChange` prop in TransactionTypeFields

**File:** `apps/web/src/components/transactions/TransactionTypeFields.tsx:46, 175, 182`

**Issue:** The `onTypeChange` optional prop was added for "Plan 06" to call `setIsDescriptionUnlocked(false)` from the parent. That path was superseded by the `UnlockResetter` component in `TransactionForm.tsx`. `TransactionForm.tsx` never passes `onTypeChange` to `TransactionTypeFields`. The prop, comment, and wiring to `TypeSelectField` are unreachable dead code.

**Fix:** Remove the `onTypeChange` prop from `TransactionTypeFieldsProps` and `TypeSelectField`, and delete the `onTypeChange?.()` call and comment.

### IN-02: Duplicated "locked/internal" type constant

**File:** `apps/web/src/components/transactions/TransactionForm.tsx:119`, `apps/web/src/components/transactions/TransactionColumns.tsx:47`

**Issue:** The set of "internal" types `['transfer', 'settlement', 'contribution']` is hardcoded in two places: `LOCKED_TYPES` in `TransactionForm.tsx` and `isInternalType` in `TransactionColumns.tsx`. Any future type addition requires updating both.

**Fix:** Export a shared `INTERNAL_TRANSACTION_TYPES` constant from `types.ts` and import it in both files.

### IN-03: `IncomeFields` is a permanently empty no-op component

**File:** `apps/web/src/components/transactions/IncomeFields.tsx`

**Issue:** The component always returns `null` and its props type is `Record<string, never>`. The comment says it was emptied in Plan 05 and its call site removed. The file and its export are now unused dead code.

**Fix:** Delete `IncomeFields.tsx` and remove any remaining import if one exists.
