## Gaps Summary

4 gaps block the phase goal. 2 are blockers that prevent correct form submission for 4 of the 6 transaction types; 2 are correctness bugs in the API layer.

**Blocker 1 — CR-01:** Transfer, settlement, contribution, and linked-refund transactions cannot be created. The locked description is rendered read-only but never written to form state. `toApiPayload` sends `description: ''`, and `createTransactionSchema.safeParse` rejects it. Only edit mode is unaffected (stored description populates state from `buildDefaultValues`). Fix: `useEffect(() => { if (isLocked) field.handleChange(lockedValue) }, [isLocked, lockedValue])` inside the `AppField name="description"` render callback.

**Blocker 2 — WR-03:** Single-member orgs see the assignee toggle auto-pressed visually, but form state has `assignees: []`. Submit sends no assignees. Fix: `useEffect(() => { if (initialAssignees.length > 0) onChange(lrmSplit(amountCents, [orgMembers[0].id])) }, [])` on mount in `AssigneeSection`.

**Warning 1 — WR-01:** PATCH requests that change transaction type leave stale FK values in DB. Transfer→expense preserves `counterpartAccountId`, so the account column still shows "A→B". Fix: detect type change in `updateTransaction` and explicitly null non-applicable type-specific columns.

**Warning 2 — WR-04:** The `categoryId is_not` filter silently excludes uncategorized transactions (NULL `categoryId`) because SQL `ne()` on NULL evaluates to UNKNOWN. Fix: `or(ne(transactions.categoryId, params.categoryId), isNull(transactions.categoryId))`.
