## Summary

Phase 02.2 adds a light/dark/system theme toggle to the app using `next-themes`. The work is narrow and well-defined: install `next-themes` in `packages/ui`, export `ThemeProvider` and `useTheme` via the existing `@ploutizo/ui` exports map, wrap the root in `ThemeProvider` inside `__root.tsx`, and add a `ThemeToggle` cycling icon button to the sidebar footer.

The existing CSS infrastructure is already complete — `globals.css` has `.dark { ... }` token overrides and `@custom-variant dark (&:is(.dark *))`. next-themes applies the `.dark` class to `<html>` at exactly the right level. No CSS changes are required. All icons (Monitor, Sun, Moon) and shadcn components (Button, Tooltip) are already installed.

The key TanStack Start integration point is how `ThemeProvider` interacts with the `shellComponent` / `RootDocument` pattern. The `RootDocument` renders `<html>` and `<body>` — `ThemeProvider` wraps the `<body>` children, not the `<html>` element itself. This means `suppressHydrationWarning` on `<html>` is the correct technique to silence React's warning about server/client class attribute mismatch.

**Primary recommendation:** Install `next-themes` in `packages/ui`, wrap `<body>` content in `ThemeProvider` inside `RootDocument`, add `suppressHydrationWarning` to `<html>`, create `ThemeToggle` in `packages/ui/src/components/theme-toggle.tsx`, export `ThemeProvider`/`useTheme` from `@ploutizo/ui`, and update `AppSidebar` footer.

### Pattern 1: next-themes ThemeProvider placement in TanStack Start

**What:** TanStack Start's `shellComponent` renders the full HTML document including `<html>` and `<body>`. `ThemeProvider` from next-themes must wrap the content rendered inside `<body>` so it can inject the `.dark` class on `<html>` (next-themes walks up to the nearest configurable ancestor). Use `attribute="class"` to specify class-based theming. Add `suppressHydrationWarning` to `<html>` to silence the React hydration warning caused by next-themes adding the class after SSR.

**When to use:** Always — this is the only correct integration pattern for TanStack Start.

**Existing `RootDocument` (before):**
```tsx
const RootDocument = ({ children }: { children: React.ReactNode }) => (
  <html lang="en">
    <head>
      <HeadContent />
    </head>
    <body>
      <QueryClientProvider client={queryClient}>
        <ClerkProvider appearance={{ theme: shadcn }}>
          <TokenInitializer />
          {children}
        </ClerkProvider>
      </QueryClientProvider>
      <Scripts />
    </body>
  </html>
)
```

**After (key changes: `suppressHydrationWarning` on `<html>`, `ThemeProvider` wrapping `<body>` content):**
```tsx
// Source: next-themes README + CONTEXT.md D-10
const RootDocument = ({ children }: { children: React.ReactNode }) => (
  <html lang="en" suppressHydrationWarning>
    <head>
      <HeadContent />
    </head>
    <body>
      <ThemeProvider
        attribute="class"
        defaultTheme="system"
        enableSystem
        disableTransitionOnChange
        storageKey="theme"
      >
        <QueryClientProvider client={queryClient}>
          <ClerkProvider appearance={{ theme: shadcn }}>
            <TokenInitializer />
            {children}
          </ClerkProvider>
        </QueryClientProvider>
      </ThemeProvider>
      <Scripts />
    </body>
  </html>
)
```

Note: `ThemeProvider` wraps `QueryClientProvider` and `ClerkProvider` so all children have theme context. `<Scripts />` stays outside since it doesn't need theme context.

### Pattern 2: ThemeToggle component (cycling icon button)

**What:** A client-side component in `packages/ui` that calls `useTheme()` from next-themes, renders the correct icon for the current resolved theme, and cycles through System → Light → Dark on click. Wrapped in a shadcn Tooltip for hover label.

**When to use:** Placed in `SidebarFooter`, left of `<UserButton />`.

```tsx
// Source: CONTEXT.md D-05, D-06, D-07 + UI-SPEC.md
"use client"

import { useTheme } from "next-themes"
import { Monitor as MonitorIcon, Sun as SunIcon, Moon as MoonIcon } from "lucide-react"
import { Button } from "@ploutizo/ui/components/button"
import {
  Tooltip,
  TooltipContent,
  TooltipProvider,
  TooltipTrigger,
} from "@ploutizo/ui/components/tooltip"

const cycleMap = { system: "light", light: "dark", dark: "system" } as const
type Theme = keyof typeof cycleMap

export function ThemeToggle() {
  const { theme, setTheme } = useTheme()
  const current = (theme ?? "system") as Theme
  const label = current.charAt(0).toUpperCase() + current.slice(1)

  return (
    <TooltipProvider>
      <Tooltip>
        <TooltipTrigger asChild>
          <Button
            variant="ghost"
            size="icon"
            className="size-8"
            onClick={() => setTheme(cycleMap[current])}
          >
            {current === "system" && <MonitorIcon />}
            {current === "light" && <SunIcon />}
            {current === "dark" && <MoonIcon />}
            <span className="sr-only">{label}</span>
          </Button>
        </TooltipTrigger>
        <TooltipContent>{label}</TooltipContent>
      </Tooltip>
    </TooltipProvider>
  )
}
```

### Anti-Patterns to Avoid

- **Dynamically switching `ClerkProvider appearance` prop:** Not needed — the shadcn Clerk theme reads the `.dark` class automatically. D-08 locks this.
- **Manual `dark:` utility overrides in ThemeToggle:** The ghost button inside sidebar context already receives the correct hover tokens via `bg-sidebar-accent`/`text-sidebar-accent-foreground` from the radix-nova preset. No `dark:` overrides needed.
- **Wrapping `<html>` with ThemeProvider:** TanStack Start renders `<html>` inside `RootDocument` — you can't make ThemeProvider a parent of `<html>`. Instead, wrap `<body>` content and let next-themes set the class attribute on the nearest document-level ancestor.
- **Using `theme` directly from `useTheme()` without null-guard:** `theme` can be `undefined` on first render. Guard with `?? "system"`.
- **Installing next-themes in `apps/web` instead of `packages/ui`:** The ThemeToggle component and ThemeProvider live in `packages/ui`. next-themes must be in the same package that imports it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| System preference detection | Manual `window.matchMedia('(prefers-color-scheme: dark)')` listener | `next-themes` with `enableSystem` | next-themes handles listener cleanup, SSR, hydration mismatch, and localStorage sync |
| localStorage persistence | Custom `useEffect` + `localStorage.setItem` | `next-themes` `storageKey` | next-themes handles storage reads before hydration to prevent flash of wrong theme |
| Hydration flash prevention | Inline script in `<head>` to read localStorage and set class | `next-themes` `disableTransitionOnChange` + standard ThemeProvider placement | Already handled by next-themes — this is the main value-add |

**Key insight:** Theme management has a well-known hydration mismatch problem: the server doesn't know the user's localStorage preference, so the initial render may use the wrong theme class. next-themes solves this with a blocking script injection approach. Rolling this correctly is non-trivial.

## Common Pitfalls

### Pitfall 1: `useTheme()` returns undefined theme on first render

**What goes wrong:** `theme` from `useTheme()` is `undefined` before the component mounts and next-themes reads localStorage/system preference. If the toggle renders `theme === "system"` and `theme` is `undefined`, no icon appears.

**Why it happens:** next-themes defers theme resolution to client-side to avoid SSR/hydration mismatch. Before mount, `theme` is `undefined`.

**How to avoid:** Guard with `const current = (theme ?? "system") as Theme`. The fallback "system" is safe because `defaultTheme="system"` is set, so the first client paint will resolve to "system" anyway.

**Warning signs:** Empty toggle button on initial page load; TypeScript type error if `theme` is passed directly to the cycle map.

### Pitfall 2: ThemeProvider placement outside body content

**What goes wrong:** If `ThemeProvider` is placed outside `<body>` (e.g., wrapping the entire `RootDocument` return including `<html>`), React will throw a hydration error because `<html>` is a special element.

**Why it happens:** TanStack Start's `shellComponent` renders the full HTML document tree. `ThemeProvider` renders a `<div>` wrapper which cannot be a parent of `<html>`.

**How to avoid:** Place `ThemeProvider` as the first child of `<body>`, wrapping only the content children. `suppressHydrationWarning` on `<html>` handles the class attribute mismatch. See Pattern 1 above.

**Warning signs:** React error: "Hydration failed because the initial UI does not match what was rendered on the server."

### Pitfall 3: next-themes installed in wrong package

**What goes wrong:** If `next-themes` is added to `apps/web` but the `ThemeToggle` and `ThemeProvider` re-export files live in `packages/ui`, you'll get a "Cannot resolve module 'next-themes'" error from `packages/ui` at build time.

**Why it happens:** Each package in the pnpm workspace has its own `node_modules` resolution scope. `packages/ui` imports from its own dependencies, not from `apps/web`'s.

**How to avoid:** Install next-themes exclusively in `packages/ui`: `pnpm add next-themes --filter @ploutizo/ui`.

**Warning signs:** Build error in `packages/ui` failing to resolve `next-themes` despite it appearing in the root lockfile.

### Pitfall 4: Missing `disableTransitionOnChange`

**What goes wrong:** Without `disableTransitionOnChange`, switching themes triggers CSS transitions on every element, causing a visible "flash" effect where background, text, and borders animate between colors simultaneously on theme switch.

**Why it happens:** Tailwind CSS transitions applied to elements conflict with the instant token swap that theme switching causes.

**How to avoid:** Always include `disableTransitionOnChange` prop on `ThemeProvider`. This is listed explicitly in the UI-SPEC.md ThemeProvider props.

**Warning signs:** Smooth color transition on theme toggle — looks unintentional and jarring.

### Pitfall 5: Tooltip nesting order

**What goes wrong:** If `<Tooltip>` wraps `<TooltipProvider>` (reversed), the tooltip will not function — `TooltipProvider` must be an ancestor of `Tooltip`.

**Why it happens:** The UI-SPEC.md button anatomy example shows `<Tooltip><TooltipProvider>` (provider nested inside), which appears to be reversed from the typical shadcn pattern.

**How to avoid:** Ensure `TooltipProvider` is the outermost wrapper: `<TooltipProvider><Tooltip><TooltipTrigger>...<TooltipContent>`. See Pattern 2 above for the correct structure.

**Warning signs:** Tooltip never appears on hover; no console error (Radix fails silently when context is missing).

## Open Questions

1. **`useTheme()` returns `theme` as the *stored* preference, not the resolved system preference**
   - What we know: `useTheme()` returns both `theme` (stored: "system" | "light" | "dark") and `resolvedTheme` ("light" | "dark" only). The icon should show the stored preference, not the resolved system value — e.g., show Monitor icon when `theme === "system"` even if the OS is dark.
   - What's unclear: Whether the UI-SPEC intends the icon to reflect the stored preference (what the user chose) or the resolved preference (what the screen actually shows). The UI-SPEC icon mapping uses `theme`, not `resolvedTheme`.
   - Recommendation: Use `theme` (stored preference) for the icon — this is what the user set, and "System" with Monitor icon correctly communicates "following OS". This is the pattern used in CONTEXT.md and UI-SPEC.md and should be followed exactly.
