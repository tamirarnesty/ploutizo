## Warnings

### WR-01: `truncate` has no effect on inline `<span>` — owners column overflows silently

**File:** `apps/web/src/components/accounts/AccountsTable.tsx:116`
**Issue:** `truncate` (`overflow: hidden; text-overflow: ellipsis; white-space: nowrap`) only works on block or inline-block elements. `<Text as="span">` renders as `display: inline`, so the class is a no-op. Long owner-name lists will overflow the cell rather than ellipsize. `min-w-0` is also ineffective here for the same reason — it only constrains flex children, not inline elements.
**Fix:**
```tsx
<Text as="span" variant="body-sm" className="block min-w-0 truncate text-muted-foreground">
  {names || '—'}
</Text>
```
Adding `block` makes `truncate` operative. Alternatively, render a `<div>` wrapper with `truncate` and keep the `<Text>` inside it without the truncation classes.

### WR-02: Archive test assertion never fails — any status is accepted

**File:** `apps/api/src/__tests__/accounts.test.ts:244-245`
**Issue:** `expect([200, 404]).toContain(res.status)` passes for both success and failure. The mock already sets up `db.update` to return a valid row, so the route always returns 200 — but the assertion would also pass if the route crashed and somehow returned 404. The test provides zero signal.
**Fix:** Mock `db.update` explicitly for the archive path and assert the exact outcome:
```ts
it('returns 200 for archive endpoint', async () => {
  const res = await app.request('/acct_1/archive', { method: 'DELETE' });
  expect(res.status).toBe(200);
  const body = (await res.json()) as { data: { id: string } };
  expect(body.data.id).toBe('acct_1');
});
```

## Info

### IN-01: Mutation responses typed as `Account` but missing `owners` field

**Files:** `apps/api/src/services/accounts.ts:39, 57, 71` / `apps/web/src/lib/data-access/accounts/useCreateAccount.ts:15` / `apps/web/src/lib/data-access/accounts/useUpdateAccount.ts:15` / `apps/web/src/lib/data-access/accounts/useArchiveAccount.ts:6`
**Issue:** `createAccount`, `updateAccount`, and `archiveAccountById` in the service return the raw `accounts` table row — no `owners` field. The web data-access hooks type these responses as `Promise<Account>`, but `Account.owners` (non-optional `AccountOwner[]`) will be absent at runtime. Today there is no crash path because all three hooks use `onSettled: invalidateQueries` and discard the returned value. However, if any hook is refactored to use `setQueryData` or `onSuccess` for optimistic updates, it will silently spread an `Account` with `owners: undefined` into the cache, causing `owners.map()` in `AccountsTable` to throw.
**Fix:** Either enrich mutation responses to include owners (match `listAccounts` join), or introduce a `AccountRow` type without `owners` for mutation return values and reserve `Account` for the enriched list response.

### IN-03: `ACCOUNT_TYPE_LABELS` typed as `Record<string, string>` — should use `AccountType`

**File:** `apps/web/src/components/accounts/AccountsTable.tsx:21`
**Issue:** `Record<string, string>` silently allows any string key. Using `Record<AccountType, string>` would make TypeScript flag any added or removed `AccountType` union member that isn't reflected in the label map.
**Fix:**
```ts
import type { Account, AccountType } from '@ploutizo/types';

const ACCOUNT_TYPE_LABELS: Record<AccountType, string> = {
  chequing: 'Chequing',
  savings: 'Savings',
  credit_card: 'Credit Card',
  prepaid_cash: 'Prepaid / Cash',
  e_transfer: 'e-Transfer',
  investment: 'Investment',
  other: 'Other',
};
```
