**Researched:** 2026-04-24
**Domain:** Bug fix — account API query + type shape + table cell render
**Confidence:** HIGH (all findings from direct code trace)

## Summary

This phase fixes two related bugs: the Owners column in AccountsTable.tsx always shows "—", and the form appears to not persist co-owners. Code trace reveals the form submit path and the API write path are fully working — `memberIds` is sent in the payload, the API handler calls `insertAccountMembers`/`replaceAccountMembers` inside a transaction, and data IS written to the `account_members` table.

The bugs are purely on the **display path**:

1. `listAccounts` query (`apps/api/src/lib/queries/accounts.ts`) selects only from `accounts` — no join on `account_members` or `org_members`. Owner names are never returned in the list response.
2. `Account` type (`packages/types/src/index.ts`) has no `owners` field. The shape does not exist for the table to consume.
3. AccountsTable.tsx Owners cell is hardcoded to render `—` with no data accessor at all.

The fix requires three coordinated changes: extend the list query to join owner data, extend the `Account` type, and wire the table cell to render the joined names.

**Primary recommendation:** Add a left-join on `account_members → org_members` in `listAccounts`, aggregate owner display names into `Account.owners`, and render them in the table cell. No changes to the form, API write handlers, or service layer are needed — those are verified correct.

## Bug Diagnosis (Verified by Code Trace)

### Bug 1: `listAccounts` returns no owner data

**File:** `apps/api/src/lib/queries/accounts.ts:9-18`

```typescript
// CURRENT — no join, returns bare account rows only
export async function listAccounts(orgId: string, includeArchived: boolean) {
  const condition = includeArchived
    ? eq(accounts.orgId, orgId)
    : and(eq(accounts.orgId, orgId), isNull(accounts.archivedAt));
  return db
    .select()
    .from(accounts)
    .where(condition)
    .orderBy(accounts.createdAt);
}
```

`account_members` and `org_members` are never joined. The route handler returns this directly — no owner names reach the client.

**Fix approach:** Add a left-join subquery or use Drizzle's `leftJoin` + `sql`array_agg`` to attach `owners: { id: string; displayName: string }[]` to each account row. Drizzle does not support `array_agg` returning typed objects natively — use raw SQL or a two-query approach (fetch accounts, then fetch all members for those account IDs in one query and merge in JS).

**Recommended: two-query merge in the query layer** — one `listAccounts` query, one `listAccountMemberDetails` for the returned account IDs, merged in the service layer. This avoids complex SQL and keeps the query functions simple. Alternatively, `array_agg` via `sql` template is also valid.

### Bug 2: `Account` type has no `owners` field

**File:** `packages/types/src/index.ts:12-23`

```typescript
// CURRENT — no owners field

```

## Architecture Patterns

### Recommended data flow after fix

```
GET /api/accounts
  └─ listAccounts(orgId) [accounts.ts query]
       ├─ SELECT accounts WHERE org_id = $orgId
       └─ SELECT account_members JOIN org_members WHERE account_id IN (...)
            └─ merge: accounts[].owners = [{ id, displayName }]
  └─ route: return { data: rows }
  
useGetAccounts() [React Query]
  └─ Account[] with owners: AccountOwner[]
  └─ AccountsTable receives accounts prop
       └─ Owners column cell: row.original.owners.map(o => o.displayName).join(', ') || '—'
```

### Two-query merge pattern (recommended)

```typescript
// In apps/api/src/lib/queries/accounts.ts
export async function listAccountMemberDetails(accountIds: string[]) {
  if (accountIds.length === 0) return [];
  return db
    .select({
      accountId: accountMembers.accountId,
      memberId: orgMembers.id,
      displayName: orgMembers.displayName,
    })
    .from(accountMembers)
    .innerJoin(orgMembers, eq(orgMembers.id, accountMembers.memberId))
    .where(inArray(accountMembers.accountId, accountIds));
}

// In apps/api/src/services/accounts.ts
export async function listAccounts(orgId: string, includeArchived: boolean) {
  const rows = await listAccountsQuery(orgId, includeArchived);
  if (rows.length === 0) return [];
  const members = await listAccountMemberDetails(rows.map(r => r.id));
  // Build index map for O(1) access (js-perf rule)
  const byAccount = new Map<string, AccountOwner[]>();
  for (const m of members) {
    const list = byAccount.get(m.accountId) ?? [];
    list.push({ id: m.memberId, displayName: m.displayName });
    byAccount.set(m.accountId, list);
  }
  return rows.map(r => ({ ...r, owners: byAccount.get(r.id) ?? [] }));
}
```

This keeps Drizzle query functions simple and uses `Map` for O(1) grouping per the `js-perf` skill rule.

### Table cell render pattern

```typescript
// AccountsTable.tsx — Owners column cell
cell: ({ row }) => {
  const owners = row.original.owners;
  if (!owners.length) {
```

## Common Pitfalls

### Pitfall 1: Modifying verified-working form/write path
**What goes wrong:** Unnecessary changes to `AccountForm.tsx`, `useCreateAccount`, `useUpdateAccount`, or `services/accounts.ts` introduce regressions.
**How to avoid:** These are verified correct by code trace. Scope the fix to query layer, types, and table cell only.

### Pitfall 2: FK mismatch — `memberId` vs `orgMembers.id`
**What goes wrong:** `accountMembers.memberId` is a FK to `org_members.id` (a UUID). But `OrgMember.id` in `@ploutizo/types` is also `orgMembers.id`. The form's `members.map(m => m.id)` uses `OrgMember.id` correctly. The join in the list query must also match on `eq(orgMembers.id, accountMembers.memberId)`.
**Warning signs:** Owner names not appearing even after the fix — verify the join condition.

### Pitfall 3: `Account` type change not propagated
**What goes wrong:** `Account` interface is extended in `packages/types` but consuming code (table, data-access hook return type) still sees old shape.
**How to avoid:** Run `pnpm turbo typecheck` after type extension — TypeScript errors will surface any consumer that needs updating.

### Pitfall 4: `inArray` with empty array panics in some DB drivers
**What goes wrong:** `inArray(accountMembers.accountId, [])` generates invalid SQL in some Drizzle/PG versions.
**How to avoid:** Guard `if (rows.length === 0) return []` before calling `listAccountMemberDetails`.
