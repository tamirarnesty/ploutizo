## Summary

This phase adds pending invitation cards to the Members list in the Household settings view. It requires three coordinated changes: (1) two new API endpoints on the Hono server to list and revoke Clerk invitations, (2) two new TanStack Query hooks plus an update to `useInviteMember`, and (3) a new `InvitedMemberRow` component integrated into `MembersSection`.

All infrastructure patterns already exist in the codebase. The new work follows established patterns exactly: raw `fetch` calls to Clerk REST API in the service layer, TanStack Query hooks in `data-access/org/`, and React component structure mirroring `MemberRow`. The non-obvious complexity is: (a) the Clerk REST API returns snake_case which must be mapped to camelCase in the service layer, (b) Clerk's revoke endpoint is `POST .../revoke` (not DELETE) and requires a `requesting_user_id` body parameter, and (c) an open question exists about whether the `status` filter on the list endpoint should include `'expired'` invitations to make D-11's expired card state reachable.

**Primary recommendation:** Follow the established `removeMember` / `useRemoveMember` / `MemberRow` patterns exactly. Every structural decision has a direct parallel in existing code.

### Pattern 1: Clerk REST API Service Function

> **Open Question:** The status filter on the GET call determines which cards appear in the UI. See Open Questions #3 before locking this pattern.

```typescript
// Source: existing apps/api/src/services/households.ts (inviteMember pattern)
// NOTE: status filter — include 'expired' if D-11's expired card state must be reachable;
//       use 'pending' only if expired invitations should never appear in the list.
export async function listInvitations(orgId: string): Promise<PendingInvitation[]> {
  // DECISION POINT: use ?status=pending OR ?status=pending&status=expired
  // See Open Questions #3 in this file.
  const url = `https://api.clerk.com/v1/organizations/${orgId}/invitations?status=pending&status=expired&limit=100`
  const res = await fetch(url, {
    headers: { Authorization: `Bearer ${process.env.CLERK_SECRET_KEY}` },
  })
  if (!res.ok) throw new DomainError(500, 'Failed to list invitations.', 'UNKNOWN')
  const body = await res.json() as {
    data: Array<{
      id: string
      email_address: string   // NOTE: snake_case from Clerk REST — NOT emailAddress
      status: string
      created_at: number      // Unix epoch — verify seconds vs ms (see Pitfall 6)
      expires_at: number | null
    }>
  }
  return body.data.map((inv) => ({
    id: inv.id,
    email: inv.email_address,
    status: inv.status as PendingInvitation['status'],
    // Clerk timestamps are seconds (10-digit) — multiply by 1000 for Date constructor.
    // Verify with console.log on first real response; 10-digit = seconds, 13-digit = ms.
    createdAt: new Date(inv.created_at * 1000).toISOString(),
    expiresAt: inv.expires_at ? new Date(inv.expires_at * 1000).toISOString() : null,
  }))
}
```

**Critical:** Clerk REST returns `email_address`, `created_at`, `expires_at` (snake_case). The service layer MUST map to camelCase before returning. Missing this transform causes silent runtime errors.

### Pattern 2: Revoke — POST to Clerk's `/revoke` Endpoint

```typescript
// Source: Clerk docs https://clerk.com/docs/reference/backend/organization/revoke-organization-invitation
export async function revokeInvitation(
  orgId: string,
  invitationId: string,
  requestingUserId: string  // REQUIRED by Clerk — pass getAuth(c).userId
) {
  const res = await fetch(
    `https://api.clerk.com/v1/organizations/${orgId}/invitations/${invitationId}/revoke`,
    {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${process.env.CLERK_SECRET_KEY}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ requesting_user_id: requestingUserId }),
    }
  )
  if (!res.ok && res.status !== 404) {
    throw new DomainError(500, 'Failed to revoke invitation.', 'UNKNOWN')
  }
  return { revoked: true }
}
```

**Critical:** Clerk revoke is `POST .../revoke`, not DELETE. It requires `requesting_user_id` in the body. The internal ploutizo API uses `DELETE /api/households/invitations/:id` (D-02), but the outbound Clerk call is always POST.

### Anti-Patterns to Avoid

- **Gating second query behind first `isLoading`:** Never do `if (membersQuery.isLoading) return <Skeleton/>` before calling `useGetOrgInvitations()`. React hooks must be called unconditionally. Both queries fire simultaneously.
- **Computing expiry state from dates only:** Never use `new Date(expiresAt) < new Date()` to determine "Expired" state. Use `status === 'expired'`. Date math is supplementary (display label only).
- **Using raw `fetch()` in web components:** All web-side API calls go through `apiFetch` — never raw `fetch`.
- **Forwarding refs on new components:** React 19 project. No `forwardRef` on `InvitedMemberRow` (vercel-composition-patterns `react19-no-forwardref`).
- **Forgetting `requesting_user_id` on Clerk revoke:** The Clerk REST call will 422 without it.
- **Fetching only `status=pending`:** If only pending invitations are fetched, `status === 'expired'` from D-11 is unreachable at render time. Include `status=expired` in the query (or resolve via Open Question #3).

## Common Pitfalls

### Pitfall 1: Clerk REST Returns Snake_Case

**What goes wrong:** `listInvitations` returns `email_address` instead of `email`, `created_at` instead of `createdAt`, `expires_at` instead of `expiresAt`. TypeScript won't catch this if the service function is typed loosely.

**Why it happens:** The Clerk REST API (raw `fetch`) uses snake_case. The Clerk JS SDK (not used here) would return camelCase. This project uses raw `fetch` in service functions.

**How to avoid:** Map explicitly in `listInvitations()`. The intermediate type for the raw Clerk response must be typed with snake_case fields. The returned `PendingInvitation[]` is camelCase.

**Warning signs:** `invitation.email` is `undefined` in the React component; `invitation.email_address` is defined.

### Pitfall 2: Revoke is POST /revoke, Not DELETE

**What goes wrong:** Service calls `DELETE https://api.clerk.com/v1/organizations/{org_id}/invitations/{id}` — which returns 405 Method Not Allowed.

**Why it happens:** The internal ploutizo API uses `DELETE /api/households/invitations/:id` (REST convention), but Clerk's API uses `POST .../revoke` as its action pattern.

**How to avoid:** Clerk revoke endpoint: `POST /v1/organizations/{orgId}/invitations/{invitationId}/revoke`.

### Pitfall 3: Missing `requesting_user_id` on Clerk Revoke

**What goes wrong:** Clerk returns 422 on every revoke call.

**Why it happens:** Clerk's revoke endpoint requires `requesting_user_id` in the JSON body. Omitting it causes validation failure.

**How to avoid:** In the route handler, extract `getAuth(c).userId` and pass it to `revokeInvitation()` as `requestingUserId`. The service encodes it as `requesting_user_id` in the POST body.

### Pitfall 4: `expires_at` May Be Null

**What goes wrong:** `new Date(null)` returns epoch; `expiresAt.toISOString()` throws on null.

**Why it happens:** The Clerk backend API docs show `expiresAt` as `number` on the type, but when `expires_in_days` is not set on the POST body (current code does NOT set it), Clerk may or may not populate `expires_at`. The mapping in `listInvitations` should guard: `inv.expires_at ? new Date(inv.expires_at * 1000).toISOString() : null`.

**Warning signs:** Runtime error "Cannot read properties of null" in `InvitedMemberRow` expiry label.

### Pitfall 5: `getInitials` with Email Input

**What goes wrong:** Email `user@example.com` contains no spaces, so `getInitials` splits into one segment and returns `U` (first char). This is correct and intentional behavior, but the planner should not expect two initials from an email.

**Why it happens:** `getInitials` splits on spaces and takes first char of each word. Emails have no spaces.

**How to avoid:** This is expected — just document it. The avatar will show a single letter, which is visually acceptable.

### Pitfall 6: Clerk Timestamps Are Seconds, Not Milliseconds

**What goes wrong:** `new Date(inv.created_at)` where `created_at = 1678886400` (10-digit) produces a date in the year 1970 (~53140 hours after epoch). "Invited X ago" shows "55 years ago" for all cards.

**Why it happens:** Clerk REST API example responses show 10-digit timestamps (Unix seconds). JavaScript's `new Date()` expects milliseconds. `new Date(1678886400)` = 1970; `new Date(1678886400 * 1000)` = 2023.

**How to avoid:** Always multiply Clerk timestamps by 1000: `new Date(inv.created_at * 1000)`. During implementation, log a real Clerk response and verify digit count: 10 digits = seconds (multiply by 1000), 13 digits = ms (use as-is).

**Warning signs:** "Invited 55 years ago" in the card. All dates show as epoch-era dates.

## Open Questions (RESOLVED)

1. **Should `expires_in_days: 7` be added to the `inviteMember` POST body?**
   - What we know: CONTEXT.md mentions a 7-day requirement; current code does not set this; there is no source REQUIREMENTS.md to confirm it.
   - What's unclear: Is this intentionally omitted or an oversight?
   - Recommendation: Planner should include setting `expires_in_days: 7` in the `inviteMember` service as part of this phase (small change, correct for both UX and the CONTEXT.md intent), but flag it to the user.
   - **RESOLVED:** The 7-day expiry requirement has no backing entry in REQUIREMENTS.md ("ASSUMED" per research). The Clerk default of 30 days is accepted for this phase. The `inviteMember` service will NOT be modified to add `expires_in_days: 7`. If the product requirement is confirmed, it should be tracked as a separate backlog item. Plans proceed with 30-day Clerk default.

2. **Does Clerk populate `expires_at` on invitations created without `expires_in_days`?**
   - What we know: The type says `expiresAt: number` (not nullable), suggesting it is always set.
   - What's unclear: Whether the value is meaningful when the default applies.
   - Recommendation: Guard with null check in service mapping anyway.
   - **RESOLVED:** Plan 01 Task 2 implements a null-guard in the service mapping (`expiresAt: invite.expires_at ? new Date(invite.expires_at * 1000) : null`). `PendingInvitation.expiresAt` typed as `Date | null` to handle this case safely.

3. **Should `GET /api/households/invitations` return `status=expired` invitations as well as `status=pending`?**
   - What we know: D-11 defines UI behavior for `status === 'expired'` (show "Expired" in destructive color). Clerk supports filtering by `status=pending` or `status=expired` as separate query params. If only `status=pending` is fetched, the `'expired'` branch in D-11 is a dead code path — invitations transition to expired asynchronously in Clerk's system, but they will never appear in the list.
   - Options: (a) Fetch `status=pending&status=expired` to show expired cards in the list (admin can see what has lapsed and revoke/clean up). (b) Fetch `status=pending` only; the `status === 'expired'` branch exists only as a race-condition guard.
   - Recommendation: Option (a) — include `status=expired` in the fetch query. This makes D-11's expired state actually reachable and gives admins visibility into expired invitations. Pattern 1 in this document defaults to option (a).
   - **RESOLVED:** Option (a) implemented. Plan 01 Task 2 uses `?status=pending&status=expired` on the Clerk API call. The `status === 'expired'` branch in D-11 is fully reachable.

- All data fetching in `apps/web` uses TanStack Query hooks from `apps/web/src/lib/data-access/`. No raw `fetch()` calls in components.
- All API requests from `apps/web` go through `apiFetch` — never raw `fetch()`.
- No `forwardRef` on new custom components — React 19 project. Pass `ref` as a regular prop.
- No `useState` for form field values — not applicable to this phase (no new forms).
- Never use raw HTML elements when a shadcn equivalent exists (`<button>` → shadcn, etc.).
- All text uses `<Text>` component from `@ploutizo/ui/components/text`.
- Never modify components in `packages/ui/src/components/reui/` or shadcn-generated files.
- `pnpm turbo typecheck` for type checking; `pnpm test` for tests. Never `npx tsc` or `npx vitest` directly.
- Icon-only buttons must have `aria-label` (revoke button: `aria-label="Revoke invite to {email}"`).
- All interactive elements must have visible focus indicators.
- Motion: any CSS transition > 200ms must respect `prefers-reduced-motion` (not applicable in this phase).
