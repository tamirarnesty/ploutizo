# Phase 03.4.6: Show invited member card in Household view (INSERTED) — Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

After sending an invite, admins see a pending member card in the Members list immediately. The API returns pending Clerk invitations via a new endpoint, and the UI renders an "Invited" card per pending invite (dimmed avatar, email as display name, time info, "Invited" badge). Admins can also revoke pending invitations from the card. No other household features are in scope.

</domain>

<decisions>
## Implementation Decisions

### API — Pending invitations endpoint
- **D-01:** Add `GET /api/households/invitations` — a new endpoint separate from `GET /api/households/members`. The existing `/members` contract is unchanged.
- **D-02:** Add `DELETE /api/households/invitations/:invitationId` for revoke — calls Clerk's DELETE invitation endpoint.
- **D-03:** `PendingInvitation` response shape: `{ id, email, createdAt, expiresAt, status }`. Expose Clerk's `status` field directly — do NOT calculate expiry/expired state from dates; use Clerk's own status field.
- **D-04:** Researcher flag: verify Clerk's pending invitation status enum values (expected: "pending", "expired", "revoked", "accepted"). Verify whether the current `inviteMember` service sets a custom expiry — app requirement is 7-day (REQUIREMENTS.md) but Clerk default is 30 days.

### Frontend — Data access
- **D-05:** New `useGetOrgInvitations()` hook with query key `['org-invitations']` — fires in parallel with `useGetOrgMembers()` in `MembersSection`.
- **D-06:** New `useRevokeInvitation()` mutation — calls `DELETE /api/households/invitations/:invitationId`. Invalidates `['org-invitations']` on settle.
- **D-07:** Update `useInviteMember` to also invalidate `['org-invitations']` on settle (currently only invalidates `['org-members']`).

### Invited card component
- **D-08:** New `InvitedMemberRow` component — not a variant of `MemberRow`. Data shape (email + inviteId) is too different from `OrgMember` to share a component.
- **D-09:** Card visual: dimmed (`opacity-60`), email as display name, initials from email for `UserAvatar`.
- **D-10:** Time display: "Invited [date]" as primary (relative time from `createdAt`) + "Expires in X days" as subtle secondary (from `expiresAt`).
- **D-11:** Warning state on expiry label when ≤ 2 days remaining OR when Clerk `status === 'expired'`. Show "Expired" text using Clerk status field — NOT manual date comparison.

### Revoke flow
- **D-12:** Trash/X button on `InvitedMemberRow` opens an `AlertDialog` with "Revoke invite to {email}?" and Revoke / Cancel actions. Matches the existing `MemberRow` remove pattern.

### Loading and empty state
- **D-13:** `MembersSection` fires both queries in parallel. Combined loading state: `isLoading = membersQuery.isLoading || invitationsQuery.isLoading`. Single skeleton block — no progressive loading.
- **D-14:** Empty state "This household has no other members." shows only when BOTH `members.length === 0` AND `invitations.length === 0`. If only pending invites exist (no active members), invites render normally with no empty state text.

### Folded Todos
- **"Show invited member card in Household view"** (`.planning/todos/pending/2026-04-21-show-invited-member-card-in-household-view.md`) — exact match for this phase.

</decisions>
