## Typed Context (Variables)

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, type it | `AppEnv = { Variables: { orgId: string } }`, applied to all routers | ✓ |
| No, keep it loose | `c.get('orgId') as string` cast in each handler | |

**User's choice:** Yes, type it
**Notes:** Full TypeScript safety preferred; boilerplate propagation is acceptable.

---

## zValidator Error Shape

| Option | Description | Selected |
|--------|-------------|----------|
| Factory wrapper | `appValidator(target, schema)` in `src/lib/validator.ts` with baked-in hook | ✓ |
| Inline hook everywhere | 3rd arg to `zValidator` repeated per call | |

**User's choice:** Factory wrapper
**Notes:** Single place to maintain error shape; `appValidator` import used everywhere.

---

## Domain Error Design

| Option | Description | Selected |
|--------|-------------|----------|
| Class-based errors | `DomainError` + `NotFoundError` extending `Error`; `app.onError` uses `instanceof` | ✓ |
| Tagged result objects | Services return `{ ok: false, code, message }` | |

**User's choice:** Class-based errors
**Notes:** Idiomatic Node.js; stack traces preserved; `onError` handler in `index.ts`.

---

## Service File Structure

| Option | Description | Selected |
|--------|-------------|----------|
| 2-file: service + queries | `services/X.ts` + `lib/queries/X.ts` for all routes | ✓ |
| 1-file: service only | Inline Drizzle queries in `services/X.ts` | |

**User's choice:** 2-file for all routes
**Notes:** Consistency with existing transactions pattern; simpler routes can still benefit from the separation.
