## Critical Issues

### CR-01: Hard-coded role mapping collapses all members to `admin`

**File:** `apps/api/src/services/webhooks.ts:93`
**Issue:** The `organizationMembership.created` handler maps both `org:admin` and every other role to `'admin'`. The ternary `data.role === 'org:admin' ? 'admin' : 'admin'` always produces `'admin'`, so org members with `org:member` (or any other role) are stored as admins in the local DB. This inflates effective privileges for any access-control logic that reads the local `role` column.
**Fix:**
```typescript
role: data.role === 'org:admin' ? 'admin' : 'member',
```

## Warnings

### WR-01: Partial regex validation — update path can save an invalid regex

**File:** `apps/api/src/services/merchant-rules.ts:46-48`
**Issue:** `updateMerchantRule` only validates the regex when **both** `data.matchType` and `data.pattern` are present in the patch payload. If a rule already has `matchType: 'regex'` in the DB and a client sends `{ pattern: '([unclosed' }` without `matchType`, the invalid pattern is persisted without validation.
**Fix:** Validate whenever `data.pattern` is being updated, regardless of whether `matchType` is included in the patch:
```typescript
// Fetch existing rule's matchType when pattern changes without explicit matchType
if (data.pattern !== undefined) {
  // If matchType not in patch, we need the stored value to decide
  const existing = await fetchMerchantRuleById(id, orgId)
  const effectiveMatchType = data.matchType ?? existing?.matchType
  if (effectiveMatchType) validateRegex(effectiveMatchType, data.pattern)
}
// OR: simpler — always try to compile if pattern is present, regardless of matchType
if (data.pattern !== undefined) {
  try { new RegExp(data.pattern) } catch {
    throw new DomainError(400, 'Invalid regular expression.', 'INVALID_REGEX')
  }
}
```

### WR-02: `updateAccount` strips type safety with `as Record<string, unknown>`

**File:** `apps/api/src/services/accounts.ts:37-39`
**Issue:** The service casts `updateData` to `Record<string, unknown>` before passing it to `updateAccountQuery`, which accepts `Partial<typeof accounts.$inferInsert>`. The cast hides the mismatch and bypasses TypeScript's check — any extra field from the schema that leaks through (e.g., `memberIds` is destructured but if the schema adds new relation fields in future, they will silently pass through or be silently dropped). The underlying query function signature already takes a `Partial<>` so this should be typed correctly.
**Fix:** Derive the type from the query function's parameter rather than casting:
```typescript
import type { updateAccount as updateAccountQuery } from '../lib/queries/accounts'
// Then type updateData as Parameters<typeof updateAccountQuery>[2]
// Or simply assert the Omit<> type directly without the Record<string,unknown> cast
const updated = await db.transaction(async (tx) => {
  const row = await updateAccountQuery(tx, id, orgId, updateData as Partial<typeof accounts.$inferInsert>)
  ...
})
```

### WR-03: `DomainError` message/code arguments are swapped in households service

**File:** `apps/api/src/services/households.ts:55, 60, 64, 68, 71, 83`
**Issue:** `DomainError` constructor signature is `(statusCode, message, code?)`. In `households.ts` the service passes the machine-readable code string as the `message` argument (e.g., `new DomainError(409, 'ALREADY_MEMBER')`). In `routes/households.ts` (lines 58, 75), the route handler reads `err.message` and returns it as `{ error: { code: err.message } }` — so the frontend receives `code: 'ALREADY_MEMBER'`, which works by accident. However, the comment on line 49 of the route ("DomainError message IS the error code") documents this intentional convention but it inverts the established contract from `errors.ts`. The `onError` handler in `index.ts` uses `err.code ?? 'DOMAIN_ERROR'` as the response code and `err.message` as the message — if any of these household DomainErrors escape the targeted try/catch, the response will have `code: 'DOMAIN_ERROR'` and `message: 'ALREADY_MEMBER'` which is backwards.

The errors should either use the proper `code` parameter, or the route catch should be audited to ensure it always catches before `onError`:
```typescript
// Option A: use the proper third argument
throw new DomainError(409, 'Already a member of this organisation.', 'ALREADY_MEMBER')
// Route reads: err.code → 'ALREADY_MEMBER' ✓

// Option B (current pattern): document that these MUST stay in the targeted catch
// and add a type guard so uncaught DomainErrors from this service are caught reliably
```

### WR-04: `archiveAccount` query does not set `updatedAt`

**File:** `apps/api/src/lib/queries/accounts.ts:93-100`
**Issue:** `archiveAccount` sets `archivedAt` but not `updatedAt`. Every other mutation in this codebase (e.g., `updateAccount` at line 52, `archiveCategory` is the only exception — but categories have a separate `updatedAt` issue too) stamps `updatedAt: new Date()`. Inconsistency means `updatedAt` does not reflect the archival timestamp, breaking any sync/audit logic that depends on it.

Note: `archiveCategory` in `lib/queries/categories.ts:58-65` has the same omission.
**Fix:**
```typescript
// accounts.ts archiveAccount
.set({ archivedAt: new Date(), updatedAt: new Date() })

// categories.ts archiveCategory
.set({ archivedAt: new Date(), updatedAt: new Date() })
```

### WR-05: `updateCategory` query does not set `updatedAt`

**File:** `apps/api/src/lib/queries/categories.ts:49-55`
**Issue:** `updateCategory` calls `.set(data)` directly, passing through the caller-supplied `Partial<>` without stamping `updatedAt: new Date()`. All other update queries (accounts, merchant-rules, households) explicitly set `updatedAt`. If the caller does not include `updatedAt` in `data`, the column is never updated on a category PATCH.
**Fix:**
```typescript
const rows = await db
  .update(categories)
  .set({ ...data, updatedAt: new Date() })
  .where(and(eq(categories.id, id), eq(categories.orgId, orgId)))
  .returning();
```

### IN-01: `DrizzleTransaction` type is duplicated across three query files

**File:** `apps/api/src/lib/queries/accounts.ts:7`, `apps/api/src/lib/queries/categories.ts:6`, `apps/api/src/lib/queries/merchant-rules.ts:6`
**Issue:** The same `DrizzleTransaction` type alias is declared identically in three files. This is minor duplication but means future changes require three edits.
**Fix:** Extract to a shared file (e.g., `apps/api/src/lib/queries/types.ts`) and re-export from each query file.

### IN-02: `isAllowedParty` exported but never used at runtime for JWT validation

**File:** `apps/api/src/index.ts:39-41`
**Issue:** `isAllowedParty` is exported (presumably for testing) but it is not wired into the `clerkMiddleware` call — `authorizedParties` is a static string array that does not use the regex function. The comment on line 36 notes "Dynamic subdomain validation (isAllowedParty) is applied at the app layer in Phase 2+", but there is no Phase 2+ guard in the current code. Until it is wired in, any `*.ploutizo.app` subdomain JWT will be rejected by Clerk's static `authorizedParties` check (not validated via `isAllowedParty`). This is safe (errs toward rejection) but the export and comment suggest intent that is not yet implemented.
**Fix:** Either wire it in when Clerk middleware supports a custom validator, or add a TODO comment making the incomplete state explicit, and do not export the function until it is used.

### IN-03: `handleOrgMembershipCreated` silently drops membership if user row does not exist

**File:** `apps/api/src/services/webhooks.ts:80-81`
**Issue:** If the `user.created` webhook was missed (e.g., delivery failure) before `organizationMembership.created` fires, `user` will be `null` and the function returns without inserting the member row. There is no logging, retry mechanism, or dead-letter handling, so the membership is permanently lost in the local DB. This is a known limitation of best-effort webhooks, but it warrants a comment so future maintainers understand why the guard exists.
**Fix:** Add a comment noting the dependency and the acceptable failure mode:
```typescript
// user.created must have been processed first; if the user row is missing (missed
// webhook), we cannot insert the membership — acceptable: tenantGuard upserts the
// org, but there is no equivalent safety net for membership. A future admin
// reconciliation job can re-sync from Clerk.
if (!user) return;
```

### IN-04: `reorderCategories` and `reorderMerchantRules` wrap single-purpose queries in unnecessary `db.transaction()`

**File:** `apps/api/src/services/categories.ts:13-19`, `apps/api/src/services/merchant-rules.ts:23-27`
**Issue:** Both service functions open a `db.transaction()` that then calls the query function (which performs the sequential updates). The transaction is appropriate — sequential updates should be atomic. However, the query layer already accepts a `tx` argument to participate in *outer* transactions, so the service layer transaction wrapper serves a purpose. This is fine. No action required — noted here only because the transaction wrapping appears inconsistent with `createMerchantRule` / `createCategory` which do not use transactions. The inconsistency is fine given reorder is multi-row; this is informational.
