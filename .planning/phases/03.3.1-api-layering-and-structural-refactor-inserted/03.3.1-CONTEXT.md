# Phase 03.3.1: API Layering and Structural Refactor - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish consistent routes → services → queries separation across all API routes. Adopt Hono's built-in validation and error handling primitives to eliminate repeated boilerplate. No new features, no schema changes, no frontend work. The API surface (endpoints, request/response shapes) does not change.

</domain>

<decisions>
## Implementation Decisions

### Hono Typed Context
- **D-01:** Add `type AppEnv = { Variables: { orgId: string } }` in a new `apps/api/src/types.ts` file. All routers and the root `app` use `new Hono<AppEnv>()`. Handlers read `c.get('orgId')` with full TypeScript inference — no cast needed.

### zValidator Error Shape
- **D-02:** Create `appValidator(target, schema)` factory wrapper in `apps/api/src/lib/validator.ts`. It calls `zValidator` with a custom hook that enforces the existing error shape: `{error: {code: 'VALIDATION_ERROR', errors: result.error.issues}}` at `400`. All routes replace manual `safeParse` blocks with `appValidator`. The frontend's existing error handling is unaffected.

### Domain Error Classes
- **D-03:** Class-based errors in `apps/api/src/lib/errors.ts`:
  - `class DomainError extends Error { constructor(public statusCode: number, message: string) }`
  - `class NotFoundError extends DomainError { constructor(message: string) { super(404, message) } }`
- **D-04:** `app.onError()` registered in `apps/api/src/index.ts` maps `instanceof` checks to JSON:
  - `NotFoundError` → `{error: {code: 'NOT_FOUND', message}}` 404
  - `DomainError` → `{error: {code: 'DOMAIN_ERROR', message}}` with `err.statusCode`
  - Fallthrough → `{error: {code: 'INTERNAL_ERROR', message: 'Unexpected error'}}` 500
- **D-05:** All per-route try/catch blocks and uses of `badRequest()` helper are removed in favour of `throw new DomainError(...)` / `throw new NotFoundError(...)` from services. The existing `lib/helpers.ts` `badRequest` export is deleted.

### Service + Queries File Structure
- **D-06:** 2-file pattern for every route: `services/X.ts` (business logic) + `lib/queries/X.ts` (raw Drizzle queries). Applies to all routes currently missing a service layer: `accounts`, `categories`, `tags`, `merchant-rules`, `households`. Transactions already follows this pattern — no change needed there. Routes become thin: validate input with `appValidator`, call service, return `c.json(result)`.
- **D-07:** `services/webhooks.ts` extracts one handler function per Clerk event type (`handleOrgCreated`, `handleUserCreated`, `handleOrgUpdated`, `handleUserUpdated`, `handleOrgMembershipCreated`). Svix signature verification moves to a new `webhookAuth` middleware in `apps/api/src/middleware/webhookAuth.ts`.

### Clerk WebhookEvent Types
- **D-08:** Replace all `event.data as { id: string; ... }` manual casts with the `WebhookEvent` discriminated union from `@clerk/backend`. Event handlers in `services/webhooks.ts` narrow via `event.type` and receive correctly typed `event.data`.

### tenantGuard
- **D-09:** `tenantGuard` adds `c.set('orgId', orgId)` before calling `next()`. The existing `seenOrgs` in-memory upsert cache remains in place (deferred to Phase 8.2 per STATE.md). All protected routes replace `const { orgId } = getAuth(c)` with `const orgId = c.get('orgId')`.

### Claude's Discretion
- Exact Drizzle query functions to extract per route (grouping strategy within queries files)
- Whether `lib/queries/webhooks.ts` is warranted or webhooks DB calls stay inline in `services/webhooks.ts` (webhooks are write-only and simple enough for inline)
- File/function naming conventions within services and queries (follow existing `transactions` pattern)

</decisions>
