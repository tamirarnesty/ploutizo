## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing critical functionality] DomainError.code field for typed error response codes**
- **Found during:** Task 2
- **Issue:** `services/merchant-rules.ts` throws `DomainError(400, 'Invalid regular expression.')` but `index.ts` onError mapped all DomainErrors to code `'DOMAIN_ERROR'`. Test expects `'INVALID_REGEX'`. Without a machine-readable code field, either the test must do fragile message-string matching or production loses the specific error code that clients depend on.
- **Fix:** Added optional `code?: string` to `DomainError` constructor. `NotFoundError` passes `'NOT_FOUND'`. `validateRegex` in merchant-rules service passes `'INVALID_REGEX'`. `index.ts` onError uses `err.code ?? 'DOMAIN_ERROR'`.
- **Files modified:** lib/errors.ts, services/merchant-rules.ts, index.ts, __tests__/merchant-rules.test.ts
- **Commits:** bcf0597

**2. [Rule 2 - Missing critical functionality] merchant-rules.test.ts needs app.onError()**
- **Found during:** Task 1 (preparation)
- **Issue:** After route refactor, thin routes have no try/catch — DomainError propagates to onError. Test app lacked onError, so invalid-regex tests would get 500 instead of 400.
- **Fix:** Added `app.onError()` to test app that mirrors production `index.ts` handler using `err.code`.
- **Files modified:** __tests__/merchant-rules.test.ts
- **Commits:** 1ba1f0d, bcf0597
