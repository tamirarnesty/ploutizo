### Locked Decisions

**D-01 — AppEnv typed context:** `type AppEnv = { Variables: { orgId: string } }` in `apps/api/src/types.ts`. All routers and root `app` use `new Hono<AppEnv>()`. Handlers read `c.get('orgId')`.

**D-03 — Domain error classes** in `apps/api/src/lib/errors.ts`:
- `class DomainError extends Error { constructor(public statusCode: number, message: string) }`
- `class NotFoundError extends DomainError { constructor(message: string) { super(404, message) } }`

**D-04 — app.onError()** in `apps/api/src/index.ts`:
- `NotFoundError` → `{error: {code: 'NOT_FOUND', message}}` 404
- `DomainError` → `{error: {code: 'DOMAIN_ERROR', message}}` with `err.statusCode`
- Fallthrough → `{error: {code: 'INTERNAL_ERROR', message: 'Unexpected error'}}` 500

**D-05 — Remove try/catch and badRequest:** All per-route try/catch blocks and `badRequest()` uses removed. Services throw `DomainError` / `NotFoundError`. `lib/helpers.ts` deleted (no other consumers confirmed).

**D-07 — Webhook extraction:** `services/webhooks.ts` with one handler per event type. `webhookAuth` middleware in `apps/api/src/middleware/webhookAuth.ts` extracts Svix verification.

**D-08 — WebhookEvent type:** Replace manual `event.data as { ... }` casts with `WebhookEvent` discriminated union from `@clerk/backend`. Handlers narrow via `event.type`.

**D-09 — tenantGuard c.set:** `tenantGuard` adds `c.set('orgId', orgId)` before `next()`. `seenOrgs` in-memory cache remains. All protected routes replace `const { orgId } = getAuth(c)` with `const orgId = c.get('orgId')`.

### Pattern 1: AppEnv Typed Context
```typescript
// apps/api/src/types.ts
export type AppEnv = {
  Variables: {
    orgId: string
  }
}
```
Applied to every `new Hono<AppEnv>()` — root app and each sub-router. Hono propagates the env type when routers are mounted with `app.route()`. [VERIFIED: Hono 4.x context.d.ts — Variables key typed on c.get/c.set]

### Pattern 2: appValidator Factory
```typescript
// apps/api/src/lib/validator.ts
import { zValidator } from '@hono/zod-validator'
import type { ZodSchema } from 'zod'
import type { ValidationTargets } from 'hono'

export const appValidator = <T extends ZodSchema>(
  target: keyof ValidationTargets,
  schema: T
) =>
  zValidator(target, schema, (result, c) => {
    if (!result.success) {
      return c.json(
        { error: { code: 'VALIDATION_ERROR', errors: result.error.issues } },
        400
      )
    }
  })
```
Usage in routes: `appValidator('json', createAccountSchema)` as middleware; body accessed via `c.req.valid('json')`. [VERIFIED: hono/dist/types — ValidationTargets exists; zValidator hook signature matches this pattern]

### Pattern 3: onError Handler
```typescript
// apps/api/src/index.ts — after route registration, before serve()
app.onError((err, c) => {
  if (err instanceof NotFoundError) {
    return c.json({ error: { code: 'NOT_FOUND', message: err.message } }, 404)
  }
  if (err instanceof DomainError) {
    return c.json({ error: { code: 'DOMAIN_ERROR', message: err.message } }, err.statusCode as StatusCode)
  }
  return c.json({ error: { code: 'INTERNAL_ERROR', message: 'Unexpected error' } }, 500)
})
```
[VERIFIED: Hono 4.x — `app.onError((err, c) => Response)` is the documented error handler signature]

### Pattern 4: webhookAuth Middleware
```typescript
// apps/api/src/middleware/webhookAuth.ts
import { createMiddleware } from 'hono/factory'
import { Webhook } from 'svix'
import type { WebhookEvent } from '@clerk/backend'

export const webhookAuth = () =>
  createMiddleware<{ Variables: { webhookEvent: WebhookEvent } }>(async (c, next) => {
    const webhookSecret = process.env.CLERK_WEBHOOK_SECRET
    if (!webhookSecret) {
      return c.json({ error: { code: 'CONFIG_ERROR', message: 'Missing webhook secret.' } }, 500)
    }
    const svix = new Webhook(webhookSecret)
    const payload = await c.req.text()
    const headers = {
      'svix-id': c.req.header('svix-id') ?? '',
      'svix-timestamp': c.req.header('svix-timestamp') ?? '',
      'svix-signature': c.req.header('svix-signature') ?? '',
    }
    let event: WebhookEvent
    try {
      event = svix.verify(payload, headers) as WebhookEvent
    } catch {
      return c.json({ error: { code: 'INVALID_SIGNATURE', message: 'Webhook signature verification failed.' } }, 400)
    }
    c.set('webhookEvent', event)
    await next()
  })
```
**Note:** `webhookAuth` uses its own local `Variables` type (not AppEnv) because it sets `webhookEvent`, not `orgId`. The webhooks router uses `new Hono()` (no AppEnv needed — not tenant-guarded). [ASSUMED — Hono sub-router env type merging behavior; however the pattern of per-middleware Variables is standard Hono]

### Anti-Patterns to Avoid

- **Returning errors from services:** Services throw domain errors; they do not return error-shaped objects. Returning `null` for "not found" (as `transactions.ts` does) is also acceptable — the service can use either pattern, but the route must not inline the 404 response directly; use `throw new NotFoundError(...)` from service instead.
- **Using `getAuth(c)` in route handlers after tenantGuard is patched:** All protected routes must read `c.get('orgId')` — `getAuth(c)` remains valid only in the tenantGuard middleware itself.
- **Applying AppEnv to webhooksRouter:** The webhooks route is outside `/api/*` tenant guard scope. Its router doesn't need `AppEnv`.
- **Manually setting `updatedAt` in query functions that already do it:** The existing queries set `updatedAt: new Date()` inline — new query files must follow the same convention.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Request validation with consistent error shape | Custom safeParse + error response block | `appValidator` factory wrapping `zValidator` | @hono/zod-validator handles type inference, hook-based customisation; 6 routes × N handlers = many duplication sites |
| Webhook signature verification | Inline Svix code per route | `webhookAuth` middleware | Single point of change; Svix library handles edge cases (timing, replay attacks) |
| HTTP error mapping | Per-route try/catch mapping errors to status codes | `app.onError()` with instanceof checks | Centralised; impossible to forget a mapping |

## Common Pitfalls

### Pitfall 1: Router AppEnv propagation
**What goes wrong:** Sub-routers declared as `new Hono()` (no type param) don't infer `c.get('orgId')` type correctly — TypeScript returns `unknown`.
**Why it happens:** Hono generic env only flows through when the router is declared with the same `AppEnv`.
**How to avoid:** Declare every sub-router as `new Hono<AppEnv>()` except webhooksRouter (which is not tenant-guarded and doesn't need orgId).
**Warning signs:** TypeScript error on `c.get('orgId')` usage — "Argument of type 'string' is not assignable to..."

### Pitfall 2: appValidator and c.req.valid() type inference
**What goes wrong:** `c.req.valid('json')` returns `unknown` if the router's type param doesn't include the validated schema.
**Why it happens:** `zValidator` injects the schema type into the route's `Input` type parameter via Hono's middleware chain inference.
**How to avoid:** The appValidator middleware must be in the same `.post(path, appValidator(...), handler)` call chain, not pre-applied separately.
**Warning signs:** `c.req.valid('json')` returns `unknown` at callsite.

### Pitfall 3: Svix payload must be raw text
**What goes wrong:** Reading `c.req.json()` before passing to `svix.verify()` causes signature failure.
**Why it happens:** Svix verifies the raw byte payload against the signature — JSON parsing changes whitespace.
**How to avoid:** `webhookAuth` middleware uses `c.req.text()` (already done in existing webhooks.ts). Do not add a `c.req.json()` call anywhere before or after in the middleware chain for this route.
**Warning signs:** Signature verification always fails in production but passes in local testing (if local doesn't enforce signatures).

### Pitfall 4: app.onError placement
**What goes wrong:** `app.onError()` registered before routes doesn't catch errors from those routes.
**Why it happens:** Hono processes middleware/handlers in registration order; `onError` must be registered after routes to capture their errors.
**How to avoid:** Register `app.onError(...)` after all `app.route(...)` calls, before `serve(...)`. [VERIFIED: existing index.ts structure — serve() is the last call]
**Warning signs:** Errors thrown from route handlers bubble uncaught to Node.js process.

### Pitfall 5: Deleting helpers.ts without checking all consumers
**What goes wrong:** If `badRequest` is imported anywhere besides `routes/transactions.ts`, deleting `helpers.ts` causes a build error.
**Why it happens:** Grep for usage may miss files.
**How to avoid:** Grep for `from.*helpers` across `apps/api/src/` before deleting. [VERIFIED: only consumers are `routes/transactions.ts` — confirmed by reading all route files]
**Warning signs:** TypeScript build error on `import { badRequest }`.

### Pitfall 6: households.ts DELETE /members/:memberId requires both orgId and userId
**What goes wrong:** After tenantGuard only sets `orgId`, the caller's Clerk user ID is still needed for the self-removal guard. If the route is converted to read only `c.get('orgId')`, the guard breaks.
**Why it happens:** tenantGuard does not propagate `userId`.
**How to avoid:** Route handler reads `const { userId: callerClerkId } = getAuth(c)` in addition to `c.get('orgId')`. Pass both to service. `getAuth(c)` remains callable in route handlers — tenantGuard doesn't remove it.
**Warning signs:** Self-removal guard always passes or always fails.

## Open Questions

1. **Should `NotFoundError` be thrown from services, or should services return `null` (current transactions pattern)?**
   - What we know: Existing `transactions.ts` service returns `null` for not-found cases; routes handle the 404 inline. D-05 says "throw NotFoundError from services."
   - What's unclear: Transactions service already uses null returns. Changing all its null returns to throws is technically in scope of D-05 but would modify the existing canonical service.
   - Recommendation: Adopt throws in NEW services (`accounts`, `categories`, etc.). Leave `transactions.ts` service unchanged (null returns). The transactions route still has inline 404 checks — those stay. This limits scope creep.

2. **TypeScript satisfaction for `err.statusCode as StatusCode` in onError**
   - What we know: Hono's `StatusCode` type is a union of valid HTTP status integers.
   - What's unclear: Whether `err.statusCode as StatusCode` will satisfy TypeScript's `c.json()` second argument without explicit cast.
   - Recommendation: Import `StatusCode` from `hono/utils/http-status` and cast — `c.json(..., err.statusCode as StatusCode)`.
