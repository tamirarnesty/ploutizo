## Summary

Phase 3.1 is a pure schema phase: write Drizzle table definitions, generate the migration SQL, and add Zod discriminated unions to the validators package. No API or UI work is involved.

The codebase already has two migrations (`0000`, `0001`) and a well-established schema authoring convention visible in `accounts.ts` and `classification.ts`. The transactions schema follows that convention exactly: `uuid` PKs with `gen_random_uuid()`, `text` org_id FK, `{ withTimezone: true }` on every timestamp, named indexes. Three new schema files are needed: `transactions.ts`, `import-batches.ts`, and corresponding entries in the schema barrel.

The two technically interesting elements are (1) a self-referential FK on `transactions.refund_of` — handled in Drizzle by passing a lazy arrow function `() => transactions.id` to `.references()`, the same pattern Drizzle uses for all circular FKs, and (2) a partial index on `deleted_at IS NULL` — supported directly in the installed Drizzle 0.45.2 via `index().where(sql\`...\`)`.

**Primary recommendation:** Write `packages/db/src/schema/transactions.ts` and `packages/db/src/schema/import-batches.ts` following the `accounts.ts` pattern exactly, then add Zod discriminated unions to `packages/validators/src/index.ts` in the established naming convention (PascalCase form schemas, camelCase API schemas).

### Pattern 1: UUID Primary Key with gen_random_uuid()
**What:** Every table uses `uuid('id').primaryKey().default(sql\`gen_random_uuid()\`)`.
**When to use:** All tables in this project — established convention.
**Example:**
```typescript
// Source: packages/db/src/schema/accounts.ts (VERIFIED)
import { sql } from 'drizzle-orm'
import { uuid } from 'drizzle-orm/pg-core'

id: uuid('id').primaryKey().default(sql`gen_random_uuid()`),
```

### Pattern 2: Org-scoped Foreign Key
**What:** `org_id` is always `text NOT NULL` referencing `orgs.id` with cascade delete.
**When to use:** Every org-scoped table.
**Example:**
```typescript
// Source: packages/db/src/schema/accounts.ts (VERIFIED)
import { orgs } from './auth.js'

orgId: text('org_id')
  .notNull()
  .references(() => orgs.id, { onDelete: 'cascade' }),
```

### Pattern 3: Partial Index for Soft-Delete
**What:** An index filtered to active (non-deleted) rows only using `.where()`.
**When to use:** `transactions.deleted_at IS NULL` — all active-data queries use this.
**Example:**
```typescript
// Source: drizzle-orm 0.45.2 — index().where() confirmed in indexes.d.ts (VERIFIED)
import { sql } from 'drizzle-orm'
import { index } from 'drizzle-orm/pg-core'

// In the table's indexes array:
index('transactions_active_idx').on(t.deletedAt).where(sql`deleted_at IS NULL`)
```
**Note:** The partial index is on `deleted_at` but the `WHERE` condition filters to `IS NULL`. Drizzle generates the correct `CREATE INDEX ... WHERE deleted_at IS NULL` SQL.

### Pattern 4: Self-Referential Foreign Key (refund_of)
**What:** A nullable FK column that points back to the same table. Drizzle requires a lazy arrow function to break the circular reference.
**When to use:** `transactions.refund_of → transactions.id`.
**Example:**
```typescript
// Source: Drizzle pattern — columns/common.d.ts references() accepts () => Column (VERIFIED)
// The transactions table reference cannot be direct — use arrow function wrapper.
refundOf: uuid('refund_of').references((): AnyPgColumn => transactions.id, {
  onDelete: 'set null',
}),
```
The `AnyPgColumn` import comes from `drizzle-orm/pg-core`. The TypeScript cast is required to satisfy Drizzle's type checker when the column references the table being defined.

### Pattern 5: Named Index Convention
**What:** All index names follow `{tablename}_{column(s)}_idx` pattern.
**When to use:** Every index — existing convention.
**Example:**
```typescript
// Source: packages/db/src/schema/accounts.ts (VERIFIED)
(t) => [index('accounts_org_idx').on(t.orgId)]
```

### Pattern 6: Composite Primary Key on Join Tables
**What:** `transaction_tags` uses `(transaction_id, tag_id)` as the composite PK (no surrogate UUID).
**When to use:** Simple join tables with no extra metadata columns.
**Example:**
```typescript
// Source: D-11 from CONTEXT.md — no surrogate PK needed on this join table
import { primaryKey } from 'drizzle-orm/pg-core'

(t) => [primaryKey({ columns: [t.transactionId, t.tagId] })]
```

### Pattern 7: Zod Discriminated Union (v4 classic API)
**What:** `z.discriminatedUnion('type', [...branches])` — each branch is a `z.object` with a literal `type` field.
**When to use:** Transaction create/update schemas — all 6 types.
**Example:**
```typescript
// Source: zod 4.3.6 classic API (VERIFIED: v4/classic/schemas.d.ts)
import { z } from 'zod'

const expenseTransactionSchema = z.object({
  type: z.literal('expense'),
  amount: z.number().int().positive(),
  date: z.string().date(),
  accountId: z.string().uuid(),
  description: z.string().optional(),
  categoryId: z.string().uuid().optional(),
  // ...
})

export const createTransactionSchema = z.discriminatedUnion('type', [
  expenseTransactionSchema,
  refundTransactionSchema,
  incomeTransactionSchema,
  transferTransactionSchema,
  settlementTransactionSchema,
  contributionTransactionSchema,
])
export type CreateTransactionInput = z.infer<typeof createTransactionSchema>
```

### Pattern 8: Enum Import (not re-definition)
**What:** Import all enums from `./enums.js` — never redefine them.
**When to use:** `transactionTypeEnum`, `incomeTypeEnum`, `investmentTypeEnum` already exist.
**Example:**
```typescript
// Source: packages/db/src/schema/enums.ts (VERIFIED)
import { transactionTypeEnum, incomeTypeEnum, investmentTypeEnum } from './enums.js'
```

### Anti-Patterns to Avoid
- **Defining enums inline in transactions.ts:** All enums are already in `enums.ts`. Duplicating them causes `pgEnum already registered` errors at migration time.
- **Direct circular reference:** `transactions.refund_of.references(() => transactions.id)` without the `AnyPgColumn` cast causes TypeScript errors because `transactions` is not yet fully typed at the point of the column definition.
- **Using `numeric` for balance math:** The `percentage` column returns as a `string` from Drizzle. The code comment must explicitly state "display cache only — never use in balance math."
- **Omitting `{ withTimezone: true }` on timestamps:** Every timestamp in this project uses it. A bare `timestamp()` is non-conforming.
- **Defining drizzle.config.ts inside packages/db:** The config lives at the repo root and points `schema` to `./packages/db/src/schema/index.ts`. Running `drizzle-kit` from within the package will fail.
- **Using `db:push` in the migration workflow:** This phase uses `db:generate` + `db:migrate` (the production-safe path). `db:push` bypasses the migration file and is only for exploratory development.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Partial index (soft-delete) | Raw SQL `CREATE INDEX` in a migration comment | `index().where(sql\`...\`)` in schema | Drizzle tracks it in snapshot; manual SQL gets out of sync |
| UUID generation | Application-level UUID | `uuid(...).default(sql\`gen_random_uuid()\`)` | DB-generated is safe for concurrent inserts with no round-trip |
| Self-referential FK | Application-level reference checking | `() => transactions.id` lazy ref | Drizzle generates correct `REFERENCES` SQL with circular dependency resolution |
| Zod union type narrowing | `if (data.type === 'expense')` manual branching | `z.discriminatedUnion` | Type-safe narrowing, exhaustiveness checking in switch statements |

## Common Pitfalls

### Pitfall 1: AnyPgColumn import missing for self-referential FK
**What goes wrong:** TypeScript error `'transactions' is referenced before it's initialized` or type mismatch on the lazy reference return type.
**Why it happens:** The `transactions` variable is being defined while Drizzle infers column types — the return type annotation is needed to break the cycle.
**How to avoid:** Import `AnyPgColumn` from `drizzle-orm/pg-core` and annotate the arrow function: `(): AnyPgColumn => transactions.id`.
**Warning signs:** TS error on the `.references()` call for `refundOf`.

### Pitfall 2: drizzle-kit run from wrong directory
**What goes wrong:** `drizzle-kit generate` reports "No schema file found" or generates an empty migration.
**Why it happens:** `drizzle.config.ts` is at the repo root; running `pnpm --filter @ploutizo/db db:generate` inside the package won't find it. The root `pnpm db:generate` script calls `drizzle-kit generate` with the root config.
**How to avoid:** Always run from repo root: `pnpm db:generate` and `pnpm db:migrate`.
**Warning signs:** Empty migration file; no new SQL statements generated.

### Pitfall 3: Partial index not appearing in generated SQL
**What goes wrong:** The generated migration has `CREATE INDEX transactions_active_idx ON transactions(deleted_at)` but no `WHERE` clause.
**Why it happens:** Passing the condition as a string instead of `sql\`...\`` template literal, or using `isNull(t.deletedAt)` (Drizzle expression) instead of raw SQL.
**How to avoid:** Use `index(...).on(t.deletedAt).where(sql\`deleted_at IS NULL\`)` exactly.
**Warning signs:** No `WHERE` clause in the generated `.sql` file for this index.

### Pitfall 4: import_batches must be defined before transactions (FK dependency order)
**What goes wrong:** `transactions.import_batch_id` references `import_batches.id`. If `transactions.ts` imports from `import-batches.ts` but `import-batches.ts` imports from `transactions.ts`, there is a circular import at module level.
**Why it happens:** `transactions.ts` has a one-way FK to `import_batches`. `import_batches` has no FK back to `transactions`.
**How to avoid:** `transactions.ts` imports `importBatches` from `./import-batches.js`. `import-batches.ts` does NOT import from `transactions.ts`. One-directional dependency, no cycle.
**Warning signs:** Module circular dependency error at runtime or TypeScript import cycle warning.

### Pitfall 5: Zod v4 `z.string().date()` vs `z.string().datetime()`
**What goes wrong:** Using `.datetime()` for the `date` field accepts ISO datetime strings (`2024-01-01T00:00:00Z`) but the DB column is a `date` type (date only). Downstream comparisons break.
**Why it happens:** Confusion between ISO date (`YYYY-MM-DD`) and ISO datetime.
**How to avoid:** Use `z.string().date()` (validates `YYYY-MM-DD` format). [VERIFIED: Zod v4 classic API has `.date()` on string schemas]
**Warning signs:** API accepts `2024-01-01T00:00:00Z` for a field that should be `2024-01-01` only.

### Pitfall 6: numeric(6,3) returns as string from Drizzle
**What goes wrong:** Code attempts arithmetic on `assignee.percentage` directly — TypeScript type is `string`.
**Why it happens:** PostgreSQL `numeric` is arbitrary precision; Drizzle maps it to `string` to avoid floating-point loss.
**How to avoid:** The `percentage` column is a display cache and must never be used in math. Add a JSDoc comment on the column explicitly stating this.
**Warning signs:** TS error `Operator '+' cannot be applied to type 'string' and 'string'` on percentage.

### Pitfall 7: Schema barrel not updated
**What goes wrong:** `@ploutizo/db` consumers (apps/api, Phase 3.2) import `transactions` from `@ploutizo/db` and get `undefined` or a TS error.
**Why it happens:** New schema files are not auto-discovered — `packages/db/src/schema/index.ts` must be updated manually.
**How to avoid:** After creating schema files, add `export * from './import-batches.js'` and `export * from './transactions.js'` to `index.ts`.
**Warning signs:** `import { transactions } from '@ploutizo/db'` — TypeScript says `transactions` does not exist.

### transactions table skeleton
```typescript
// packages/db/src/schema/transactions.ts
import { sql, type AnyColumn } from 'drizzle-orm'
import {
  index,
  integer,
  numeric,
  pgTable,
  primaryKey,
  text,
  timestamp,
  uniqueIndex,
  uuid,
} from 'drizzle-orm/pg-core'
import type { AnyPgColumn } from 'drizzle-orm/pg-core'

import { transactionTypeEnum, incomeTypeEnum, investmentTypeEnum } from './enums.js'
import { orgs } from './auth.js'
import { orgMembers } from './auth.js'
import { accounts } from './accounts.js'
import { categories, tags } from './classification.js'
import { importBatches } from './import-batches.js'

export const transactions = pgTable(
  'transactions',
  {
    id: uuid('id').primaryKey().default(sql`gen_random_uuid()`),
    orgId: text('org_id').notNull().references(() => orgs.id, { onDelete: 'cascade' }),
    accountId: uuid('account_id').notNull().references(() => accounts.id, { onDelete: 'restrict' }),
    type: transactionTypeEnum('type').notNull(),
    amount: integer('amount').notNull(),           // unsigned cents, D-02
    date: text('date').notNull(),                  // ISO date string YYYY-MM-DD
    description: text('description'),
    merchant: text('merchant'),

    // --- expense / refund columns ---
    categoryId: uuid('category_id').references(() => categories.id, { onDelete: 'set null' }),
    refundOf: uuid('refund_of').references((): AnyPgColumn => transactions.id, { onDelete: 'set null' }),

    // --- income columns ---
    incomeType: incomeTypeEnum('income_type'),
    incomeSource: text('income_source'),

    // --- transfer columns ---
    toAccountId: uuid('to_account_id').references(() => accounts.id, { onDelete: 'restrict' }),

    // --- settlement columns ---
    settledAccountId: uuid('settled_account_id').references(() => accounts.id, { onDelete: 'restrict' }),

    // --- contribution columns ---
    investmentType: investmentTypeEnum('investment_type'),

    // --- import / recurring (schema gate columns) ---
    importBatchId: uuid('import_batch_id').references(() => importBatches.id, { onDelete: 'set null' }),
    recurringTemplateId: uuid('recurring_template_id'), // no FK — generation logic deferred to v2, D-07

    // --- soft-delete ---
    deletedAt: timestamp('deleted_at', { withTimezone: true }),

    createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
    updatedAt: timestamp('updated_at', { withTimezone: true }).notNull().defaultNow(),
  },
  (t) => [
    index('transactions_org_account_idx').on(t.orgId, t.accountId),        // D-14
    index('transactions_org_date_idx').on(t.orgId, t.date),                // D-15
    index('transactions_active_idx').on(t.deletedAt).where(sql`deleted_at IS NULL`), // D-16
    index('transactions_org_idx').on(t.orgId),
  ]
)
```

## Open Questions

1. **`ON DELETE RESTRICT` vs `ON DELETE SET NULL` on `transactions.account_id`**
   - What we know: Account deletion with existing transactions is an edge case. The accounts table uses `archived_at` soft-delete.
   - What's unclear: The CONTEXT.md does not specify the `onDelete` action for `transactions.account_id` and `transactions.to_account_id`.
   - Recommendation: Use `RESTRICT` for `account_id` (prevents deleting an account with transactions — safer) and `SET NULL` for `to_account_id` (nullable transfer target). Planner can override.

2. **`transactions.date` column type: `text` vs `date`**
   - What we know: The existing codebase uses `text` for the `date` ISO string. REQUIREMENTS.md specifies `date` field as ISO date string. Drizzle has a `date()` column type.
   - What's unclear: Whether to use Drizzle's `date()` column (maps to `date` in Postgres) vs `text` with application-level validation.
   - Recommendation: Use Drizzle `date('date')` column type — Postgres will enforce format, and Drizzle returns it as a string by default. This is more correct than `text` for a date field and enables native Postgres date comparisons.
