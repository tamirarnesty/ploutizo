## Assignee Split Storage

| Option | Description | Selected |
|--------|-------------|----------|
| amount_cents NOT NULL + percentage nullable | amount_cents is authoritative; percentage is a nullable display cache. Balance queries use SUM(amount_cents) only. | ✓ |
| amount_cents only | Skip percentage entirely — always derivable from amount_cents / transaction.amount. | |
| Both NOT NULL | Both columns always written. Enforces consistency at DB level but requires computing percentage at import time too. | |

**User's choice:** `amount_cents NOT NULL` + `percentage numeric(6,3)` nullable
**Notes:** User asked about optionality of each field before deciding. Clarified that amount_cents is always required (authoritative), percentage can be skipped during import flows.

## Zod Discriminated Union Depth

| Option | Description | Selected |
|--------|-------------|----------|
| Field presence only | Required fields per type, no value constraints. API handles amount > 0, date format, etc. | |
| Basic value constraints too | Required fields + amount.int().positive(), date.date(). Cross-entity rules (refund_of ownership) stay in API only. | ✓ |

**User's choice:** Basic value constraints included in shared validators
**Notes:** User asked for explanation of the difference. After explanation, chose to include basic constraints (amount > 0, valid date) in the shared Zod schemas since they're always true and benefit both web forms and API without adding maintenance burden.
