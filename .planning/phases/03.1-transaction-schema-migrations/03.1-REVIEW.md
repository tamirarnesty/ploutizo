# Phase 03.1: Code Review Report

**Reviewed:** 2026-04-09
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Reviewed the transaction schema migration, Drizzle table definitions, and Zod validators introduced in phase 03.1. The overall structure is clean and well-documented. Three correctness issues were found: `updated_at` will silently remain stale on row updates (no trigger or ORM hook), the `amount` column has no database-level CHECK constraint enforcing the "unsigned cents" invariant documented in the schema, and the `updateTransactionSchema` allows cross-type field pollution. Two informational items cover a misleading error message and a fragile raw-SQL string in a partial index.

## Warnings

### WR-01: `updated_at` not auto-updated on row edits

**File:** `packages/db/src/schema/transactions.ts:94`
**Issue:** `updatedAt` is defined with `.defaultNow()`, which only fires on `INSERT`. Drizzle ORM has no automatic `ON UPDATE` behavior for timestamp columns. Any `UPDATE` issued through Drizzle (or raw SQL) will leave `updated_at` frozen at the original insert time, silently producing stale values and making audit/cache-invalidation logic unreliable.
**Fix:** Add a PostgreSQL trigger to keep `updated_at` current, and mirror it in a subsequent Drizzle migration:

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_transactions_updated_at
BEFORE UPDATE ON transactions
FOR EACH ROW EXECUTE PROCEDURE set_updated_at();
```

Alternatively, set `updated_at` explicitly in every Drizzle update call:
```ts
await db.update(transactions)
  .set({ ...patch, updatedAt: new Date() })
  .where(eq(transactions.id, id))
```

The trigger approach is safer because it is enforced at the database layer regardless of the caller.

### WR-02: `amount` column has no CHECK constraint — "unsigned" invariant is not enforced

**File:** `packages/db/drizzle/0002_easy_stellaris.sql:29` / `packages/db/src/schema/transactions.ts:47`
**Issue:** The schema documentation states "amount is unsigned cents (D-02). Direction is encoded in type, never in sign." However, the DB column is a plain `integer` with no `CHECK (amount > 0)` constraint. A bug in the API layer (or a direct SQL statement) can insert a negative or zero amount without any database-level rejection. Because balance calculations rely on `amount` being strictly positive, a corrupted row would silently produce wrong totals.
**Fix:** Add a CHECK constraint in the schema and migration:

In `transactions.ts`, add a check inside the table config callback:
```ts
import { check } from 'drizzle-orm/pg-core'
import { sql } from 'drizzle-orm'

// inside the (t) => [...] array:
check('transactions_amount_positive', sql`${t.amount} > 0`),
```

And in the migration SQL:
```sql
ALTER TABLE "transactions"
  ADD CONSTRAINT "transactions_amount_positive" CHECK (amount > 0);
```

### WR-03: `updateTransactionSchema` allows cross-type field pollution

**File:** `packages/validators/src/index.ts:179-190`
**Issue:** `updateTransactionSchema` is a flat partial object that includes all type-specific fields (`incomeType`, `toAccountId`, `settledAccountId`, `investmentType`, etc.) with no discriminated-union enforcement. This means a caller can submit an update payload that sets `toAccountId` on an `income` transaction, or `incomeType` on a `transfer`, and the schema will accept it silently. The API layer must guard against this independently — but there is nothing in the schema to catch or document the constraint.
**Fix:** Two options:

Option A — leave as-is but document the gap and add API-layer enforcement explicitly (least change, acceptable for v1 if the API handler validates type consistency before applying updates).

Option B — introduce a discriminated-union update schema that carries the transaction `type` (read from the existing row and injected server-side), though this adds complexity and requires the handler to fetch the current type first.

At minimum, add a comment to the schema to prevent future contributors from trusting it as the sole gate:
```ts
/**
 * Flat partial update schema — all type-specific fields are included.
 * Cross-type field constraints (e.g. toAccountId only valid for transfer) are
 * NOT enforced here; callers MUST validate type compatibility at the API layer.
 */
export const updateTransactionSchema = baseTransactionSchema
  .partial()
  // ...
```

### IN-01: Misleading error message in `HouseholdSettingsFormSchema`

**File:** `packages/validators/src/index.ts:100-103`
**Issue:** The `refine` predicate is `parseFloat(v) >= 0` (non-negative), but the error message says `'Must be a positive number.'` Zero is accepted but labelled as "not positive." The test at line 119 confirms zero passes. The message does not match the actual validation rule.
**Fix:** Align the message with the actual rule:
```ts
.refine((v) => !isNaN(parseFloat(v)) && parseFloat(v) >= 0, {
  message: 'Must be a non-negative number.',
})
```
Or, if zero should truly be disallowed, tighten the predicate to `> 0` and the message is already correct.

### IN-02: Partial index WHERE clause uses raw SQL string — column rename not caught by TypeScript

**File:** `packages/db/src/schema/transactions.ts:99`
**Issue:** The partial index for active transactions uses `sql\`deleted_at IS NULL\`` — a raw SQL string referencing the database column name directly. Drizzle cannot type-check this string. If the `deletedAt` column were ever renamed in the schema, the partial index definition would silently remain pointing to the old column name and the index would become invalid or be dropped on next schema push.
**Fix:** This is a known Drizzle limitation — partial index `WHERE` clauses must be raw SQL. Add an inline comment noting the column dependency so it is visible during refactors:
```ts
// NOTE: raw SQL — must match column name 'deleted_at' if renamed
index('transactions_active_idx').on(t.deletedAt).where(sql`deleted_at IS NULL`),
```
