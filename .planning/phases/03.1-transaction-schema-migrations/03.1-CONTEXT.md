# Phase 3.1: Transaction Schema & Migrations - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Creates the database schema foundation for all 6 transaction types. Delivers the
`transactions` table, `transaction_assignees` join table, `transaction_tags` join
table, and `import_batches` table, plus all required indexes and Drizzle migrations.
No API or UI work — this is the schema gate that Phases 3.2–3.4 and 5.x depend on.

Also delivers Zod discriminated unions for all 6 transaction types in `@ploutizo/validators`.

</domain>

<decisions>
## Implementation Decisions

### `transactions` table columns
- **D-01:** Single flexible `transactions` table with nullable type-specific columns (locked from PROJECT.md — not revisited).
- **D-02:** `amount` column is `integer NOT NULL` (unsigned cents). Direction implied by type, never by sign.
- **D-03:** `org_id` is `text NOT NULL REFERENCES orgs(id) ON DELETE CASCADE` — same pattern as all existing tables (Clerk org ID is a text string, not UUID).
- **D-04:** `deleted_at timestamptz` for soft-delete. All active-data queries use partial index `WHERE deleted_at IS NULL`.
- **D-05:** Type-specific nullable column groups:
  - expense/refund: `category_id uuid`, `refund_of uuid` (FK → transactions.id, self-referential)
  - income: `income_type income_type_enum`, `income_source text`
  - transfer: `to_account_id uuid`
  - settlement: `settled_account_id uuid` (source account for CSV dedup)
  - contribution: `investment_type investment_type_enum`
- **D-06:** `import_batch_id uuid` nullable FK → `import_batches.id` on transactions (schema gate for Phase 5).
- **D-07:** `recurring_template_id uuid` reserved column, nullable, no FK — generation logic deferred to v2.

### `transaction_assignees` table
- **D-08:** `amount_cents integer NOT NULL` — authoritative split share per assignee, computed via Largest Remainder Method at write time.
- **D-09:** `percentage numeric(6,3)` nullable — display cache only, never used in balance computations. May be omitted during import flows.
- **D-10:** `UNIQUE(transaction_id, member_id)` constraint — prevents duplicate assignee rows.

### `transaction_tags` join table
- **D-11:** Include `transaction_tags` in Phase 3.1 (complete the schema in one migration). Simple join table: `(transaction_id, tag_id)` composite primary key.

### `import_batches` table
- **D-12:** Include `import_batches` table in Phase 3.1. Columns: `id uuid PK`, `org_id text NOT NULL`, `source text NOT NULL` (bank identifier or 'ploutizo'), `imported_at timestamptz NOT NULL`, `row_count integer NOT NULL`, `created_at timestamptz NOT NULL`.
- **D-13:** `transactions.import_batch_id` is a nullable FK → `import_batches.id`. Manually-created transactions will have NULL here.

### Indexes
- **D-14:** `(org_id, account_id)` composite index on `transactions` — required for balance/settlement queries.
- **D-15:** `(org_id, date)` composite index on `transactions` — for date-range filters in list view and budget period queries.
- **D-16:** Partial index `WHERE deleted_at IS NULL` on `transactions` — all active-data queries use this.

### Zod discriminated unions
- **D-17:** Discriminated unions keyed on `type` covering all 6 types: `expense`, `refund`, `income`, `transfer`, `settlement`, `contribution`. Each branch enforces its required fields at the type level.
- **D-18:** Validators include basic value constraints that are always true regardless of context: `amount.int().positive()`, `date.date()` (ISO date string). Cross-entity rules (e.g. refund_of must belong to same org) stay in the API layer only.
- **D-19:** Validators live in `@ploutizo/validators` (shared package) — both `apps/web` forms and `apps/api` handlers import from here.

### Claude's Discretion
- Exact Drizzle column definition style (`.references()` vs inline FK naming)
- Whether `investment_type` belongs in `transactions` or a separate `contributions` table (single-table design is locked — this is just column naming)
- SUM aggregate cast pattern: `sql\`SUM(${transactions.amount})::bigint\`` — established in REQUIREMENTS.md, planner decides where to apply

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/db/src/schema/enums.ts` — `transactionTypeEnum`, `incomeTypeEnum`, `investmentTypeEnum` already defined. Phase 3.1 only needs to import them.
- `packages/db/src/schema/accounts.ts` — Pattern to follow for new schema files: uuid PK with `gen_random_uuid()`, text org_id FK, `withTimezone: true` on all timestamps, named indexes.
- `packages/validators/src/index.ts` — Established pattern: plain Zod objects + `z.infer<>` type exports. Add transaction discriminated union alongside existing schemas.

### Established Patterns
- All tables: `org_id text NOT NULL REFERENCES orgs(id) ON DELETE CASCADE` — no nullable org rows.
- UUID primary keys: `uuid('id').primaryKey().default(sql\`gen_random_uuid()\`)`.
- Timestamps: always `{ withTimezone: true }`.
- Schema barrel: `packages/db/src/schema/index.ts` — add new exports here after creating schema files.
- Migrations: `pnpm --filter @ploutizo/db db:generate` then `pnpm --filter @ploutizo/db db:migrate`.

### Integration Points
- Phase 3.2 (Transaction API) will import `transactions`, `transactionAssignees`, `transactionTags`, `importBatches` from `@ploutizo/db`.
- Phase 3.2 will import transaction Zod schemas from `@ploutizo/validators`.
- Phase 5.x (Import) will write to `import_batches` and set `transactions.import_batch_id`.
- `seedOrg(orgId)` in Phase 1 webhook handler does NOT need changes — no transaction seed data.

</code_context>

<specifics>
## Specific Ideas

- `org_id` on `transactions` is `text` (not uuid) — matches Clerk org ID format (`org_2abc...`), consistent with every other org-scoped table in the codebase.
- `transaction_assignees.percentage` is explicitly a display cache — the planner must document this in code comments so future engineers don't accidentally use it in balance math.
- `refund_of` is a self-referential nullable FK on `transactions` — Drizzle handles this with a deferred/lazy reference.
