# Plan 03.4.2-01 — Transaction Form Polish & UI Bug Fixes

## Objective

Fix four targeted bugs surfaced after phase 03.4.1: (1) edit-mode amount change doesn't recalculate assignee splits, (2) Notes/Tags layout is cramped equal-width flex row, (3) Enter key on "Create" tag option does nothing, (4) category edits don't reflect in the list without a page reload.

## must_haves

- In edit mode, changing the Amount field causes assignee split amountCents values to recalculate via lrmSplit
- Notes field is rendered full-width above the Tags field (stacked, not side-by-side)
- Notes textarea has at least 3 rows minimum height
- Pressing Enter while the "Create" ComboboxItem is highlighted in TransactionTagPicker creates the tag
- After editing a category colour/name/icon, the updated category is visible in CategoriesSettings list immediately (without reload)

## Tasks

### Task 1: Fix edit-mode amount recalculation in AssigneeSection

<read_first>
- apps/web/src/components/transactions/AssigneeSection.tsx
- apps/web/src/lib/lrm.ts
</read_first>

<action>
In `AssigneeSection.tsx`, the `useEffect` that calls `onChange(lrmSplit(amountCents, pressedMemberIds))` guards with `!transaction`, preventing recalculation in edit mode.

**Replace the existing effect (lines ~63–69) with:**

```typescript
const isInitialRender = useRef(true)
useEffect(() => {
  if (isInitialRender.current) {
    isInitialRender.current = false
    return
  }
  if (amountCents > 0 && pressedMemberIds.length > 0) {
    onChange(lrmSplit(amountCents, pressedMemberIds))
  }
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [amountCents])
```

Add `useRef` to the React import — the current import is `{ useCallback, useEffect, useState }` and does NOT include `useRef`. Change it to `{ useCallback, useEffect, useRef, useState }`. Remove the now-unused `transaction` reference from this particular effect (the `transaction` prop is still used elsewhere for initialPressedIds and customizeOpen).

The `isInitialRender` guard skips the first render (preserving saved edit-mode splits on mount) and then fires on every subsequent `amountCents` change in both create and edit mode.
</action>

<acceptance_criteria>
- `AssigneeSection.tsx` contains `const isInitialRender = useRef(true)` (or equivalent ref name)
- The effect that calls `onChange(lrmSplit(amountCents, pressedMemberIds))` does NOT contain `!transaction`
- The effect still has `// eslint-disable-next-line react-hooks/exhaustive-deps` comment
- `useRef` is present in the React import destructure at the top of the file
</acceptance_criteria>

### Task 3: Fix Enter key for "Create" option in TransactionTagPicker

<read_first>
- apps/web/src/components/transactions/TransactionTagPicker.tsx
- apps/web/src/components/settings/CategoriesSettings.tsx (lines 60–75 for reference — CategoriesSettings uses `filteredItems={createOption ? [createOption] : []}` which makes the create option keyboard-selectable)
</read_first>

<action>
In `TransactionTagPicker.tsx`, the current `Combobox` props are:

```tsx
<Combobox
  multiple
  value={selectedNames}
  onValueChange={handleValueChange}
  onInputValueChange={setTagInputValue}
  items={allTagNames}
  filteredItems={filteredTagNames}
>
```

`filteredItems={filteredTagNames}` only contains real tag names — the `createOption` is excluded. The Base UI Combobox uses `filteredItems` for keyboard navigation, so the "Create" item is not reachable via Enter.

**Fix:** Include the create option in `filteredItems` when it exists:

```tsx
<Combobox
  multiple
  value={selectedNames}
  onValueChange={handleValueChange}
  onInputValueChange={setTagInputValue}
  items={allTagNames}
  filteredItems={createOption ? [...filteredTagNames, createOption] : filteredTagNames}
>
```

This matches how `CategoriesSettings.tsx` handles it (`filteredItems={createOption ? [createOption] : []}`), making the "Create" item keyboard-selectable so Enter triggers `handleValueChange` with the `__create__` value.
</action>

<acceptance_criteria>
- `TransactionTagPicker.tsx` `filteredItems` prop uses `createOption ? [...filteredTagNames, createOption] : filteredTagNames` (or equivalent expression that includes createOption when non-null)
- `TransactionTagPicker.tsx` does NOT have `filteredItems={filteredTagNames}` (bare, without create option)
</acceptance_criteria>

### Task 4: Fix category edit not reflecting in CategoriesSettings list

<read_first>
- apps/web/src/components/settings/CategoriesSettings.tsx (lines 60–90)
- apps/web/src/lib/data-access/categories/useUpdateCategory.ts
</read_first>

<action>
In `CategoriesSettings.tsx`, the `localCategories` sync effect (around line 75) gates on `!initialized.current`, which prevents re-syncing when the query refetches after a category mutation:

```typescript
const initialized = useRef<boolean>(false)
useEffect(() => {
  if (!initialized.current && !catLoading && categories.length > 0) {
    initialized.current = true
    setLocalCategories(categories)
  }
}, [categories, catLoading])
```

`useUpdateCategory.onSettled` calls `qc.invalidateQueries({ queryKey: ['categories'] })`, which refetches `categories`. But the effect sees `initialized.current === true` and does nothing, so `localCategories` stays stale.

**Fix:** Remove the `initialized` ref entirely and drop the guard:

```typescript
useEffect(() => {
  if (!catLoading && categories.length > 0) {
    setLocalCategories(categories)
  }
}, [categories, catLoading])
```

Also remove: `const initialized = useRef<boolean>(false)` and the `initialized.current = true` line.

The drag-to-reorder optimistic state (`handleReorder` sets `localCategories` directly then mutates) is safe: the reorder mutation's `invalidateQueries` fires after the reorder is committed to the server. When `categories` refetches, `setLocalCategories(categories)` runs again with the authoritative server order — which is what we want.
</action>

<acceptance_criteria>
- `CategoriesSettings.tsx` does NOT contain `initialized.current` or `const initialized = useRef`
- `CategoriesSettings.tsx` useEffect for syncing localCategories contains `if (!catLoading && categories.length > 0)` without an `initialized.current` guard
- `useRef` import is removed from CategoriesSettings.tsx if `initialized` was its only usage (check other useRef usages in the file first)
</acceptance_criteria>
