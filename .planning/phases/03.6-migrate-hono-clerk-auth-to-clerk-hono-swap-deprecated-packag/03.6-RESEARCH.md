## Primary sources (official)

1. **Published package & quickstart:** [`@clerk/hono` on npm](https://www.npmjs.com/package/@clerk/hono) — documents `import { clerkMiddleware, getAuth } from '@clerk/hono'`, env-based keys (`CLERK_SECRET_KEY`, `CLERK_PUBLISHABLE_KEY`), optional `getAuth(c, { acceptsToken: 'api_key' })`, and `verifyWebhook` from `@clerk/hono/webhooks`. **Prereqs:** Hono 4+, Node ≥ 20.9.0 (this repo requires Node ≥ 22 — satisfies).

2. **Implementation source (authoritative API surface):** [`packages/hono/src/clerkMiddleware.ts` in clerk/javascript](https://github.com/clerk/javascript/blob/main/packages/hono/src/clerkMiddleware.ts) — `clerkMiddleware(options?: ClerkMiddlewareOptions)` where options default `secretKey` / `publishableKey` (and optionally `machineSecretKey`, `apiUrl`, `apiVersion`) from the Hono **environment adapter** (`hono/adapter`'s `env(c)`) **or** from the explicit option object. Remaining spread (`...rest`) is forwarded to `createClerkClient` and `clerkClient.authenticateRequest(...)`. **Exports:** `ClerkMiddlewareOptions` from the same module.

3. **Shared middleware option semantics:** [Clerk — `clerkMiddleware()` options (reference)](https://clerk.com/docs/reference/nextjs/clerk-middleware) — documents cross-SDK concepts including:
   - `authorizedParties`: `string[]` — origin allowlist (subdomain cookie-leaking mitigation).
   - `clockSkewInMs`: `number` — allowed skew when validating JWTs (doc default cited as **5000** ms); our app uses **10000**.
   - `publishableKey` / `secretKey` — instance credentials.

> **Note:** The Clerk reference page titles “Next.js” but the **options block** aligns with Clerk’s shared `authenticateRequest`/middleware layering; `@clerk/hono` forwards the same-shaped options into `@clerk/backend`’s `authenticateRequest`.

## Mapping: current `@hono/clerk-auth` usage → `@clerk/hono`

| Location | Today | Target |
|----------|-------|--------|
| `apps/api/src/index.ts` | `import { clerkMiddleware } from '@hono/clerk-auth'` | `import { clerkMiddleware } from '@clerk/hono'` |
| `apps/api/src/middleware/tenantGuard.ts` | `import { getAuth } from '@hono/clerk-auth'` | `import { getAuth } from '@clerk/hono'` |
| `apps/api/src/routes/households.ts` | `import { getAuth } from '@hono/clerk-auth'` | `import { getAuth } from '@clerk/hono'` |
| Vitest mocks | `vi.mock('@hono/clerk-auth', …)` | `vi.mock('@clerk/hono', …)` |

**Middleware call site (`index.ts`):** Preserve the explicit object passed today:

```ts
clerkMiddleware({
  secretKey: process.env.CLERK_SECRET_KEY,
  publishableKey: process.env.CLERK_PUBLISHABLE_KEY,
  clockSkewInMs: 10000,
  authorizedParties,
});
```

Per `clerkMiddleware` implementation above, explicit `secretKey` / `publishableKey` override env defaults — behavior matches `@hono/clerk-auth`-style wiring. **`authorizedParties`** remains **`string[]`-shaped** in Clerk’s documented options (no callback form in Clerk’s middleware options docs). **`clockSkewInMs`** remains a supported skew parameter forwarded with `...rest`.

**`getAuth(c)`:** Hono readme shows `const { userId } = getAuth(c)` — our routes use `{ orgId }` / `{ userId }` from the Clerk auth object on the Hono context; **no Express-style `req` signature** — `getAuth` from `@clerk/hono` is the correct API.

## Deprecation rationale

[`@hono/clerk-auth` (JSR)](https://jsr.io/@hono/clerk-auth) remains the legacy Hono community middleware; migrating to **`@clerk/hono`** is Clerk’s supported path and aligns with CONTEXT D-01 / D-04.

---

## Pitfalls / watch-outs

1. **Mocks must track the module path.** All `vi.mock('@hono/clerk-auth'` and imports of `getAuth` from `@hono/clerk-auth` **must** change together or Vitest resolves the wrong factory.

2. **`c.get('clerk')`.** `@clerk/hono` attaches `clerk` client on context (`clerkMiddleware` sets `c.set('clerk', clerkClient)`). Existing codepaths that rely only on `getAuth(c)` remain valid; webhook route already uses Clerk verification separately — **confirm no hidden import** of `@hono/clerk-auth` outside grep results (execute-phase should grep after edits).

3. **Version coupling.** Published `@clerk/hono` (e.g. `0.1.x`) depends on **`@clerk/backend`** internally; workspace already uses `@clerk/backend` at `apps/api` devDependency for tests (`^3.x`). After install, **`pnpm turbo typecheck`** and **`pnpm --filter apps/api test`** validate peer compatibility.
