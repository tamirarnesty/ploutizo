## Summary

This is a dependency swap phase: `@hono/clerk-auth` → `@clerk/hono`. The diff was examined against the base commit (`7b6086d`) to scope findings to what this phase actually changed.

The migration is mechanically correct. All import paths are updated consistently across routes, middleware, and tests. Middleware order (CORS → `clerkMiddleware` → `tenantGuard`) is preserved. JWT auth options (`clockSkewInMs: 10000`, `authorizedParties`) pass through correctly to `VerifyJwtOptions` — both are valid fields on the chain `ClerkMiddlewareOptions → AuthenticateRequestOptions → VerifyTokenOptions → VerifyJwtOptions`. The new `households.ts` routes (`GET /invitations`, `DELETE /invitations/:invitationId`) are well-structured and follow existing patterns.

One warning was introduced: `as any` casts in `transactions.test.ts` were added where the pre-existing code had no cast. Three info items are noted below.

## Warnings

### WR-01: `as any` casts introduced in `transactions.test.ts` remove type safety from assertions

**File:** `apps/api/src/__tests__/transactions.test.ts:232,249,319,373,384`
**Issue:** This phase changed `await res.json()` to `await res.json() as any` at lines 232, 249, 263, 287, 307, 319, 373, 384. The pre-existing code (pre-diff) had no type annotation at those sites — meaning the result was implicitly typed. The added `as any` casts are strictly worse: they suppress any type errors that the `body.data.X` and `body.error.X` property accesses would otherwise catch.

For example:
```typescript
// Before (implicit unknown — at least triggers TS error if .data.id is accessed without type guard)
const body = await res.json();

// After (silences all errors — accessing nonexistent property is fine now)
const body = await res.json() as any;
```

The pattern used in every other test file in this PR (e.g., `households.test.ts`, `accounts.test.ts`) is to use explicit type assertions:
```typescript
const body = (await res.json()) as { data: { id: string; orgId: string } };
```

**Fix:** Replace `as any` with specific expected types matching the mock service return shapes. The `mockTxRow` object exported at the top of the file provides the exact shape:
```typescript
// TXN-POST-01
const body = (await res.json()) as { data: { id: string; orgId: string } };

// TXN-POST-02
const body = (await res.json()) as { error: { code: string } };

// TXN-GET-01
const body = (await res.json()) as { data: unknown[]; total: number; page: number; limit: number };

// TXN-GET-02
const body = (await res.json()) as { data: { assignees: unknown[]; tags: unknown[] } };

// TXN-GET-03
const body = (await res.json()) as { error: { code: string } };
```

## Info

### IN-01: Inconsistent optional chaining on `mock.calls[0]` in sort tests

**File:** `apps/api/src/__tests__/transactions.test.ts:333,341,349`
**Issue:** This phase changed three sort test assertions from `mock.calls[0]?.[0]` to `mock.calls[0][0]` (removing optional chaining). Lines 358 and 367 in the same file were not changed and still use `?.[0]`. The non-optional form is safe in these three tests because each guards with `expect(res.status).toBe(200)` before accessing call args — if the route failed, the status assertion would fail first. However, the inconsistency within the same `describe` block is confusing.

**Fix:** Restore optional chaining for consistency with lines 358/367, or convert all five to the non-optional form with a comment explaining the guard:
```typescript
const callArgs = vi.mocked(listTransactions).mock.calls[0]?.[0];
expect(callArgs?.sort).toBe('type');
```

### IN-02: `isAllowedParty` is exported but never wired into `authorizedParties` (pre-existing, comment updated this phase)

**File:** `apps/api/src/index.ts:39-48`
**Issue:** `isAllowedParty` performs wildcard subdomain validation but is not passed to `authorizedParties`. The comment was updated this phase from `@hono/clerk-auth` → `@clerk/hono`. The orphan state is pre-existing and intentionally deferred ("applied at the app layer in Phase 2+"), but there is no tracked issue to ensure it gets wired up. Until it is, subdomain JWTs will be rejected by Clerk's verifier, and `isAllowedParty` is dead code. This creates a false sense of security — a reader who sees the export assumes it is enforced somewhere.

**Fix:** Add a `// TODO(phase-2)` comment with a specific reference, or open a tracking issue. No code change required for this phase.

### IN-03: `secretKey` and `publishableKey` are not part of the typed `ClerkMiddlewareOptions` interface

**File:** `apps/api/src/index.ts:54-58`
**Issue:** `secretKey` and `publishableKey` are passed explicitly to `clerkMiddleware()`. Inspecting the `@clerk/hono` source, these are destructured from `options` before the typed spread (`...rest`) and handled internally — they work at runtime but are not declared on the `ClerkMiddlewareOptions` type (which is `Omit<AuthenticateRequestOptions, 'acceptsToken'>`). TypeScript may or may not warn depending on strict settings.

**Fix:** Verify `pnpm turbo typecheck` passes clean. If TypeScript complains, the keys can be omitted from the call since `@clerk/hono` reads them from `env(c)` automatically when not passed explicitly:
```typescript
// CLERK_SECRET_KEY and CLERK_PUBLISHABLE_KEY are read from process.env automatically by @clerk/hono.
app.use('*', clerkMiddleware({ clockSkewInMs: 10000, authorizedParties }));
```
