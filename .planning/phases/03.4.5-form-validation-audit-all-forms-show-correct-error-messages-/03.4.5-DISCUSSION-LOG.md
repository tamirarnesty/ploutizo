## Error Extraction Pattern

| Option | Description | Selected |
|--------|-------------|----------|
| `String(errors[0])` | Coerces ValidationError object to string — gives `[Object object]` | |
| `errors[0]?.toString()` | Same problem — object's toString() is not overridden | |
| `errors[0]?.message` | Works but only handles first error, no null safety | |
| `<FieldError errors={field.state.meta.errors as { message?: string }[]} />` | Prop-based: handles deduplication, multiple errors, null safety | ✓ |

**Rationale:** `FieldError` with `errors` prop is already used correctly in `CategoryForm.tsx` and is the canonical approach. The `children` form of `FieldError` (passing `String(errors[0])`) bypasses the built-in message extraction.

## Text vs FieldError

| Option | Description | Selected |
|--------|-------------|----------|
| Keep `<Text variant="error">` | Already in use in HouseholdSettingsForm, InviteMemberForm | |
| Migrate to `<FieldError errors={...} />` | Consistent with rest of codebase; handles object errors correctly | ✓ |

**Rationale:** `<Text variant="error">{errors[0]}</Text>` is broken (renders object) and inconsistent. `FieldError` is the right component.

## Missing Error Display (IncomeTypeField)

| Option | Description | Selected |
|--------|-------------|----------|
| Leave as-is (only data-invalid set) | Field turns red but user has no text telling them what's wrong | |
| Add `<FieldError errors={...} />` | Consistent with all other validated fields | ✓ |

**Rationale:** `data-invalid` without error text is incomplete — the pattern in all other fields includes both the visual state and the error message.
