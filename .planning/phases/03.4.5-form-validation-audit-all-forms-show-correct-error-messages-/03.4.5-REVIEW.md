# Phase 03.4.5: Code Review Report

**Reviewed:** 2026-05-01
**Depth:** standard
**Files Reviewed:** 9
**Status:** issues_found

## Summary

Reviewed all 9 form components for validation correctness — the primary concern of this phase. The codebase uses a consistent `useAppForm` + `FieldError` pattern and most fields wire up correctly. Three issues would cause users to submit forms without seeing errors: a missing `errorMap.onSubmit` subscriber in `InviteMemberForm`, stale default values in `HouseholdSettingsForm`, and missing `onSubmit` validators in the transfer/settlement counterpart account fields.

---

## Warnings

### WR-01: InviteMemberForm — form-level submit error never rendered

**File:** `apps/web/src/components/settings/InviteMemberForm.tsx:15-99`

**Issue:** The `onSubmit` validator (lines 16–22) attaches a concatenated Zod error to `errorMap.onSubmit`, but the JSX never subscribes to `errorMap.onSubmit` to render it. The field validator only fires `onBlur` (line 61). A user who types an invalid email and clicks "Send invite" without tabbing away will see no error message at all.

**Fix:** Add a `form.Subscribe` block after the flex row that surfaces the form-level error:

```tsx
<form.Subscribe selector={(s) => s.errorMap.onSubmit}>
  {(err) =>
    err ? (
      <Text variant="error" className="mt-1">
        {String(err)}
      </Text>
    ) : null
  }
</form.Subscribe>
```

Alternatively, add an `onChange` validator to the email field so inline errors appear without the blur requirement. The simpler fix is to also add `onSubmit` to the field-level validator and drop the form-level one entirely, consistent with how `AccountForm` handles the `name` field.

---

### WR-02: HouseholdSettingsForm — defaultValues snapshot before data loads

**File:** `apps/web/src/components/settings/HouseholdSettingsForm.tsx:14-24`

**Issue:** `defaultValues` reads `data?.settlementThreshold` at mount (line 21). The component has no loading gate and no `key` prop, so when the query is not already cached the form mounts with `thresholdDollars: ''` and never picks up the fetched value. The user opens Settings, sees a blank input, and any change they submit overwrites the real value with an empty/zero threshold.

**Fix:** Either gate on loading (matching the AccountForm/TransactionForm pattern):

```tsx
const { data, isLoading } = useGetHouseholdSettings();
if (isLoading) return <Spinner />;
```

Or add a `key` so the form remounts once data arrives:

```tsx
<HouseholdSettingsFormInner key={data?.settlementThreshold ?? 'loading'} data={data} />
```

The loading gate is cleaner; the `key` approach avoids splitting the component but resets user edits on refetch.

### WR-03: TransferFields / SettlementFields — counterpart account validators skip on submit

**Files:**
- `apps/web/src/components/transactions/TransferFields.tsx:99-101`
- `apps/web/src/components/transactions/SettlementFields.tsx:25-28` (counterpartAccountId)
- `apps/web/src/components/transactions/SettlementFields.tsx:62-65` (accountId)

**Issue:** The `counterpartAccountId` and settlement `accountId` fields only validate `onChange`. If a user opens the form with these fields empty (create mode, type selected programmatically) and submits without touching the selects, no validation fires and no error is shown. The `accountId` field in `TransactionForm` correctly uses `onSubmit` (line 293) — these companion fields should match.

**Fix:** Add `onSubmit` alongside `onChange` for each:

```tsx
// TransferFields.tsx — counterpartAccountId
validators={{
  onChange: ({ value }) => !value ? 'Destination account is required.' : undefined,
  onSubmit: ({ value }) => !value ? 'Destination account is required.' : undefined,
}}

// SettlementFields.tsx — counterpartAccountId (Source)
validators={{
  onChange: ({ value }) => !value ? 'Source account is required.' : undefined,
  onSubmit: ({ value }) => !value ? 'Source account is required.' : undefined,
}}

// SettlementFields.tsx — accountId (Destination)
validators={{
  onChange: ({ value }) => !value ? 'Destination account is required.' : undefined,
  onSubmit: ({ value }) => !value ? 'Destination account is required.' : undefined,
}}
```

## Info

### IN-01: RuleForm — no-op onSubmit validator

**File:** `apps/web/src/components/settings/RuleForm.tsx:55`

**Issue:** `validators: { onSubmit: () => undefined }` always returns undefined — it does nothing. Both the `pattern` field validators (`onChange` and `onSubmit`) already handle validation correctly.

**Fix:** Remove the no-op:

```tsx
// Remove:
validators: {
  onSubmit: (): string | undefined => undefined,
},
```

### IN-02: Inconsistent FieldError cast pattern

**Files:**
- `apps/web/src/components/accounts/AccountForm.tsx:239` — uses `as { message?: string }[]`
- `apps/web/src/components/transactions/TransactionForm.tsx:327,375,437,558,656` — uses `as unknown as { message?: string }[]`
- All transaction sub-field files use the double-cast form

**Issue:** Two patterns exist for the same cast. The double-cast (`as unknown as`) is a wider suppression. The single-cast form is preferred where the type is close enough to satisfy the compiler.

**Fix:** Standardize on the single-cast form `as { message?: string }[]` across all `<FieldError>` calls.
