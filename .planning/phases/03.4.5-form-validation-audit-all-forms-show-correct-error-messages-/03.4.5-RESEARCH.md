# Phase 03.4.5: Form Validation Audit — Research

**Researched:** 2026-05-01
**Domain:** TanStack Form field error rendering (React SPA)
**Confidence:** HIGH

## Summary

This phase is a surgical audit-and-fix: replace every broken `field.state.meta.errors` render site with the canonical `<FieldError errors={field.state.meta.errors as { message?: string }[]} />` pattern. The root cause is that TanStack Form's `ValidationError` type is an object (with a `.message` property), not a string — rendering it via `String(...)`, `.toString()`, or bare JSX produces `[Object object]`.

The canonical pattern is already live and correct in `CategoryForm.tsx`. All 14 broken sites across 9 files need to match it. Two files (`HouseholdSettingsForm`, `InviteMemberForm`) must also add a `FieldError` import and remove the `Text variant="error"` usage. One file (`TransactionTypeFields`) needs a `FieldError` added where no error display exists at all. The `RuleForm` uses `FieldError` as a wrapper but passes children instead of the `errors` prop — same fix.

No validator logic, schema changes, or API changes are involved. Every change is a JSX render substitution.

**Primary recommendation:** At each broken site, replace the broken render expression with `<FieldError errors={field.state.meta.errors as { message?: string }[]} />` and add/update imports as needed. Follow CategoryForm.tsx exactly.

### Anti-Patterns to Avoid

- **`String(errors[0])`:** Calls `.toString()` on a ValidationError object → `[Object object]`
- **`{errors[0]?.toString()}`:** Same result
- **`{errors[0]}`:** React renders the object → `[Object object]` in JSX
- **`<Text variant="error">{errors[0]}</Text>`:** Non-canonical; produces same object coercion problem
- **`<FieldError>{...}</FieldError>` (children):** FieldError renders children directly only if provided — passing an object as children still breaks

## Common Pitfalls

### Pitfall 1: Leaving the conditional guard inconsistent with CategoryForm
**What goes wrong:** Some sites use `{field.state.meta.errors.length > 0 ? <FieldError ... /> : null}` and some render unconditionally. `FieldError` returns null when `errors` is empty so unconditional is safe, but the conditional form is cleaner and consistent with the reference.
**How to avoid:** Follow CategoryForm exactly — keep the `field.state.meta.errors.length > 0` guard.
**Warning signs:** `FieldError` with no `errors` prop and no children — renders nothing but adds empty DOM.

### Pitfall 2: Partial import update in field import
**What goes wrong:** `HouseholdSettingsForm` and `InviteMemberForm` have no field import yet. Adding `FieldError` without also verifying `Field` and `FieldLabel` are imported (they are used in both files) creates a linting error.
**How to avoid:** Check full field component usage in each file before writing the import line. Both files already use `Field` via field import — confirm before assuming it's missing.

### Pitfall 3: Removing `Text variant="error"` import prematurely
**What goes wrong:** `HouseholdSettingsForm` and `InviteMemberForm` import `Text` — if no other `Text` usage remains after migrating the error site, the `Text` import becomes unused and will fail lint.
**How to avoid:** Grep for other `Text` usage in each file before removing the import.
