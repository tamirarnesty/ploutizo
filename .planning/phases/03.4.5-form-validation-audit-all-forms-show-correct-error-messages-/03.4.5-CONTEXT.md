# Phase 03.4.5: Form Validation Audit — Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Audit all `field.state.meta.errors` render sites across `apps/web/src/components/` and fix broken error display patterns. No form should show `[Object object]` or silently swallow error messages. New capabilities (new validators, new form fields, form UX redesign) are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Fix pattern
- **D-01:** Use `<FieldError errors={field.state.meta.errors as { message?: string }[]} />` at every error render site — this is the correct approach already used in `CategoryForm.tsx` and it handles `.message` extraction, deduplication, and multiple-error lists automatically.
- **D-02:** Do NOT use `String(errors[0])`, `errors[0]?.toString()`, or bare `{errors[0]}` in JSX — all of these produce `[Object object]` when TanStack Form's `ValidationError` objects are in the array.

### Text vs FieldError migration
- **D-03:** Sites currently using `<Text variant="error">{field.state.meta.errors[0]}</Text>` (HouseholdSettingsForm, InviteMemberForm) must be migrated to `<FieldError errors={field.state.meta.errors as { message?: string }[]} />`. `FieldError` is the canonical error display component — `Text variant="error"` is not.

### Missing error display
- **D-04:** `TransactionTypeFields.tsx` — `IncomeTypeField` sets `data-invalid` but renders no error text. Add `<FieldError errors={field.state.meta.errors as { message?: string }[]} />` after the `<Select>` closer, inside the `<Field>`.

### RuleForm error text
- **D-05:** `RuleForm.tsx` renders `{field.state.meta.errors[0]?.toString()}` as raw JSX inside a conditional — wrap in `<FieldError errors={field.state.meta.errors as { message?: string }[]} />`.

### TagForm scope
- **D-06:** No separate TagForm exists — tags are managed via `TransactionTagPicker.tsx` which has no `field.state.meta.errors` render sites. TagForm is excluded from audit scope.

### Claude's Discretion
- Import organization within each file (FieldError import placement)
- Whether to consolidate duplicate cast type `{ message?: string }[]` into a type alias or keep it inline

</decisions>
