## Gaps Summary

The phase successfully eliminated all three forbidden patterns (`String()`, `toString()`, `<Text variant="error">`) and introduced the canonical `<FieldError errors={...}/>` pattern syntactically. TypeScript compiles clean (0 errors). 73 API tests pass.

However, the fix is functionally hollow for 10 of 15 error sites. The root cause is a mismatch between what validators return and what `FieldError` expects:

- `FieldError.errors` prop extracts `error?.message` from each entry (packages/ui/src/components/field.tsx lines 192-196)
- Zod schema validators (CategoryFormSchema.shape.name, AccountFormSchema.shape.name) produce `ZodIssue` objects with a `.message` property — these work correctly
- Inline arrow validators (`() => 'Required.'`, `() => 'Amount must be greater than zero.'`) return plain strings — `string.message` is `undefined`, so FieldError renders null

The pre-fix behavior (`<FieldError>{String(errors[0])}</FieldError>`) used the `children` path of FieldError (line 183: `if (children) return children`), which rendered strings correctly. The new errors= prop path does not handle strings.

**Root cause of the gap:** The phase assumed `field.state.meta.errors` always contains `{message: string}` objects (per the PLAN's model). In practice, only Zod schema validators produce objects; all inline function validators produce plain strings.

**Fix options (ordered by blast radius):**
1. Update `FieldError` in `packages/ui/src/components/field.tsx` to handle both: `typeof error === 'string' ? error : error?.message` — one-line change, fixes all sites.
2. Update all inline validators to return objects: `{ message: 'Required.' }` — ~10 validators across 6 files.
3. Add a normalizer wrapper at each call site.
