## Summary

Phase 02.4 is a pure layout/navigation refactor. No new API routes, no data fetching, and no new shadcn components need to be installed. The work falls into four discrete sub-tasks: (1) restructure `_layout.tsx` (the actual active shell) to move `SidebarProvider` below a new `TopBar` component, (2) rebuild `AppSidebar.tsx` with `collapsible="icon"`, a wordmark+OrganizationSwitcher header row, and a footer that holds ThemeToggle + Settings link + collapse toggle, (3) update `_layout.settings/route.tsx` to add a two-column vertical sub-nav layout, and (4) clean up the orphaned `components/layout/LayoutShell.tsx` file.

**Critical discovery:** The CONTEXT.md and UI-SPEC.md reference `apps/web/src/components/layout/LayoutShell.tsx` as the shell target, but the actual live shell is the inline `LayoutShell` function in `apps/web/src/routes/_layout.tsx`. The component file in `components/layout/` is not imported anywhere â€” it is a dead file. All layout changes must be made to `apps/web/src/routes/_layout.tsx`, not `components/layout/LayoutShell.tsx`.

The shadcn Sidebar component already installed in `packages/ui` natively supports `collapsible="icon"` via `group-data-[collapsible=icon]` Tailwind data-attribute selectors. `SidebarMenuButton` already applies `group-data-[collapsible=icon]:size-8! group-data-[collapsible=icon]:p-2!` automatically, centering the icon. The header collapse-to-nothing behaviour requires adding `group-data-[collapsible=icon]:hidden` to the header row's content wrapper â€” this is the correct CSS data-attribute approach confirmed in the installed component source.

**Primary recommendation:** Make all changes in `_layout.tsx` (shell) + `AppSidebar.tsx` (sidebar) + `_layout.settings/route.tsx` (settings layout). Delete or leave `components/layout/LayoutShell.tsx` (it is dead code). No new packages to install.

### Pattern 1: Shell restructure â€” TopBar above SidebarProvider

**What:** `TopBar` is a standalone component rendered above the `SidebarProvider` wrapper so it spans the full viewport width. `SidebarProvider` is moved inside the inner `flex flex-1` row alongside `AppSidebar` + `SidebarInset`.

**When to use:** Any time a top bar must span the full width including above the sidebar rail.

**Key detail:** The current `_layout.tsx` wraps everything inside `SidebarProvider`. The new structure lifts the top bar out of `SidebarProvider`. `SidebarTrigger` (which calls `toggleSidebar()` from the `useSidebar()` context) must remain inside `SidebarProvider` â€” the mobile `SidebarTrigger` in `TopBar` therefore requires `TopBar` to be rendered as a child of `SidebarProvider`, or `TopBar` must live inside the `SidebarProvider` tree. [VERIFIED: sidebar.tsx `useSidebar()` throws if used outside provider context]

**Resolution:** The UI-SPEC.md layout places `TopBar` above `SidebarProvider`. This means `SidebarTrigger` in `TopBar` cannot use `useSidebar()` context directly. The correct approach is to render `TopBar` inside `SidebarProvider`'s tree â€” either restructure so `SidebarProvider` wraps both `TopBar` and the sidebar+content row, or use a different approach for the mobile trigger. The simplest solution: put `SidebarProvider` at the outermost level and have `TopBar` as a child of `SidebarProvider` â€” this satisfies context availability while still allowing `TopBar` to span full width with `w-full`. The UI-SPEC layout diagram is the visual target; the actual DOM nesting that achieves it may differ.

**Recommended implementation:**

```tsx
// apps/web/src/routes/_layout.tsx
// Source: [VERIFIED: sidebar.tsx context requirements + UI-SPEC.md]

function LayoutShell() {
  return (
    <SidebarProvider>
      <div className="flex flex-col h-screen w-full">
        <TopBar />
        <div className="flex flex-1 min-h-0">
          <AppSidebar />
          <SidebarInset>
            <main className="flex-1 overflow-auto p-6">
              <Outlet />
            </main>
          </SidebarInset>
        </div>
      </div>
    </SidebarProvider>
  )
}
```

This wraps everything in `SidebarProvider` so `TopBar`'s `SidebarTrigger` has context access, and the inner `div` handles the flex layout. `SidebarProvider` renders `group/sidebar-wrapper flex min-h-svh w-full` â€” the inner `flex flex-col h-screen w-full` div overrides this to achieve the top-bar + content-row stacking.

### Anti-Patterns to Avoid

- **Placing `SidebarTrigger` outside `SidebarProvider`:** `useSidebar()` throws an error if called outside `SidebarProvider` context. `SidebarTrigger` calls `toggleSidebar()` from that context. The `TopBar` component must be rendered inside `SidebarProvider`.
- **Using `data-[state=collapsed]` instead of `data-[collapsible=icon]`:** The sidebar sets `data-collapsible="icon"` only when collapsed with `collapsible="icon"` mode (line 211 sidebar.tsx). The `data-state` is `"collapsed"` but the group selector for icon mode uses `data-collapsible`. Use `group-data-[collapsible=icon]:hidden` not `group-data-[state=collapsed]:hidden`.
- **Modifying `components/layout/LayoutShell.tsx`:** This file is not imported anywhere â€” it is dead code. All changes go to `apps/web/src/routes/_layout.tsx`.
- **Defining inline components inside LayoutShell render:** Per `rerender-no-inline-components` rule â€” `TopBar` must be defined as a named function outside the shell, not inline inside `LayoutShell()`.

## Common Pitfalls

### Pitfall 1: `SidebarTrigger` outside `SidebarProvider` context

**What goes wrong:** If `TopBar` is rendered above/outside `SidebarProvider`, calling `<SidebarTrigger />` inside it throws: `"useSidebar must be used within a SidebarProvider."` at runtime.

**Why it happens:** UI-SPEC.md's layout diagram shows `TopBar` above `SidebarProvider` visually, but that is the rendered visual layout, not the component tree. `SidebarTrigger` calls `useSidebar()` which requires the React context from `SidebarProvider`.

**How to avoid:** Nest `TopBar` inside `SidebarProvider` in the component tree. Use a wrapper `div` inside `SidebarProvider` to achieve the visual top-bar-spans-full-width layout. See Pattern 1 above.

**Warning signs:** `Error: useSidebar must be used within a SidebarProvider.` in the browser console.

### Pitfall 2: Wrong `group-data-` selector for icon collapse

**What goes wrong:** Using `group-data-[state=collapsed]:hidden` on the sidebar header content hides the header whenever the sidebar is closed on mobile (drawer closed = `state=collapsed`), not just when in icon-only rail mode on desktop.

**Why it happens:** `data-state="collapsed"` is set any time the sidebar is not open, including when the mobile Sheet is closed. `data-collapsible="icon"` is only set when `collapsible="icon"` AND the sidebar is in the collapsed rail state on desktop.

**How to avoid:** Use `group-data-[collapsible=icon]:hidden` (not `group-data-[state=collapsed]:hidden`) on the header content wrapper.

**Warning signs:** Sidebar header disappears when mobile sheet closes.

### Pitfall 3: Double padding in settings layout

**What goes wrong:** The `<main className="flex-1 overflow-auto p-6">` in `LayoutShell` adds `p-6` to all route content. The settings layout also starts with `<div className="flex gap-8 p-6">` per the UI-SPEC â€” this results in 48px of padding on all sides of the settings content.

**Why it happens:** The settings layout defines its own internal padding structure (the two-column layout with `p-6`) but the outer `main` already pads the content area.

**How to avoid:** The settings sub-routes (categories, merchant-rules, household) individually own their content padding. The `SettingsLayout` should NOT add `p-6` on the outer wrapper â€” the outer `main`'s `p-6` is sufficient. The two-column layout should be `<div className="flex gap-8">` without re-adding `p-6`. Alternatively, keep `p-6` on the settings layout and remove it from `main` (not recommended â€” would break non-settings pages).

**Warning signs:** Excessive whitespace around settings content that doesn't match other pages.

### Pitfall 4: Dead file confusion

**What goes wrong:** Planner or executor edits `apps/web/src/components/layout/LayoutShell.tsx` â€” the changes have no effect because this file is not imported by anything.

**Why it happens:** CONTEXT.md canonical refs mention `LayoutShell.tsx` as a target, but the actual shell is `_layout.tsx`. This was likely a refactor from an earlier phase that left the component file orphaned.

**How to avoid:** All shell changes go to `apps/web/src/routes/_layout.tsx`. The component file in `components/layout/` is dead code and should be deleted in this phase.

**Warning signs:** Changes to `LayoutShell.tsx` have no visible effect in the browser.

### Pitfall 5: `OrganizationSwitcher` in a collapsed sidebar header breaking layout

**What goes wrong:** `OrganizationSwitcher` is a Clerk composite component with its own sizing. In a small header row (`h-12`) next to the "Ploutizo" wordmark, if the org switcher button is wider than expected it can overflow or push the wordmark off screen.

**Why it happens:** Clerk's `OrganizationSwitcher` renders a button + dropdown; its width depends on the org name length and the button appearance mode.

**How to avoid:** Wrap the right slot in `min-w-0 overflow-hidden` or restrict the org switcher to a fixed/max width. Use `appearance` prop or CSS to constrain the button size. Verify visually with a long org name.

**Warning signs:** Header row wraps or overflows with a long organization name.

1. **`p-6` on `main` vs settings layout padding**
