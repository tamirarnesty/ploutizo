<decisions>
## Implementation Decisions

### Package Installation
- **D-01:** Install `@tanstack/react-form` in `apps/web`. Install shadcn form components
  into `packages/ui` (not `apps/web`) so they are available to all apps. Follow the
  shadcn TanStack Form integration guide: https://ui.shadcn.com/docs/forms/tanstack-form
- **D-02:** Zod is the validation adapter. Use `@tanstack/zod-form-adapter` alongside
  `@tanstack/react-form`. Zod is already installed at the API layer; use the same library
  for client-side form validation.

### Zod Schema Location
- **D-03:** Form Zod schemas live in `packages/validators` — the same package where API
  Zod schemas already live. Do not co-locate schemas in component files. Do not create
  per-feature schema files in `apps/web`. One canonical location for all schemas.

### AccountSheet — Ownership as Form Fields
- **D-04:** `ownership` (`"personal" | "shared"`) and `memberIds` (`string[]`) become
  first-class TanStack Form fields, not separate React state. The full form payload
  (including ownership + memberIds) is validated by Zod at submit time.
- **D-05:** **Loading gate pattern for async defaults.** `existingMembers` (co-owner list
  for edit mode) is loaded async via `useGetAccountMembers`. Do NOT render the form until
  this query resolves — show a spinner inside the sheet while loading. Form initializes
  with fully-loaded `defaultValues`; no `useEffect` or `form.reset()` needed after mount.
  This extends the existing `key` prop pattern at the AccountSheet call site: form is
  freshly mounted with complete data every time.

### RuleDialog — Replace __none__ Sentinel
- **D-06:** Replace the `__none__` sentinel value for the optional `categoryId` Select
  with proper `null`/`undefined` handling. The Zod schema should type this field as
  `z.string().uuid().nullable()` (or `.optional()`). The researcher/planner must verify
  that Radix `<Select>` reliably handles an unset/null value in Radix UI's current version
  before confirming the approach. If Radix Select cannot handle null as a controlled value,
  the sentinel may need to remain as a UI-layer shim (transform to null in Zod `.transform()`
  before submission).

### HouseholdSettings — Migrate
- **D-07:** Migrate `HouseholdSettings` to TanStack Form + Zod despite having only one
  field. Consistency across all forms outweighs the simplicity argument. Zod schema
  validates the threshold as a non-negative number.

### Component Decomposition — Atomic Hierarchy
- **D-09:** Every component must do one thing. No component should be "too nested" or
  responsible for both container logic and render details. Follow the React "Thinking in
  React" hierarchy: container → feature component → sub-components.
- **D-10:** **Form content extracted into dedicated form components.** The sheet/dialog
  component owns open/close state and renders the chrome (Sheet, Dialog, header, footer,
  buttons). The form JSX lives in a separate sibling file:
  - `AccountSheet.tsx` → shell; `AccountForm.tsx` → form fields
  - `CategoryDialog.tsx` → shell; `CategoryForm.tsx` → form fields
  - `RuleDialog.tsx` → shell; `RuleForm.tsx` → form fields
  - `HouseholdSettings.tsx` → shell; `HouseholdSettingsForm.tsx` → form fields
  The form component receives the TanStack Form instance (or its own `useAppForm` call)
  and renders fields. The shell calls the mutation and passes `onSuccess`/`onClose`.
- **D-11:** **Repeated list items extracted into atomic row components.** Any map() over
  items that produces non-trivial JSX must be its own component. Examples:
  - `MerchantRulesSettings.tsx` sortable item → `MerchantRuleRow.tsx`
  - Apply this audit to all list/table renders found during implementation — if the
    item JSX is more than ~5 lines, extract it.
- **D-12:** One component per file. No exporting multiple components from the same file
  (already in STANDARDS.md — enforce strictly during this phase).

### Custom Inputs (LucideIconPicker, ColourPicker)
- **D-08:** `LucideIconPicker` and `ColourPicker` are custom non-native components. They
  must be wrapped as TanStack Form fields using `form.Field` + `field.handleChange`. The
  researcher should confirm the exact adapter pattern from TanStack Form docs for
  custom/controlled inputs. No changes to the picker component APIs — only the binding
  changes.

### Claude's Discretion
- Exact Zod schema field names and `.transform()` calls (match existing API payload shapes)
- Whether to use a shared `AppForm` wrapper component or call `useAppForm` directly per form
- Error message strings (keep existing copy unless schema validation requires changes)
- Loading skeleton/spinner design inside AccountSheet while existingMembers loads

</decisions>

## Existing Code Insights

- `apps/web/src/lib/data-access/accounts/index.ts` — `useGetAccountMembers(accountId)`: provides async co-owner data for AccountSheet edit mode (used in loading gate pattern)

### Established Patterns
- `key` prop at AccountSheet call site (`key={editingAccount?.id ?? "new"}`) — form re-mounts on account switch; loading gate extends this naturally
- All data-access hooks import from barrel: `@/lib/data-access/{domain}` (enforced by lint rule)
- `apiFetch<T>()` in `queryClient.ts` — React Query mutations use this for API calls; form onSubmit calls the same mutations
- Zod schemas in `packages/validators` already export to both API routes and `packages/types`

### Integration Points
- `packages/ui/package.json` — Add `@tanstack/react-form` and `@tanstack/zod-form-adapter` as deps; shadcn form components installed here
- `apps/web/package.json` — Add `@tanstack/react-form` as a dep (peer with packages/ui)
- Each migrated form file: remove `useState` per-field boilerplate; import `useAppForm` from `@ploutizo/ui/components/form`

  diagram explicitly. Sheet/Dialog = chrome only. Form content = dedicated sibling component.

  called out by name as the canonical example.
