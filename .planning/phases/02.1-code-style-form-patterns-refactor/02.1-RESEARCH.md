## Summary

This phase is a pure internal refactor: migrate 4 forms (`AccountSheet`, `CategoryDialog`, `RuleDialog`, `HouseholdSettings`) from manual `useState` per-field state to TanStack Form v1 + Zod, while also splitting form JSX into dedicated sibling components (D-10) and extracting `MerchantRuleRow` (D-11).

Neither `@tanstack/react-form` nor `@tanstack/zod-form-adapter` are installed in the project yet ŌĆö both must be added. The shadcn `tanstack-form` component block is **not in the registry** for the `radix-nova` style used by this project. The `form.tsx` wiring file (`createFormHookContexts` + `createFormHook` ŌåÆ `useAppForm`) must be written by hand following the TanStack Form v1 composition docs. This is straightforward ŌĆö approximately 20 lines of setup code.

The critical D-06 question about Radix Select and null values is fully resolved: Radix Select 2.2.6 (the version in this project) does **not** accept `null` as a controlled `value`. The `shouldShowPlaceholder` function treats `value === ""` or `value === undefined` as "show placeholder". Empty string `""` is the correct UI-layer controlled value for "no selection". `SelectItem` throws if given `value=""`. Therefore the pattern is: `value={field.state.value ?? ""}` on the trigger, `onValueChange={(v) => field.handleChange(v === "" ? null : v)}` on change, and the `null` lives only in the form state and API payload ŌĆö not in the Radix Select `value` prop. No sentinel constant needed in this project going forward.

**Primary recommendation:** Hand-write `packages/ui/src/components/form.tsx` using `createFormHookContexts` + `createFormHook` from `@tanstack/react-form`, then migrate each form in the order given by the planner. The CODE-EXAMPLES.md file provides exact patterns for every scenario.

### Pattern 1: form.tsx Setup (createFormHookContexts + createFormHook)

**What:** One-time wiring file that creates the project-wide `useAppForm` hook and `AppField` component.

**Source:** TanStack Form v1 form-composition docs + shadcn TanStack Form guide

```typescript
// packages/ui/src/components/form.tsx
import { createFormHookContexts, createFormHook } from '@tanstack/react-form'

export const { fieldContext, formContext, useFieldContext } = createFormHookContexts()

export const { useAppForm } = createFormHook({
  fieldContext,
  formContext,
  fieldComponents: {},
  formComponents: {},
})
```

`useAppForm` accepts all `useForm` options. It returns the form instance augmented with `form.AppField` and `form.AppForm`.

**Export path:** Add to `packages/ui` package.json exports:
```json
"./components/form": "./src/components/form.tsx"
```

Consuming forms import:
```typescript
import { useAppForm } from "@ploutizo/ui/components/form"
import { zodValidator } from "@tanstack/zod-form-adapter"
```

### Pattern 2: useAppForm with zodValidator

```typescript
const form = useAppForm({
  defaultValues: { ... } satisfies MyFormType,
  validatorAdapter: zodValidator(),          // enables Zod schemas in validators options
  validators: { onSubmit: MyFormSchema },    // form-level: run on submit
  onSubmit: ({ value }) => {
    // value is typed as MyFormType
    mutation.mutate(transformedPayload, { onSuccess: onClose })
  },
})
```

`zodValidator()` returns a validator factory object consumed by `validatorAdapter`. With `validatorAdapter` set, you may pass Zod schemas directly to `validators.onSubmit`, `validators.onChange`, or field-level `validators.onBlur`.

**Gotcha:** `validatorAdapter` must be set at the form level, not only at the field level, for form-wide `validators.onSubmit` to work.

### Pattern 3: form.AppField (render prop pattern)

`form.AppField` is the context-aware field wrapper. It binds the field into form context automatically.

```tsx
<form.AppField name="fieldName" validators={{ onChange: FieldSchema }}>
  {(field) => (
    <div className="space-y-1">
      <Label htmlFor="field-id" className="text-xs font-medium">Label</Label>
      <Input
        id="field-id"
        value={field.state.value}
        onChange={(e) => field.handleChange(e.target.value)}
        onBlur={field.handleBlur}
        aria-invalid={field.state.meta.errors.length > 0}
      />
      {field.state.meta.errors.length > 0 && (
        <p className="text-xs text-destructive">{field.state.meta.errors[0]}</p>
      )}
    </div>
  )}
</form.AppField>
```

- `field.state.value` ŌĆö current field value (typed)
- `field.state.meta.errors` ŌĆö array of error strings (from Zod or custom validators)
- `field.handleChange(value)` ŌĆö update field value
- `field.handleBlur()` ŌĆö mark touched, trigger onBlur validators

### Pattern 4: form.Subscribe for submit button state

```tsx
<form.Subscribe selector={(s) => s.isSubmitting}>
  {(isSubmitting) => (
    <Button type="submit" disabled={isSubmitting}>
      {isSubmitting && <Spinner className="mr-1" />}
      Save changes
    </Button>
  )}
</form.Subscribe>
```

### Pattern 5: Custom inputs (LucideIconPicker, ColourPicker)

No changes to picker component APIs. `field.handleChange` is the bridge:

```tsx
<form.AppField name="icon">
  {(field) => (
    <LucideIconPicker
      value={field.state.value ?? null}
      onChange={(v) => field.handleChange(v)}
    />
  )}
</form.AppField>
```

Both pickers use `value: string | null` + `onChange: (value: string) => void` ŌĆö compatible with field binding.

### Pattern 6: Radix Select with nullable field (D-06 resolved)

**Finding (HIGH confidence, verified from Radix Select 2.2.6 source):**
- `shouldShowPlaceholder(value)` returns `true` when `value === "" || value === undefined`
- `SelectItem` throws a runtime error if `value=""` is passed to an item
- `value` prop is typed as `string` ŌĆö Radix Select does NOT handle `null` directly

**Resolution:** Use `""` (empty string) as the UI-layer bridge value, never pass `null` to the `value` prop:

```tsx
<form.AppField name="categoryId">
  {(field) => (
    <Select
      value={field.state.value ?? ""}       // null ŌåÆ "" for Radix
      onValueChange={(v) => field.handleChange(v === "" ? null : v)}  // "" ŌåÆ null in form state
    >
      <SelectTrigger className="w-full">
        <SelectValue placeholder="No category" />  // shown when value is ""
      </SelectTrigger>
      <SelectContent>
        {/* No item with value="" ŌĆö would throw */}
        {categories.map((cat) => (
          <SelectItem key={cat.id} value={cat.id}>{cat.name}</SelectItem>
        ))}
      </SelectContent>
    </Select>
  )}
</form.AppField>
```

Form state holds `null`. API payload receives `null`. Radix never sees `null`.

**No `__none__` sentinel needed.** The previous `__none__` sentinel is eliminated entirely.

### Pattern 7: Loading gate (D-05)

Gate on the query's `isLoading` state before mounting the form. The form component is split into a loader + inner component so the inner always mounts with complete data.

```tsx
// AccountForm.tsx ŌĆö loading gate
export const AccountForm = ({ account, onClose }: Props) => {
  const { data: existingMembers, isLoading } = useGetAccountMembers(account?.id ?? null)

  if (isLoading) {
    return (
      <div className="flex flex-1 items-center justify-center">
        <Spinner />
      </div>
    )
  }

  return <AccountFormInner account={account} existingMembers={existingMembers ?? []} onClose={onClose} />
}
```

The inner component calls `useAppForm({ defaultValues: { ...fully-loaded-data } })` ŌĆö no `useEffect` resets.

**Important:** `useGetAccountMembers` is already `enabled: accountId !== null` ŌĆö for create mode (`account === null`), the query never fires and `isLoading` is `false` immediately. The loading gate only blocks in edit mode while co-owners load.

### Pattern 8: Cross-field validation (regex on blur)

```tsx
<form.AppField
  name="pattern"
  validators={{
    onBlur: ({ value, fieldApi }) => {
      const matchType = fieldApi.form.getFieldValue("matchType")
      if (matchType !== "regex") return undefined
      try { new RegExp(value); return undefined }
      catch { return "Invalid regular expression." }
    },
  }}
>
```

`fieldApi.form.getFieldValue("matchType")` reads another field's current value inside a validator.

### Pattern 9: Form-level error after mutation failure

```tsx
onSubmit: ({ value }) => {
  mutation.mutate(payload, {
    onSuccess: onClose,
    onError: () => form.setErrorMap({
      onSubmit: "Couldn't save changes. Check your connection and try again."
    }),
  })
}
```

`form.setErrorMap` sets form-level errors accessible via `form.state.errorMap.onSubmit`.

### Anti-Patterns to Avoid

- **`useState` per field** ŌĆö replaced by `form.AppField`; existing forms have 4ŌĆō7 per-field state vars to eliminate
- **`useEffect(() => form.reset(...), [data])` for async defaults** ŌĆö use loading gate pattern (D-05) instead; form always mounts with complete data
- **Empty string as `SelectItem` value** ŌĆö Radix throws; use placeholder pattern for "no selection"
- **Passing `null` to Radix `Select value` prop** ŌĆö silent malfunction; use `?? ""` bridge
- **Co-locating Zod schemas in component files** ŌĆö D-03c mandates `packages/validators`
- **Multiple component exports from one file** ŌĆö D-12 strictly forbids; one file per component

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Form field state | Per-field `useState` | `useAppForm` + `form.AppField` | Eliminates 4ŌĆō7 state vars per form; handles touched, dirty, isSubmitting |
| Form-level error display | `useState` for mutation error | `form.setErrorMap` + `form.state.errorMap` | Colocates error state in form instance |
| Zod schema validation | Manual `if (!name.trim())` blocks | `validators: { onSubmit: Schema }` | Declarative; reuses API schema logic |
| Submit button disabled state | `const isSaving = mutation.isPending` | `form.Subscribe selector={(s) => s.isSubmitting}` | Prevents stale closure bugs |

## Common Pitfalls

### Pitfall 1: Forgetting `validatorAdapter` on the form
**What goes wrong:** Passing a Zod schema to `validators.onSubmit` does nothing ŌĆö no validation fires.
**Why it happens:** Without `validatorAdapter: zodValidator()`, TanStack Form doesn't know how to run Zod schemas.
**How to avoid:** Always include `validatorAdapter: zodValidator()` in `useAppForm` options when using Zod schemas in `validators`.
**Warning signs:** Form submits with invalid data, no errors shown.

### Pitfall 2: Radix Select receiving `null` as value
**What goes wrong:** The Select renders with no visible value, but the placeholder may or may not show ŌĆö behavior is inconsistent.
**Why it happens:** Radix `shouldShowPlaceholder` only returns true for `""` or `undefined`, not `null`.
**How to avoid:** Always use `value={field.state.value ?? ""}` for nullable Select fields.
**Warning signs:** Select appears blank but not showing placeholder text.

### Pitfall 3: `isLoading` is `false` when `enabled: false` (create mode)
**What goes wrong:** Developer thinks the loading gate might block create mode unnecessarily.
**Why it happens:** `useGetAccountMembers(null)` has `enabled: false`; React Query immediately sets `isLoading: false`.
**How to avoid:** No change needed ŌĆö the gate is safe for both create and edit modes.

### Pitfall 4: Accessing stale `field.state.value` in submit handler
**What goes wrong:** Using closure-captured state instead of `value` from `onSubmit: ({ value }) => {...}`.
**Why it happens:** Developer habit from `useState` pattern.
**How to avoid:** Always use the `value` parameter in `onSubmit`. Never reference form field `useState` variables.

### Pitfall 5: `form.AppField` vs `form.Field`
**What goes wrong:** Using `form.Field` instead of `form.AppField` ŌĆö field component does not receive form context.
**Why it happens:** Both exist on the form instance; `form.Field` is the lower-level primitive.
**How to avoid:** Always use `form.AppField` (from `createFormHook`). `form.Field` is available but bypasses the composition context.

### Pitfall 6: `packages/ui` exports map not updated
**What goes wrong:** `import { useAppForm } from "@ploutizo/ui/components/form"` fails at runtime with module resolution error.
**Why it happens:** `packages/ui/package.json` uses explicit `exports` map; new files must be added.
**How to avoid:** Add `"./components/form": "./src/components/form.tsx"` to `packages/ui/package.json` exports when creating `form.tsx`.

### Pitfall 7: `satisfies` on defaultValues breaks type inference
**What goes wrong:** Using `satisfies` with a type that has optional fields causes TypeScript to widen the type.
**Why it happens:** TanStack Form infers `TFormData` from `defaultValues`. If you say `{} satisfies Partial<AccountForm>`, the inferred type may be `{}`.
**How to avoid:** `defaultValues` should be a complete object typed with the exact form type. Use `satisfies AccountFormType` with a full object, not a partial.

## D-06 Resolution: Radix Select Null Handling (VERIFIED)

**Source:** `@radix-ui/react-select@2.2.6` source code (installed in project node_modules)

| Finding | Evidence |
|---------|---------|
| `value` prop is typed as `string` | TypeScript types in `@radix-ui/react-select` |
