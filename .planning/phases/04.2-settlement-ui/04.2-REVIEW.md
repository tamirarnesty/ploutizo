## Summary

29 source files reviewed covering the settlement UI, data access layer, validators, API service, and shared UI components. No security vulnerabilities or data-loss blockers were found. The service layer, validator schema, and test coverage are solid.

Eleven warnings were found, concentrated in `SettleDialog.tsx` and `CardBalancesGrid.tsx`. The most impactful are: (1) the `useEffect` in `SettleDialog` that can reset a partially-filled form when accounts refetch while open, (2) the amount field being unclearable once set, (3) the radio member-select silently overwriting a user-typed custom amount, and (4) the URL search params accepting unparseable date strings that produce "Invalid Date" in the rendered label.

## Warnings

### WR-01: useEffect resets form on accounts refetch while dialog is open

**File:** `apps/web/src/components/dashboard/SettleDialog.tsx:152`

**Issue:** The effect dep array includes `firstSourceId` (derived from `sourceAccountOptions[0]?.id`). If TanStack Query refetches accounts while the dialog is open (focus refetch, 60s staleTime expiry, or cache invalidation from another mutation), `sourceAccountOptions` re-evaluates, `firstSourceId` gets a new identity/value, and the effect fires — calling `form.reset()` and discarding any in-progress user edits (custom amount, custom date, notes). The comment says `firstSourceId` is "intentionally excluded" but it is in the dep array.

**Fix:** Gate the effect on the open transition (false → true) only. The simplest approach is tracking a ref for the previous account id:

```tsx
const prevAccountIdRef = useRef<string | null>(null);
useEffect(() => {
  if (!open || !account) return;
  if (prevAccountIdRef.current === account.account.id) return; // already seeded for this account
  prevAccountIdRef.current = account.account.id;
  form.reset(getInitialValues(account, firstSourceId, todayIso));
}, [open, account?.account.id]); // firstSourceId / todayIso intentionally excluded — seeded once on open
```

### WR-02: Amount input cannot be cleared — zero becomes permanent

**File:** `apps/web/src/components/dashboard/SettleDialog.tsx:326-329`

**Issue:** `parseFloat('')` returns `NaN`; `Number.isFinite(NaN)` is false, so `field.handleChange(0)` is called. The display then shows `'0'` (via `String(0)`). The user cannot clear the field — any attempt to delete all digits snaps the value back to `0`. This blocks entering a precise partial amount without first selecting all text.

**Fix:** Treat empty string as a distinct state — either store it and coerce to 0 only at submit time, or use `Number.isNaN(v)` as the "cleared" sentinel:

```tsx
onChange={(e) => {
  const raw = e.target.value;
  if (raw === '') {
    field.handleChange(0); // or a dedicated "empty" sentinel
    return;
  }
  const v = parseFloat(raw);
  if (Number.isFinite(v)) field.handleChange(v);
  // if not finite and not empty, leave field unchanged (ignore garbage chars)
}}
```

Additionally expose the empty-string display separately so the input reads blank instead of `'0'` when the user clears:

```tsx
value={field.state.value === 0 ? '' : String(field.state.value)}
```

### WR-03: Member radio selection overwrites user-typed custom amount without warning

**File:** `apps/web/src/components/dashboard/SettleDialog.tsx:206-218`

**Issue:** `onValueChange` unconditionally calls `form.setFieldValue('amountDollars', Math.abs(picked.balanceCents) / 100)` on every member selection. If a user types a custom partial amount and then switches the "Settling for" member (e.g. to see the other member's balance displayed), their typed amount is silently replaced. This is a correctness defect — the overwrite is destructive and not user-initiated.

**Fix:** Only seed the amount when the field has not been dirtied by the user. Check `form.getFieldMeta('amountDollars').isDirty` before overwriting:

```tsx
onValueChange={(next) => {
  field.handleChange(next);
  const picked = account.members.find((m) => m.member.id === next);
  if (picked && !form.getFieldMeta('amountDollars').isDirty) {
    form.setFieldValue('amountDollars', Math.abs(picked.balanceCents) / 100);
  }
}}
```

### WR-04: URL search params accept non-date strings — produces "Invalid Date" label

**File:** `apps/web/src/components/dashboard/dashboardSearch.ts:6-9`

**Issue:** `from` and `to` are `z.string().optional().catch(undefined)` — any non-empty string passes. In `PeriodRangePicker.tsx:79`, the raw string is passed directly to `new Date(search.from + 'T00:00:00')`. If `search.from` is e.g. `"garbage"`, `new Date('garbageT00:00:00')` is an Invalid Date object. `formatRangeLabel` then calls `range.from.getFullYear()` → `NaN`, producing a visible label like `"garbage, NaN"`. Additionally, `getFullYear()` calls on Invalid Date objects can throw in some environments.

**Fix:** Add a date format check in the schema:

```ts
export const dashboardSearchSchema = z.object({
  from: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional().catch(undefined),
  to: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional().catch(undefined),
});
```

The `.catch(undefined)` already handles the rejection gracefully — the regex just needs to be added before it.

### WR-05: Error state not handled in SettlementSummaryPane, Dashboard, and SidebarMembersSection

**File:** `apps/web/src/components/dashboard/SettlementSummaryPane.tsx:19`
**Also:** `apps/web/src/components/dashboard/Dashboard.tsx:20-21`, `apps/web/src/components/SidebarMembersSection.tsx:45`

**Issue:** All three consumers of `useGetSettlements` destructure only `{ data, isLoading }`. On a network error (`isError: true`, `data: undefined`), the skeleton disappears and members render with `$0.00 across 0 cards` (SettlementSummaryPane) or all-zero balances in the sidebar. The user sees no error signal and may incorrectly believe every account is balanced.

**Fix:** Destructure `isError` and render an error state:

```tsx
const { data, isLoading, isError } = useGetSettlements();
// ...
{isError ? (
  <Text variant="caption" className="text-muted-foreground">
    Could not load balances.
  </Text>
) : isLoading ? (
  <Skeleton className="h-10 w-full" />
) : (
  /* members.map(...) */
)}
```

### WR-06: Duplicate error reporting — toast + inline error on settlement failure

**File:** `apps/web/src/lib/data-access/settlements/useCreateSettlement.ts:22-24`
**Also:** `apps/web/src/components/dashboard/SettleDialog.tsx:135-138`

**Issue:** On mutation failure, `useCreateSettlement`'s `onError` fires `toast.error('Failed to record settlement. Try again.')`. The mutation caller in `SettleDialog.onSubmit` also provides a per-call `onError` that calls `form.setErrorMap(...)` to show inline error text. Both fire simultaneously — the user sees a toast and a red inline message for the same error. The duplication is confusing and wastes UI real estate.

**Fix:** Pick one channel. Inline error (via `setErrorMap`) is more contextual for a form dialog. Remove the `toast.error` call from `useCreateSettlement.onError`, or remove the per-call `onError` override in `SettleDialog` and rely solely on the toast.

### WR-07: BREAKDOWN cell renders zero-balance members (visual noise + misleading 0% bars)

**Issue:** `account.members.map(...)` renders all members including those with `balanceCents === 0`. For a member with no balance on this card, the row shows a 0% progress bar and `$0.00` with a Settle button. This produces visual clutter and implies an action is needed where none is.

### WR-08: STATUS column defaults unknown status values to "On Track" badge

**File:** `apps/web/src/components/dashboard/CardBalancesGrid.tsx:236-260`

**Issue:** The final `return` in the status cell renders an "On Track" badge for any non-`'due_soon'` truthy `status` value. If the API adds `'overdue'` or `'past_due'` in a future phase, those will silently display as green "On Track" — a correctness hazard. The `computeNextDueDate` utility currently only returns `'due_soon'` or `null`, but the type should be constrained rather than relying on that invariant holding.

**Fix:** Use an exhaustive check or explicit allow-list:

```tsx
if (s === 'due_soon') {
  return <Badge ...>Due Soon</Badge>;
}
if (s === 'on_track') {
  return <Badge ...>On Track</Badge>;
}
// Unknown status — render nothing rather than mislabeling
return <Text variant="caption" className="text-muted-foreground">{s}</Text>;
```

### WR-09: `progressFill` exported but never used — `Progress` indicator color not applied

**File:** `apps/web/src/lib/memberColors.ts:8-10`
**Also:** `apps/web/src/components/dashboard/CardBalancesGrid.tsx:168-172,194`

**Issue:** Every `MemberColorSlot` exports `progressFill` (e.g. `'bg-blue-500'`), which is intended for per-member progress bar coloring per the UI-SPEC. However, no caller reads `progressFill`. `CardBalancesGrid` only uses `slot.bg` as a decorative `data-color-slot` attribute. The `<Progress>` component in `progress.tsx` renders a `<ProgressIndicator>` internally with no way to pass a custom indicator className — so there is no active path to apply per-member colors to the progress bars. The color intent from the spec is silently dropped.

**Fix (short-term):** Remove the unused `progressFill` field from `MEMBER_COLORS` to avoid the false impression the feature is wired up. Document that per-member progress fill is deferred to Phase 7.3 when the Progress component is extended.

**Fix (if wiring is in scope now):** Extend `Progress` in `packages/ui` to accept an `indicatorClassName` prop, then pass `slot.progressFill` through from `CardBalancesGrid`.

### WR-10: Ref assigned during render body in AppSidebar

**File:** `apps/web/src/components/AppSidebar.tsx:50-51`

**Issue:** `sidebarRef.current = sidebarCtx` executes during the render function body. Under React 19 concurrent mode, renders may be thrown away and re-executed; each throw-away render still mutates the ref. While in practice this pattern typically "works," it is documented as an anti-pattern by the React team and violates the render-purity constraint. The correct approach per React docs is `useLayoutEffect` (synchronous, runs after paint) or `useInsertionEffect`.

**Fix:**

```tsx
const sidebarRef = useRef(sidebarCtx);
useLayoutEffect(() => {
  sidebarRef.current = sidebarCtx;
});
```

### WR-11: Date validation parity gap between form and server schemas

**File:** `apps/web/src/components/dashboard/settleFormSchema.ts:18`
**Also:** `packages/validators/src/settlements.ts:24`

**Issue:** `settleFormSchema.date` uses `z.string().min(1, 'Date is required.')`. The server schema (`createSettlementSchema`) uses `z.string().date()` which enforces ISO YYYY-MM-DD format and calendar validity. A value like `"2026-02-30"` (invalid calendar date) or `"2026/05/08"` (wrong delimiter) passes the form schema and reaches the server where it fails. The form should reject this class of bad input before the network round-trip.

**Fix:**

```ts
date: z.string().date('Enter a valid date (YYYY-MM-DD).'),
```

### IN-02: `form.setErrorMap` cast hides TanStack Form error shape

**File:** `apps/web/src/components/dashboard/SettleDialog.tsx:280,336,388,424,466`

**Issue:** `field.state.meta.errors as { message?: string }[]` is a type cast that silences TypeScript. TanStack Form's actual error value type (post-v1 schema validators) may be `string | ZodIssue | ValidationError` depending on which validator ran. If the runtime shape ever diverges from the cast, errors will silently not display (accessing `.message` on a plain string returns `undefined`).

**Fix:** Check the actual runtime shape the project's version of TanStack Form emits and either use a proper type guard or change the validator to guarantee string errors before passing to `<FieldError>`.

### IN-03: `getPresets()` called on every render of PeriodRangePicker

**File:** `apps/web/src/components/dashboard/PeriodRangePicker.tsx:41-67,123`

**Issue:** `getPresets()` is called inline in JSX (`{getPresets().map(...)}`), creating a new array every render. Since the popover is conditionally rendered with `open` state, this only runs when the popover is open, so the cost is low. Still, the function creates six `Date` objects per call and the result is never memoized.

**Fix:** `useMemo(() => getPresets(), [])` — or hoist to module scope if real-time "today" isn't required for the preset range boundaries on every render (it isn't — the open-popover render is infrequent enough that stale values are not a real concern, but the pattern invites scrutiny).

### IN-04: `useCreateSettlement` does not invalidate `['transactions']`

**File:** `apps/web/src/lib/data-access/settlements/useCreateSettlement.ts:20`

**Issue:** `onSuccess` invalidates `['settlements']` only (intentional, per the comment citing Phase 4.1 D-20). However, a settlement creates a real transaction row. If the user navigates to `/transactions` immediately after settling, the list will show stale data until the 60s staleTime expires. This is a documented intentional trade-off, not a bug, but it is a user-visible consistency issue for future phases to address.

**Fix:** No immediate fix required — track as a known limitation. Phase 7.3 should add `qc.invalidateQueries({ queryKey: ['transactions'] })` here when the transactions page becomes a primary post-settlement workflow.
