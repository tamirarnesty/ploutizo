# Phase 4.2: Settlement UI - Research

**Researched:** 2026-05-08
**Domain:** React SPA UI — TanStack Router routes, ReUI DataGrid, shadcn Dialog, TanStack Query, TanStack Form
**Confidence:** HIGH

## Summary

Phase 4.2 builds the Settlement UI on top of the Phase 4.1 API. The upstream CONTEXT.md (D-01..D-21) and 04.2-UI-SPEC.md together constitute a nearly complete implementation contract — this research file focuses on what those documents leave open: verified install commands for missing components, codebase-derived code patterns, DataGrid hover-reveal mechanics, the sidebar route-driven values pattern, and the testing infrastructure.

The codebase is mature and consistent. Every pattern needed exists in a neighbor component. The main open questions are: (1) how to wire sidebar member values without a new API call, and (2) how hover-reveal works on ReUI DataGrid rows. Both are answered below.

**Primary recommendation:** Use TanStack Query dedup for sidebar member values (call `useGetSettlements` directly inside `SidebarMembersSection` — no context provider needed), and add `tableClassNames={{ bodyRow: 'group/row' }}` to the DataGrid for hover-reveal. Install missing `card` and `progress` shadcn primitives into `packages/ui` before building.

### Pattern 1: Data Access Hook (useGetSettlements)

Follow the exact same shape as `useGetAccounts.ts` and `useGetOrgMembers.ts`.

```typescript
// apps/web/src/lib/data-access/settlements/useGetSettlements.ts
// Source: pattern from apps/web/src/lib/data-access/accounts/useGetAccounts.ts [VERIFIED: codebase]
import { useQuery } from '@tanstack/react-query';
import type { UseQueryResult } from '@tanstack/react-query';
import { apiFetch } from '@/lib/queryClient';

export interface SettlementMemberBalance {
  member: { id: string; name: string; avatarUrl: string | null };
  balanceCents: number;
}

export interface SettlementAccount {
  account: {
    id: string;
    name: string;
    type: string;
    institution: string | null;
    lastFour: string | null;
    statementDueDay: number | null;
  };
  totalBalanceCents: number;
  members: SettlementMemberBalance[];
  dueDate: string | null;
  status: 'due_soon' | 'on_track' | null;
}

export interface SettlementsResponse {
  accounts: SettlementAccount[];
}

export const fetchSettlements = async (): Promise<SettlementsResponse> => {
  return apiFetch<SettlementsResponse>('/api/settlements');
};

export const useGetSettlements = (): UseQueryResult<SettlementsResponse> => {
  return useQuery({
    queryKey: ['settlements'],
    queryFn: fetchSettlements,
  });
};
```

### Pattern 2: Settlement Mutation (useCreateSettlement)

Follow `useCreateTransaction.ts` — thin transport layer, toast on success/error, invalidate `['settlements']` only.

```typescript
// apps/web/src/lib/data-access/settlements/useCreateSettlement.ts
// Source: pattern from apps/web/src/lib/data-access/transactions/useCreateTransaction.ts [VERIFIED: codebase]
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { toast } from '@ploutizo/ui/components/sonner';
import { apiFetch } from '@/lib/queryClient';

interface CreateSettlementPayload {
  payerMemberId: string;
  accountId: string;
  amountCents: number;
  date: string; // YYYY-MM-DD
}

export const useCreateSettlement = () => {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (body: CreateSettlementPayload) =>
      apiFetch('/api/settlements', {
        method: 'POST',
        body: JSON.stringify(body),
      }),
    onSuccess: () => {
      toast.success('Settlement recorded');
      qc.invalidateQueries({ queryKey: ['settlements'] });
      // Do NOT invalidate ['transactions'] — locked by Phase 4.1 D-20
    },
    onError: () => {
      toast.error('Failed to record settlement. Try again.');
    },
  });
};
```

### Pattern 7: Member Color Slot Assignment

New pattern — no existing equivalent. Define as a module-level constant in a shared file (e.g., `apps/web/src/lib/memberColors.ts`).

```typescript
// Stable sort by memberId (ascending string order) → deterministic color assignment
// Source: 04.2-UI-SPEC.md Color section [CITED: 04.2-UI-SPEC.md]
export const MEMBER_COLORS = [
  { bg: 'bg-blue-500', text: 'text-blue-700', progressFill: 'bg-blue-500' },
  { bg: 'bg-emerald-500', text: 'text-emerald-700', progressFill: 'bg-emerald-500' },
] as const;

export const getMemberColorSlot = (
  members: { id: string }[],
  memberId: string
): (typeof MEMBER_COLORS)[number] => {
  const sorted = [...members].sort((a, b) => a.id.localeCompare(b.id));
  const idx = sorted.findIndex((m) => m.id === memberId);
  return MEMBER_COLORS[idx % MEMBER_COLORS.length];
};
```

### Anti-Patterns to Avoid

- **`display:none` on hover-only Settle button:** Removes from tab order. Use `opacity-0` + `group-hover/row:opacity-100 focus-within:opacity-100` always.
- **`useState` for form field values:** All SettleDialog field values must go through `useAppForm`. [VERIFIED: CLAUDE.md Critical Constraints]
- **Raw `fetch()` in components:** All API calls use `apiFetch` from `@/lib/queryClient`. [VERIFIED: CLAUDE.md]
- **Modifying base components in `packages/ui/src/components/reui/`:** Override at usage site only. [VERIFIED: CLAUDE.md Base Components]
- **Calling `useGetTransactions` with both type filters and the full pagination:** On `/expenses` and `/income`, the default pagination (page 1, limit 25) applies unless the planner decides to expose it in URL params.
- **`color-only` status indicator:** Status badges must have text labels (WCAG 1.4.1). Always use "Due Soon" + "On Track" text inside the badge. [CITED: 04.2-UI-SPEC.md Accessibility Contract]

## Common Pitfalls

### Pitfall 1: DataGrid Row Group Class Not Applied
**What goes wrong:** Hover-reveal Settle button never appears because `group/row` is not on the `<tr>`.
**Why it happens:** DataGrid `<tr>` has no `group` class by default.
**How to avoid:** Pass `tableClassNames={{ bodyRow: 'group/row' }}` to `<DataGrid>`.
**Warning signs:** `opacity-0 group-hover/row:opacity-100` has no effect in browser.

### Pitfall 2: Settlement Mutation Invalidates Wrong Query Keys
**What goes wrong:** Transactions list cache is invalidated on settlement, causing an unnecessary refetch.
**Why it happens:** Developer copies `useCreateTransaction` and keeps `['transactions']` invalidation.
**How to avoid:** Phase 4.1 D-20 is explicit — invalidate `['settlements']` only.
**Warning signs:** Network tab shows `/api/transactions` fetch after a settlement is recorded.

### Pitfall 3: Member Picker Uses Native Radio Input
**What goes wrong:** UI-SPEC member picker styling (card rows, ring-primary selection) is impossible with native `<input type=radio>`.
**Why it happens:** Default instinct to use `RadioGroup` primitive directly.
**How to avoid:** Use `RadioGroup` as the state container but render each member as a custom card div. Use `data-state` from the radio primitive to drive `ring-primary` class on the card.
**Warning signs:** Member picker rows look like plain radio buttons, not styled card rows.

### Pitfall 4: Sidebar MEMBERS Section Visible When Sidebar Collapsed
**What goes wrong:** Member names overflow or the MEMBERS label appears in icon-collapsed sidebar.
**Why it happens:** Missing `group-data-[collapsible=icon]:hidden` wrapper.
**How to avoid:** Wrap the entire `SidebarGroup` for MEMBERS in a div with `className="group-data-[collapsible=icon]:hidden"` — same treatment as ThemeToggle in `AppSidebar.tsx`. [VERIFIED: AppSidebar.tsx line 92]
**Warning signs:** MEMBERS section shows in icon sidebar mode.

### Pitfall 5: Period Selector Wires Data in Phase 4.2
**What goes wrong:** Dashboard queries use `from`/`to` search params and break because the API doesn't support them in Phase 4.2.
**Why it happens:** Temptation to wire up the URL params to queries since they're in the URL anyway.
**How to avoid:** D-02 is explicit — URL updates, data does NOT re-fetch. Dashboard queries have no `from`/`to` params in Phase 4.2.

### Pitfall 6: Progress Component Missing
**What goes wrong:** Build fails with "Cannot find module" for progress component.
**Why it happens:** `progress` is not installed in `packages/ui` (confirmed by filesystem search).
**How to avoid:** Wave 0 task must install `card` and `progress` via `pnpm dlx shadcn@latest add card progress` from `packages/ui` before any component is built.

## Open Questions (RESOLVED)

1. **Expense/income sidebar total aggregation**
   - What we know: `/api/transactions` is paginated; no per-member aggregate endpoint exists until Phase 6.3
   - What's unclear: Should sidebar show member expense totals at all in Phase 4.2, or only show values on `/dashboard` (settlement balance)?
   - Recommendation: Simplest safe choice — only show sidebar values on `/dashboard` (settlements). Leave `/expenses` and `/income` sidebar sublines blank until Phase 6.3 adds the aggregation endpoint. This avoids the limit/accuracy issue entirely and matches D-16's intent.
   - RESOLVED: Plan 03 implements route-gated queries using `enabled` flag via direct `useQuery` calls. `useGetSettlements` fires on all routes (needed for /dashboard sidebar values); expense/income queries only fire on their respective routes. Sidebar sublines are blank on all other routes.

2. **`useGetTransactions` params for sidebar — staleTime**
   - What we know: Global staleTime is 60s; sidebar will reuse cached data
   - What's unclear: If `/expenses` route mounts with `{type: 'expense', limit: 25}` and sidebar calls with `{type: 'expense', limit: 1000}`, these are different queryKeys — two fetches
   - Recommendation: If planner pursues sidebar expense/income totals, ensure the sidebar and the page component use identical queryKey params, or accept two separate fetches
   - RESOLVED: `useGetTransactions` accepts a single `params: TransactionQueryParams` argument (confirmed from source). `enabled` gating is implemented via direct `useQuery` calls in `SidebarMembersSection` with `enabled: isExpensesRoute` / `enabled: isIncomeRoute`. Per D-15: `/expenses` shows member expense totals, `/income` shows member income totals, `/dashboard` shows settlement balance. All other routes show blank sublines.
