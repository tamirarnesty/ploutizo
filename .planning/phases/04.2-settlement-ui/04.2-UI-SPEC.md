## Spacing Scale

All values from Tailwind v4 default scale. The `p-6` (24px) main area padding is already established in `_layout.tsx` — do not override at page level.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (p-1) | Icon gaps |
| sm | 8px (p-2) | Compact row spacing, inline element gaps |
| md | 16px (p-4) | Default card internal padding, form field gaps |
| lg | 24px (p-6) | Main area padding (established in _layout.tsx), card section gaps |
| xl | 32px (p-8) | Dialog internal padding |
| 2xl | 48px (p-12) | Reserved for major section breaks |
| 3xl | 64px (p-16) | Not used in this phase |

Exceptions:
- Member avatar: 32px (h-8 w-8) in sidebar, 28px (h-7 w-7) in DataGrid rows — consistent with existing MemberAvatarGroup usage
- Settle button: 44px min touch target via `min-h-11` override when revealed on mobile (accessibility gate)

Note: shadcn Badge internal padding (`px-2.5 py-0.5`) is component-owned — not part of this spacing contract.

## Typography

All text uses the `<Text>` component from `@ploutizo/ui/components/text`. Font sizes come from variant classes — do not apply raw Tailwind size classes to user-visible text.

| Role | Variant | Size | Weight | Line Height | Usage |
|------|---------|------|--------|-------------|-------|
| Page heading | `h2` | 2xl (24px) | semibold (600) | tight | "March 2026" page title |
| Section heading | `h3` | xl (20px) | semibold (600) | snug | "Card Balances", "Settlement" panel headers |
| Body default | `body` | base (16px) | regular (400) | relaxed | Main content, amounts, member names, table cell text, form section eyebrows (uppercase via `uppercase tracking-wider`) |
| Caption/meta | `caption` | xs (12px) | regular (400) | normal | "across N cards", "owed to her", placeholder KPI subtext, form validation errors |

Font weight disambiguation: use `font-semibold` (600) on `body` variant to distinguish labels/eyebrows from regular body text. Do not introduce a medium (500) weight.

Font families: heading variants use `font-heading` (Noto Serif Variable). Body/label/caption use the inherited `font-serif` base. The `font-sans` (Geist) class is available for numeric amounts — acceptable to apply `font-sans tabular-nums` on monetary amount cells for alignment.

## Color

Tokens are CSS variables defined in `packages/ui/src/styles/globals.css`. Always use semantic token classes (`text-foreground`, `bg-card`, etc.) — never raw oklch values.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-background` / `bg-card` | Main content area, DataGrid rows, dialog surface |
| Secondary (30%) | `bg-sidebar` | Left sidebar surface |
| Accent (10%) | `bg-primary` / `text-primary` | "Record settlement" CTA button, active nav item, period selector focus ring |
| Destructive | `text-destructive` / `bg-destructive` | Not used in this phase (no delete actions on settlement screen) |
| Success / Credit | `text-success` (`text-emerald-500`) | Negative balance amounts ("$480 owed to her"), sidebar MEMBERS credit line |
| Warning / Due Soon | `text-warning` (`text-yellow-500`), `bg-warning/10` | "Due Soon" status badge |
| Info / On Track | `text-success` / `bg-success/10` | "On Track" status badge (green) |

Accent reserved for:
1. "Record settlement" primary button (solid `bg-primary`)
2. Active/selected member picker row in Settle dialog (border ring `ring-primary`)
3. Active sidebar nav item
4. Period selector calendar selected date

Per-member data-viz colors (deterministic — not the 10% accent):
- Member slot 0: `bg-blue-500` / `text-blue-700` — progress bar fill + avatar bg
- Member slot 1: `bg-emerald-500` / `text-emerald-700` — progress bar fill + avatar bg
- Slot assignment: stable sort by `memberId` (ascending) maps index → color. Store as `MEMBER_COLORS[idx % MEMBER_COLORS.length]` constant. Same approach as chart-1..5 tokens but expressed as Tailwind color classes since progress bars are not SVG.

Negative balance treatment: amount text uses `text-success` (emerald-500). Subtext "owed to her/him" also uses `text-success`. Never red for credits.

### Sidebar MEMBERS Section (D-03, D-15)

Positioned below existing nav items, above the Settings footer group. Rendered as a `SidebarGroup` with `SidebarGroupLabel` "MEMBERS" (uppercase, caption style). Each member row: `SidebarMenuItem` with Avatar + name + contextual value subline.

```
MEMBERS
  [TA] Tamir         $1,950 owed         ← /dashboard: settlement balance, text-foreground
  [EM] Emily         $480 owed to her    ← negative balance: text-success
```

Route-value mapping (D-15):
- `/dashboard` — settlement balance from `useGetSettlements` aggregated client-side
- `/expenses` — member expense total from transaction query
- `/income` — member income total from transaction query
- All other routes — no value shown (omit subline, no blank gap)

Sidebar MEMBERS is hidden when sidebar is in icon-collapsed state (`group-data-[collapsible=icon]:hidden` wrapper on entire section — consistent with existing ThemeToggle treatment).

## CardBalancesGrid Specification

Uses ReUI `DataGrid`. Columns:

| Col | Header | Content | Width |
|-----|--------|---------|-------|
| CARD | "CARD" | Bank logo chip (colored square initials, 28px, rounded-sm) + card name | 140px |
| OWNER | "OWNER" | Avatar + **account owner** from `account.owners` (same as `account_members` / GET accounts): one owner → name; multiple → **"Shared"**; none → "—". *Not* derived from settlement split balances. | 100px |
| BALANCE | "BALANCE" | `formatCurrency(balanceCents)`, `font-sans tabular-nums font-semibold` | 90px |
| BREAKDOWN BY MEMBER | "BREAKDOWN BY MEMBER" | Per-member sub-rows (avatar chip + Progress bar + amount + hover Settle button) — see below | flex-1 |
| DUE | "DUE" | "Mar 15" formatted short date | 70px |
| STATUS | "STATUS" | shadcn Badge pill | 90px |

**Breakdown column sub-rows:** Each member with a non-zero balance on this card gets one sub-row:
- Avatar chip (h-6 w-6, initials, member color background)
- `Progress` bar (flex-1, value=memberAmount/cardTotal*100, fill color = member color)
- `formatCurrency(memberAmountCents)` amount (`text-sm font-sans tabular-nums`)
- Settle button — `opacity-0 group-hover/row:opacity-100 focus-within:opacity-100` (see Accessibility below)

**Negative balance member row:** amount in `text-success`. No Settle button for credits (credit cannot be "settled" out, but D-12 says CTA remains visible — keep button visible, dialog pre-fills with outstanding balance of the account, not the credit).

## Settle Dialog Specification

Trigger: "Settle" button on hover in BREAKDOWN column (D-10). Opens with card pre-selected.

Title: `<Text variant="h3">Settle {Card Name}</Text>`
Subtitle: `<Text variant="caption">Total balance: {formatCurrency(totalCents)} · Due {dueDate}</Text>`

Sections (in order):

**SETTLING FOR** (eyebrow label: `body` variant + `font-semibold uppercase tracking-wider text-muted-foreground`)
- Radios rendered as selectable card rows (not native `<input type="radio">` — custom compound using shadcn pattern)
- Each row: radio indicator dot + Avatar + member name + "{$X outstanding}" subline + outstanding balance amount right-aligned
- Selected row: `ring-1 ring-primary bg-primary/5 rounded-lg`
- Unselected row: `border border-border rounded-lg`
- Pre-selection: first member with balance > 0 (D-11)
- All household members shown, ordered by balance descending

**AMOUNT** (eyebrow label)
- `InputGroup` with `$` prefix addon
- Pre-filled with selected member's outstanding balance in dollars (not cents)
- Right-aligned hint text "Partial OK" in `text-muted-foreground text-xs` (inline, not a label)
- `autocomplete="off"` (custom settlement amount — D-CLAUDE.md constraint)
- Validation: must be > 0; must not exceed outstanding balance (warning, not hard block — D-12 allows credit)

**DATE** (full width below amount)
- `Input` `type="date"` with `autocomplete="off"` — matches HTML5 date output to `createSettlementSchema.date` (ISO `YYYY-MM-DD`).

**Implementation note (2026-05):** The dashboard settle flow does **not** include **PAID FROM**. `createSettlement` posts against the credit card `accountId` only; there is no counterpart bank account in the API payload. A separate transactions-grid settlement form may still expose “Paid from” where product/API support it.

**NOTES OPTIONAL** (eyebrow label)
- `Textarea` placeholder "Add a note…" — min 3 rows (`rows={3}`)
- `autocomplete="off"`

Footer: `DialogFooter` with "Discard changes" (`Button variant="outline"`) + "Record settlement" (`Button variant="default"`, primary blue).

Dialog width: `max-w-lg` (512px).

Form state: `useAppForm` from `@ploutizo/ui/components/form` (TanStack Form + Zod) — never `useState` for field values.

## Accessibility Contract

| Concern | Requirement |
|---------|-------------|
| Hover-only Settle button | Must also reveal on keyboard `focus-within` of the row. Use `opacity-0 group-hover/row:opacity-100 focus-within:opacity-100` (or `group-focus-within/row:opacity-100`). Never `display:none` — that removes from tab order. |
| Settle button aria-label | `aria-label="Settle for {member name} on {card name}"` (icon-only context when label is hidden) |
| CalendarIcon button | `aria-label="Open date picker"` — icon-only button in date field |
| Status badges | Sufficient color + text label (not color alone). "Due Soon" amber + "On Track" green — text label satisfies WCAG 1.4.1. |
| Dialog focus trap | shadcn Dialog handles via Radix — no extra work needed. Initial focus on first interactive element (member picker). |
| Progress bars | `role="progressbar" aria-valuenow={percent} aria-valuemin={0} aria-valuemax={100} aria-label="{memberName} portion"` |
| Disabled nav items | `aria-disabled="true"` + `tabIndex={-1}` on "soon" items. Not wrapped in Link. |
| Amount input | `autocomplete="off"`, `inputMode="decimal"` |
| All interactive elements | Visible focus via `outline-ring/50` (already set globally in `@layer base`) |
| Motion | Skeleton loaders: `motion-safe:animate-pulse` (consistent with existing TransactionColumns pattern) |
| Truncation | All flex children with variable-length text: `min-w-0` applied to prevent overflow |

## Status Badge Spec

Per CONTEXT.md specifics: same visual style as Transactions type badges.

```tsx
// Due Soon
<Badge
  className="bg-warning/10 text-warning-foreground border-warning/20 font-serif rounded-full"
  variant="outline"
>
  Due Soon
</Badge>

// On Track
<Badge
  className="bg-success/10 text-success-foreground border-success/20 font-serif rounded-full"
  variant="outline"
>
  On Track
</Badge>
```

`rounded-full` achieves the ~18.72px border radius per badge height. `font-serif` matches Noto Serif base established in existing badges. Do not override border-radius with arbitrary values.
