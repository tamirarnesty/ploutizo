## Severity-Ranked Findings

1. **BLOCKER — Required `Paid from` can prevent settlement completion while being dropped from payload**  
   - Evidence: `apps/web/src/components/dashboard/settleFormSchema.ts:21` makes `sourceAccountId` required; `apps/web/src/components/dashboard/SettleDialog.tsx:77` explicitly discards it; `apps/web/src/components/dashboard/SettleDialog.tsx:168` still renders/select-validates it.  
   - User impact: households with no eligible source account option cannot submit settlement; households that can submit may assume bank-account attribution is persisted when it is not.  
   - Concrete fix: for dashboard settle flow, remove `sourceAccountId` from `settleFormSchema`, initial values, and dialog UI; only reintroduce when API accepts and stores a counterpart/source account.

2. **WARNING — Card-balances information hierarchy diverges from contract emphasis**  
   - Evidence: column ordering in `apps/web/src/components/dashboard/card-balances/buildCardBalancesColumns.tsx` places `due/status` before split breakdown; header title is visually downsized in `apps/web/src/components/dashboard/card-balances/CardBalancesGrid.tsx:67`; action control is default-hidden (`opacity-0`) in `apps/web/src/components/dashboard/card-balances/CardBalancesActionCell.tsx:51`.  
   - User impact: reduced scanability of "who owes what" and reduced discoverability of settlement action.  
   - Concrete fix: promote breakdown earlier in table flow (or stronger visual grouping), increase section-title prominence to match `h3` contract intent, and keep Action trigger visible at low-emphasis opacity instead of fully hidden.

3. **WARNING — Typography/spacing drift introduces style fragmentation**  
   - Evidence: hard-coded text sizes and spacing values in `CardBalancesGrid.tsx`, `CardBalancesActionCell.tsx`, `CardBalancesDueCell.tsx`, `SettlementStatusBadge.tsx`, and `CardBalancesHeaderLegend.tsx` (e.g., `text-[0.938rem]`, `text-[0.625rem]`, `text-[13px]`, `size-[7px]`, `gap-[5px]`).  
   - User impact: inconsistent density and readability across card components, especially at responsive breakpoints.  
   - Concrete fix: standardize to `Text` variants + token-scale spacing utilities; reserve arbitrary values only for strict table column constraints.

---

## Top 3 Priority Fixes

1. **Remove non-persisted required `Paid from` from dashboard settle flow** — prevents blocked submissions and mismatched mental model — update `settleFormSchema`, `SettleDialog`, and `getSettleInitialValues` to match actual mutation payload.
2. **Rebalance card-balances hierarchy for scanability** — improves "balance/split/action" comprehension — adjust column order/visual emphasis and make Action trigger persistently visible.
3. **Normalize typography + spacing tokens in card surfaces** — improves readability consistency and maintainability — replace arbitrary px/rem text/spacing in card-balance cells and legend with contract variants.

## Detailed Findings

### Pillar 1: Copywriting (2/4)

- **WARNING:** Status text casing does not match contract tone (`due soon` / `on track` rendered lowercase) in `apps/web/src/components/dashboard/card-balances/SettlementStatusBadge.tsx:29-56`.  
  **Recommendation:** map to title-cased labels (`Due Soon`, `On Track`) explicitly.
- **WARNING:** Action-menu label copy (`Settle member`) is functional but less explicit than row action + dialog contract language in `apps/web/src/components/dashboard/card-balances/CardBalancesActionCell.tsx:76-84`.  
  **Recommendation:** align menu labeling with task language, e.g. "Settle for…" entries or "Choose member to settle."

### Pillar 2: Visuals (2/4)

- **WARNING:** Main data hierarchy is compressed by small section-title treatment (`text-[0.938rem]`) in `apps/web/src/components/dashboard/card-balances/CardBalancesGrid.tsx:67`.  
  **Recommendation:** restore heading prominence closer to section heading scale.
- **WARNING:** Action affordance is hidden by default (`opacity-0`) and only appears on row hover/focus in `apps/web/src/components/dashboard/card-balances/CardBalancesActionCell.tsx:51-56`.  
  **Recommendation:** keep visible resting state (e.g. muted/ghost) so first-time users can discover settlement actions without exploratory hover.
- **WARNING:** Column sequence in `buildCardBalancesColumns.tsx` prioritizes due/status before split detail, reducing immediate split comprehension for the settlement task.  
  **Recommendation:** move split context earlier or visually group due/status as secondary metadata.

### Pillar 3: Color (3/4)

- **WARNING:** Positive owed amounts in settlement rows use warning color (`text-warning`) in `apps/web/src/components/dashboard/SettlementMemberListRow.tsx:81-88`, conflicting with contract guidance that positive owed should remain foreground-emphasis.  
  **Recommendation:** use `text-foreground` for owed amounts; reserve warning for due-risk/status signals.
- **WARNING:** `on_track` status badge omits serif treatment while `due_soon` includes it in `apps/web/src/components/dashboard/card-balances/SettlementStatusBadge.tsx:11-17`.  
  **Recommendation:** unify badge typography to preserve consistent status semantics.
- `text-primary`/`bg-primary` usage remains contained (mostly selection state), and no hardcoded hex/rgb literals were found in the audited dashboard components.

### Pillar 4: Typography (2/4)

- **WARNING:** Multiple user-facing elements bypass `Text` variant sizing via arbitrary text classes (`text-[0.625rem]`, `text-[11px]`, `text-[13px]`, `text-[0.938rem]`) across card-balance components.  
  **Recommendation:** map these to contract variants (`caption`, `body-sm`, `h3`) with minimal custom overrides.
- **WARNING:** Weight usage drifts to local `font-bold`/`font-medium` mixes in table cells, creating inconsistent emphasis rhythm.  
  **Recommendation:** define one consistent weight strategy per semantic role (heading, value, metadata).

### Pillar 5: Spacing (2/4)

- **WARNING:** Repeated arbitrary spacing/sizing values (`size-[7px]`, `gap-[5px]`, several `w-[...]` and `min-w-[...]`) in `CardBalancesHeaderLegend.tsx` and `buildCardBalancesColumns.tsx` weaken scale rhythm.  
  **Recommendation:** use token scale for internal spacing; keep arbitrary widths only where needed for strict table columns.
- **WARNING:** Dense card header spacing (`gap-0`, compact padding overrides) in `CardBalancesGrid.tsx` reduces breathing room between title and legend.  
  **Recommendation:** reintroduce small tokenized separation (`gap-1`/`gap-2`) while preserving compact density.

### Pillar 6: Experience Design (1/4)

- **BLOCKER:** Required-but-unused `sourceAccountId` field can block submission and is not persisted (`settleFormSchema.ts:21`, `SettleDialog.tsx:77-79`, `SettlePaidFromField.tsx`).  
  **Recommendation:** remove from dashboard flow until supported end-to-end.
- **WARNING:** Breakdown chips currently show static amounts with no delayed tooltip affordance in `CardBalancesBreakdownMemberChips.tsx`; this misses the intended richer scan/hover detail behavior for dense rows.  
  **Recommendation:** add per-chip tooltip treatment with delayed reveal and keyboard-trigger parity.
- **WARNING:** Empty state CTA copy in `CardBalancesEmpty.tsx` references adding accounts but provides no direct action affordance from the component.  
  **Recommendation:** add an inline route action to `/accounts` to reduce dead-end behavior.

## Shadcn Chart Opportunities

### Current Phase (04.2) — high-confidence quick wins

1. **P1: Stacked member split chart inside card-balance rows**  
   - **Pattern/components:** shadcn `chart` wrapper + Recharts `BarChart` (horizontal stacked), `ChartTooltip` for member amounts.  
   - **Why it fits now:** current `SettlementAccountRow.members[].balanceCents` already provides exactly the split proportions shown in manual segment bars (**orthogonal to the Owner column**, which uses `account.owners` — amendment 2026-05-16).  
   - **Expected value:** improves split readability and unlocks richer hover/focus details without custom SVG/div math.

2. **P1: Settlement summary composition mini-chart**  
   - **Pattern/components:** Recharts `RadialBarChart` or compact `PieChart` in shadcn chart container.  
   - **Why it fits now:** summary pane already computes owed vs credit (`useCreditCardMemberRollup`); charting can visualize total exposure mix at a glance.  
   - **Expected value:** faster household risk scan before opening per-card details.

3. **P2: Status mix micro-chart in card-balances header**  
   - **Pattern/components:** tiny `BarChart`/`PieChart` with due_soon vs on_track counts.  
   - **Why it fits now:** status values are present per account row.  
   - **Expected value:** immediate due-risk signal without sorting.

### Future Phases — roadmap-mapped recommendations

- **Budgets:**  
  - **Chart type:** grouped/stacked `BarChart` (`budgeted` vs `actual`, by category and month).  
  - **Priority:** P1 for budget rollout phase.  
  - **User value:** quickly identifies overspend categories and trend direction.

- **Overview Dashboard:**  
  - **Chart type:** `ComposedChart` (bars for spend/income + line for net), plus compact `AreaChart` trend for rolling period.  
  - **Priority:** P1.  
  - **User value:** single-view financial momentum and volatility.

- **Net Worth:**  
  - **Chart type:** `AreaChart` for net-worth timeline + stacked area for asset/liability composition.  
  - **Priority:** P1.  
  - **User value:** long-term trajectory and composition shifts.

- **Imports:**  
  - **Chart type:** `BarChart` for import error classes + `PieChart` for mapped/unmapped transaction share.  
  - **Priority:** P2.  
  - **User value:** faster cleanup prioritization after import runs.

### Adoption note

- `@ploutizo/ui` currently has no exported shadcn `chart` component and no `recharts` dependency (`packages/ui/package.json`, `apps/web/package.json`).  
- Recommended foundation step: add shadcn chart primitive to `packages/ui`, standardize chart token mappings there, then consume from `apps/web` via existing design-token classes.
