## Critical Issues

### CR-01: Clerk DELETE call failure silently produces a success response

**File:** `apps/api/src/routes/households.ts:141-150`

**Issue:** The `DELETE /members/:memberId` handler calls the Clerk API to remove the membership (`fetch(..., { method: 'DELETE' })`), but the response is never checked. If Clerk returns a non-2xx (e.g. the membership does not exist in Clerk, token expired, network error), the handler still deletes the local `org_members` row and returns `{ data: { removed: true } }`. This creates a split-brain state: the local row is gone but Clerk still has the membership, leaving the user able to re-authenticate into the org.

**Fix:**
```typescript
const clerkMembershipRes = await fetch(
  `https://api.clerk.com/v1/organizations/${orgId}/memberships/${member.externalId}`,
  {
    method: 'DELETE',
    headers: { Authorization: `Bearer ${process.env.CLERK_SECRET_KEY}` },
  }
);

if (!clerkMembershipRes.ok) {
  // Do NOT delete local row — Clerk still has the membership
  return c.json({ error: { code: 'CLERK_REMOVAL_FAILED' } }, 502);
}

await db.delete(orgMembers).where(eq(orgMembers.id, memberId));
return c.json({ data: { removed: true } });
```

## Warnings

### WR-01: Invite endpoint accepts unvalidated email body

**File:** `apps/api/src/routes/households.ts:83`

**Issue:** `POST /invitations` reads `email` directly from the raw JSON body with no schema validation (`const { email } = await c.req.json()`). A missing, null, or non-email value is forwarded to Clerk, which may return a `form_param_format_invalid` error — but it could also cause unexpected behaviour if Clerk's error response shape differs. The `InviteMemberFormSchema` (client-side) and `updateHouseholdSettingsSchema` (used on PATCH /settings) show the established pattern for server-side validation is already in place.

**Fix:**
```typescript
import { InviteMemberFormSchema } from '@ploutizo/validators';

householdsRouter.post('/invitations', async (c) => {
  const { orgId } = getAuth(c);
  const result = InviteMemberFormSchema.safeParse(await c.req.json());
  if (!result.success) {
    return c.json({ error: { code: 'VALIDATION_ERROR', errors: result.error.issues } }, 400);
  }
  const { email } = result.data;
  // ... rest unchanged
```

### WR-02: `isLoading` early-return pattern gates parallel query mount in MembersSection

**File:** `apps/web/src/components/settings/MembersSection.tsx:16-37`

**Issue:** The component renders either the skeleton or the member list based on `isLoading`. This is a conditional branch, not a conditional hook call, so it does not violate the rules-of-hooks. However, `InviteMemberForm` (line 39) is always rendered outside the conditional — this is fine. The actual issue is subtler: `HouseholdSettings` renders `HouseholdOverviewSection` and `MembersSection` sequentially. Each fires an independent query on mount (`household-overview` and `org-members`). These are parallel, which is correct. No waterfall exists here.

The real warning is in `MembersSection` line 32: `onRemove={removeMutation.mutate}` passes `mutate` directly as a prop without stabilising the reference. `removeMutation.mutate` is recreated on every render of `MembersSection`. If `MemberRow` were memoised in the future this would silently break memoisation. Given the current codebase this is low-risk but is a known footgun per `rerender-advanced-event-handler-refs`.

**Fix:**
```tsx
const handleRemove = useCallback(
  (memberId: string) => removeMutation.mutate(memberId),
  [removeMutation.mutate]
);
// ...
onRemove={handleRemove}
```

### WR-03: `TokenInitializer` re-registers token getter on every `getToken` identity change

**File:** `apps/web/src/routes/__root.tsx:38-41`

**Issue:** `setTokenGetter` is called inside a `useEffect` with `[getToken]` as the dependency. Clerk's `getToken` reference is stable within a session but re-created on token refresh events in some Clerk versions. Each call to `setTokenGetter` overwrites the module-level getter, which is the intended behaviour — however if `getToken` changes identity during an in-flight query the query may briefly use a stale getter. This is a known Clerk/TanStack interaction edge case. Storing the getter in a ref and reading it indirectly prevents the window entirely.

**Fix:**
```tsx
const TokenInitializer = () => {
  const { getToken } = useAuth()
  const getTokenRef = useRef(getToken)
  useEffect(() => { getTokenRef.current = getToken }, [getToken])
  useEffect(() => {
    setTokenGetter(() => getTokenRef.current())
  }, []) // register once; ref stays current
  return null
}
```

## Info

### IN-01: `AvatarImage` receives `undefined` instead of omitting `src` when no image

**File:** `apps/web/src/components/settings/HouseholdOverviewSection.tsx:16`, `apps/web/src/components/settings/MemberRow.tsx:26`

**Issue:** Both call `<AvatarImage src={x ?? undefined} />`. Passing `src={undefined}` is functionally equivalent to omitting the prop in React, so this works correctly. It is a minor style inconsistency — if the underlying `@base-ui/react/avatar` `Image` component ever type-narrows `src` to `string` (non-optional), this would produce a type error. The pattern is safe today.

**Fix:** No change required. If the team prefers consistency, the `AvatarImage` wrapper in `packages/ui/src/components/avatar.tsx` could accept `src?: string` and handle the `undefined` internally — but this is a base-component change and out of scope per project constraints.

it('returns 502 when Clerk membership deletion fails', async () => {
