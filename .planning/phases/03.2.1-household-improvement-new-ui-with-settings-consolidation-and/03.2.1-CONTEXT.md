# Phase 03.2.1: Household Settings Consolidation & Invitation Flow - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Consolidate the separate "Household" and "Members" settings tabs into a single unified Household settings page. Add member invitation via a new API endpoint. Add member removal via a new API endpoint. No transaction, budget, or settlement work in this phase.

**In scope:**
- Remove "Members" tab from settings nav (`route.tsx` + delete route file)
- Expand `/settings/household` page into sections: overview → members → invite → threshold
- New `POST /api/households/invitations` endpoint (calls Clerk backend API)
- New `DELETE /api/households/members/:memberId` endpoint (calls Clerk + removes local row)
- `OrganizationMembersSettings.tsx` absorbed into `HouseholdSettings.tsx` (delete standalone file)

**Out of scope:**
- Role management (all members are admin in v1)
- Org name editing (read-only display only)
- Local invitations table tracking (Clerk manages invitation state)
- Remove-self (button disabled for current user)

</domain>

<decisions>
## Implementation Decisions

### Page Layout
- **D-01:** Flat sections with `<Separator />` between them. No card wrappers. Same visual pattern as existing HouseholdSettings — single `flex flex-col gap-8 max-w-2xl`.
- **D-02:** Section order: Household Overview → Members → Settlement Threshold.
- **D-03:** The invite form lives inline at the bottom of the Members section (below the member list), not in a dialog.

### Household Overview Section
- **D-04:** Shows org name + avatar (shadcn `<Avatar>` with `imageUrl`, fallback to org name initials). Read-only — no editing UI.
- **D-05:** Org data sourced from a new `GET /api/households` endpoint (or extend existing households route) that returns `{ name, imageUrl }` from the local `orgs` table (already stored via webhook).

### Member Display
- **D-06:** Each member row: `<Avatar>` (imageUrl with initials fallback) + display name + "You" badge (if current user) + role label + remove button.
- **D-07:** Remove button is hidden/disabled on the current user's own row (no self-removal).
- **D-08:** Remove flow: click remove → confirmation dialog ("Remove [name] from this household?") → `DELETE /api/households/members/:memberId`.
- **D-09:** `useGetOrgMembers` hook and `/api/households/members` API endpoint stay as-is (no backend changes to the GET).

### Invite Form
- **D-10:** Inline email input + "Send invite" button directly below the member list, within the Members section.
- **D-11:** Uses `useAppForm` (TanStack Form + Zod) with email validation.
- **D-12:** On success: input clears + success toast ("Invite sent to email@example.com").
- **D-13:** On error: error toast with specific message based on API error code:
  - Already a member → "Already a member of this household"
  - Invitation already pending → "An invitation is already pending for this email"
  - Invalid email (caught by Zod) → inline field validation error
  - Generic fallback → "Failed to send invite. Try again."

### Invite API (`POST /api/households/invitations`)
- **D-14:** Hono route calls Clerk's REST API `POST /v1/organizations/{orgId}/invitations` with a `CLERK_SECRET_KEY` service-level call. No local DB record.
- **D-15:** Role hardcoded to `"org:admin"` (all members are admins in v1).
- **D-16:** Returns `200` on success. Maps Clerk error codes to app error codes:
  - Clerk `already_a_member_of_this_org` → `409 ALREADY_MEMBER`
  - Clerk `invitation_already_pending` → `409 INVITATION_PENDING`
  - Invalid email → `400 INVALID_EMAIL`

### Remove Member API (`DELETE /api/households/members/:memberId`)
- **D-17:** Hono route calls Clerk `DELETE /v1/organizations/{orgId}/memberships/{userId}` then deletes the row from local `org_members` table.
- **D-18:** `memberId` in the route is the local `org_members.id` UUID. Route looks up `externalId` from local DB to construct the Clerk API call.
- **D-19:** Returns `403` if the caller attempts to remove themselves (server-side guard matching `getAuth(c).userId` against the target member's `externalId`).

### Settings Nav
- **D-20:** Remove the "Members" tab entry from `settingsTabs` in `_layout.settings/route.tsx`. Delete `_layout.settings/organization-members.tsx` route file.

### Claude's Discretion
- Avatar component size for member rows (sm variant likely)
- Exact toast wording beyond what's specified above
- Spacing/padding within sections
- Whether to inline `OrganizationMembersSettings` logic or delete the file and rewrite inline in `HouseholdSettings`
- Loading skeleton for the overview section

- **consolidate-household-settings-page-with-member-invitations** — Merge HouseholdSettings + OrganizationMembersSettings, remove Members tab, add invite form. Folded in full: all points of the todo are captured in D-01 through D-20 above.
