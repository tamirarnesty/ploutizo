**Primary recommendation:** Wave 0 installs Avatar + Sonner. Wave 1 patches the members API query and adds the three new Hono routes. Wave 2 builds `HouseholdSettings.tsx` (the consolidated component). Wave 3 removes dead files.

### Pattern 1: New Mutation Hook (useInviteMember)

Follows the `useUpdateHouseholdSettings` pattern exactly: `useMutation` + `apiFetch` + `onSettled` invalidation.

```typescript
// Source: apps/web/src/lib/data-access/household/useUpdateHouseholdSettings.ts (reference)
export const useInviteMember = () => {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: (email: string) =>
      apiFetch("/api/households/invitations", {
        method: "POST",
        body: JSON.stringify({ email }),
      }),
    onSettled: () => void qc.invalidateQueries({ queryKey: ["org-members"] }),
  })
}
```

### Anti-Patterns to Avoid

- **Raw `fetch()` in components or hooks** тАФ always use `apiFetch` from `@/lib/queryClient`
- **`useState` for form field values** тАФ `useAppForm` only (CLAUDE.md)
- **Modifying `packages/ui/src/components/reui/` or shadcn-generated files** тАФ override at usage site
- **`forwardRef` in new custom components** тАФ React 19 project; pass `ref` as regular prop (CLAUDE.md skills)
- **`&&` for conditional rendering that returns nothing** тАФ use ternary (vercel-react-best-practices)
- **`isLoading` early return gating other hooks** тАФ hoist all queries to top level, show combined skeleton

## Common Pitfalls

### Pitfall 1: D-09 Misread тАФ "GET stays unchanged" Masks a Query Bug

**What goes wrong:** Planner reads D-09 as "no changes to GET /members", skips the query patch, and member avatars render broken/missing.
**Why it happens:** The `OrgMember` type was updated in quick-260411-r4m but the SQL query was not.
**How to avoid:** The query in `households.ts` GET /members must select `users.imageUrl`, `users.firstName`, `users.lastName`. This is a one-task fix in Wave 1 before any avatar rendering.
**Warning signs:** TypeScript won't catch this тАФ the return type is inferred from Drizzle, so the type will be wrong at the API layer even if `OrgMember` expects the fields.

### Pitfall 2: Avatar / Sonner Not Yet Installed

**What goes wrong:** Plan tasks reference `@ploutizo/ui/components/avatar` or `@ploutizo/ui/components/sonner` but the files do not exist, causing import errors.
**Why it happens:** These components are listed in the UI-SPEC as available but were never installed. [VERIFIED: no avatar.tsx or sonner.tsx in packages/ui/src/components/]
**How to avoid:** Wave 0 task: run `shadcn add` for both components into `packages/ui`. Commands must be run from `apps/web/` (where `components.json` lives).
**Warning signs:** TypeScript error "cannot find module '@ploutizo/ui/components/avatar'".

### Pitfall 3: Clerk REST API тАФ Field Name Mismatch

**What goes wrong:** Sending `email` instead of `email_address` in the Clerk invitation POST body silently fails or returns a 422.
**Why it happens:** Clerk's REST API uses snake_case (`email_address`) while the frontend form uses `email`.
**How to avoid:** Hono route maps: `{ email }` (from request) тЖТ `{ email_address: email, role: 'org:admin' }` to Clerk.

### Pitfall 4: Remove Self тАФ Client vs Server Guard

**What goes wrong:** Only client-side guard (hide button) with no server-side validation; direct API calls can bypass it.
**Why it happens:** Easy to implement only the UI hide.
**How to avoid:** D-19 specifies a server-side 403 guard in the DELETE route. Both must be implemented.

### Pitfall 5: Clerk `memberships` vs `members` URL

**What goes wrong:** Using wrong Clerk REST path for remove тАФ Clerk uses `/memberships/{userId}` not `/members/{userId}`.
**Why it happens:** Naming inconsistency in Clerk docs vs intuition.
**How to avoid:** Per CONTEXT.md D-17: `DELETE /v1/organizations/{orgId}/memberships/{userId}` where `{userId}` is the Clerk user ID (`externalId`).

### Pitfall 6: `CLERK_SECRET_KEY` in Hono тАФ Must Use `process.env`

**What goes wrong:** Using `getAuth(c).token` for Clerk backend API calls (that is the user JWT, not the service key).
**Why it happens:** Conflating user session auth with service-level Clerk API auth.
**How to avoid:** All Clerk backend REST calls use `Authorization: Bearer ${process.env.CLERK_SECRET_KEY}`.

### Pitfall 7: `orgs.name` / `orgs.imageUrl` Sourcing

**What goes wrong:** Attempting to call Clerk `useOrganization()` for the overview section instead of the new API endpoint.
**Why it happens:** REQUIREMENTS.md ┬з1 says "Org name displayed via `useOrganization()`" тАФ but D-05 locked the decision to use the local `orgs` table via `GET /api/households` instead.
**How to avoid:** D-05 is locked. `GET /api/households` returns `{ name, imageUrl }` from local DB (already populated by webhook handlers). [VERIFIED: webhooks.ts handles `organization.created` and `organization.updated`, storing both fields]

### Invite Form (useAppForm pattern)

```typescript
// Source: apps/web/src/components/settings/HouseholdSettingsForm.tsx (reference)
const form = useAppForm({
  defaultValues: { email: "" },
  validators: {
    onSubmit: ({ value }) => {
      const r = z.object({ email: z.string().email("Enter a valid email address") }).safeParse(value)
      if (!r.success) return r.error.issues.map((i) => i.message).join(", ")
    },
  },
  onSubmit: ({ value }) => {
    mutation.mutate(value.email, {
      onSuccess: () => {
        form.reset()
        toast.success(`Invite sent to ${value.email}.`)
      },
      onError: (err) => {
        const code = (err as { error?: { code?: string } })?.error?.code
        if (code === "ALREADY_MEMBER") toast.error("Already a member of this household.")
        else if (code === "INVITATION_PENDING") toast.error("An invitation is already pending for this email.")
        else toast.error("Failed to send invite. Try again.")
      },
    })
  },
})
```

### Hono DELETE /members/:memberId

```typescript
// Source: CONTEXT.md D-17, D-18, D-19
householdsRouter.delete('/members/:memberId', async (c) => {
  const { orgId, userId: callerClerkId } = getAuth(c)
  const { memberId } = c.req.param()

  // Look up local member row to get externalId
  const member = await db.select({ externalId: users.externalId })
    .from(orgMembers)
    .innerJoin(users, eq(users.id, orgMembers.userId))
    .where(and(eq(orgMembers.id, memberId), eq(orgMembers.orgId, orgId!)))
    .then((r) => r.at(0))

  if (!member) return c.json({ error: { code: 'NOT_FOUND' } }, 404)

  // Server-side self-removal guard
  if (member.externalId === callerClerkId) {
    return c.json({ error: { code: 'SELF_REMOVAL_FORBIDDEN' } }, 403)
  }

  // Call Clerk API
  await fetch(`https://api.clerk.com/v1/organizations/${orgId}/memberships/${member.externalId}`, {
    method: 'DELETE',
    headers: { Authorization: `Bearer ${process.env.CLERK_SECRET_KEY}` },
  })

  // Remove local row
  await db.delete(orgMembers).where(eq(orgMembers.id, memberId))

  return c.json({ data: { removed: true } })
})
```
